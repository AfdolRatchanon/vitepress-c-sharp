# Enum Methods & Flags (เจาะลึกการใช้ประโยคและบิต)

> 💡 **เป้าหมาย:** สยายปีกพลังของ Enum ด้วยการฝึกใช้คลาสเสริม (Helper Methods) เพื่อแปลงข้อความจากมนุษย์ให้กลับเป็น Enum และทะลวงเข้าสู่วิชาขั้นสูงสุดของสถาปัตยกรรมระดับล่าง นั่นคือ **Bitwise Flags** การผสมผสานสถานะหลายตัวเข้าด้วยกันในตัวแปรเดียว!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. วิชาแปลงข้อความให้กลายเป็น Enum (The Parsing Magic)

ในบทที่แล้ว เราแปลง `int` เป็น `enum` ด้วยการทำ Type Casting แต่ในโลกความจริง ข้อมูลที่เข้ามาจากฝั่งเว็บแอปพลิเคชัน (เช่น ไฟล์ JSON หรือตอนคนพิมพ์ใน Console) มักจะอยู่ในรูปแบบของ **ข้อความ `string` (เช่น "Pending")**
คุณไม่สามารถ Cast โง่ๆ แบบ `(OrderStatus)"Pending"` ได้! คอมไพเลอร์จะด่าเอา!

เราต้องใช้เครื่องจักรที่ชื่อว่า **`Enum.Parse`** หรือ **`Enum.TryParse`** เพื่อทำหน้าที่ "อ่านข้อความ แล้วไปเทียบในพจนานุกรม Enum ว่ามีชื่อนี้ไหม ถ้ามีก็ค่อยเสกเป็นค่า Enum กลับมาให้เรา"

---

### 2. วิกฤติการมีหลายสถานะพร้อมกัน (The Multi-State Problem)

ปกติแล้ว 1 ตัวแปรของ Enum จะเก็บได้แค่ **1 สถานะ** เท่านั้น 
แต่ลองนึกถึงระบบ "สิทธิ์การเข้าถึงไฟล์ (File Permissions)" ในระบบปฏิบัติการ Linux
- ผู้ใช้คนนึงอาจจะมีสิทธิ์ **อ่าน (Read)** อย่างเดียว
- อีกคนอาจจะมีสิทธิ์ **อ่าน + เขียน (Read & Write)**
- แอดมินอาจจะมีสิทธิ์ **อ่าน + เขียน + รันโปรแกรม (Read, Write & Execute)**

ถ้าคุณใช้ Enum แบบปกติ คุณอาจจะต้องทะลึ่งสร้างสเตตัสผสมขึ้นมาเองให้วุ่นวาย เช่น `ReadAndWrite`, `ReadWriteAndExecute` ซึ่งมันทุเรศมาก! 
วิธีแก้ที่ทั่วโลกใช้คือการประยุกต์ใช้วิชาคณิตศาสตร์ฐานสอง (Binary Bitwise) ผสานกับ Attribute พิเศษที่ชื่อว่า **`[Flags]`**

---

### 3. คัมภีร์แห่ง `[Flags]` และ Bitwise Operators

เมื่อเราแปะป้าย `[Flags]` ไว้บนหัว Enum มันคือการประกาศกร้าวว่า **"Enum ตัวนี้ สามารถเอาสเตตัสมารวมร่างกันได้!"**
แต่มีกฎเหล็ก 1 ข้อ: **ตัวเลขที่อยู่หลังสถานะ ห้ามเรียง 1, 2, 3, 4 เด็ดขาด! คุณ "บังคับ" ต้องใช้เลขทวีคูณของฐานสอง (Powers of Two) คือ 1, 2, 4, 8, 16... เท่านั้น!**

**ทำไมต้อง 1, 2, 4, 8? (The Binary Secret):**
- 1 ในเลขฐานสองคือ `0001`
- 2 ในเลขฐานสองคือ `0010`
- 4 ในเลขฐานสองคือ `0100`
สังเกตไหมครับ? เลข 1 ของมันจะ "ไม่เคยมีตำแหน่งชนกันเลย!" ทำให้คอมพิวเตอร์สามารถเอาหน้ากากบิต (Bit Mask) มาซ้อนทับกันได้โดยไม่ผิดเพี้ยน!

**อาวุธสำหรับการหลอมรวมสถานะ:**
- ใช้เครื่องหมาย **`|` (Bitwise OR):** เอาไว้ **"รวม"** สิทธิ์เข้าด้วยกัน (เช่น `Read | Write` แปลว่าให้สิทธิ์ทั้งคู่นะ)
- ใช้เครื่องหมาย **`&` (Bitwise AND):** คู่กับเมธอด `.HasFlag()` เอาไว้ **"ถาม"** ว่ามีสิทธิ์ย่อยนี้ผสมอยู่ข้างในหรือเปล่า?

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: ระบบ Parse ข้อความจาก API (Text-to-Enum)

ตัวอย่างนี้แสดงวิธีรับข้อความดิบๆ จากผู้ใช้ แล้วพยายามแปลงกลับเป็น Enum โดยใช้ระบบดักจับแบบเดียวกับ `TryParse` ที่เรียนในบทแรกๆ

::: code-group
```csharp [Program.cs]
using System;

namespace EnumParsing
{
    enum PaymentMethod
    {
        Cash,
        CreditCard,
        PromptPay
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบเลือกช่องทางชำระเงิน ===");
            Console.WriteLine("พิมพ์ตัวเลือก: Cash, CreditCard, หรือ PromptPay");
            Console.Write(">> ");
            
            string userInput = Console.ReadLine();

            // -----------------------------------------------------------
            // วิชาการแปลง String เป็น Enum อย่างปลอดภัย (TryParse)
            // -----------------------------------------------------------
            // พารามิเตอร์ตัวที่ 1: ข้อความดิบ
            // พารามิเตอร์ตัวที่ 2: true หมายถึง "จงละทิ้งอคติพิมพ์เล็กพิมพ์ใหญ่ซะ (IgnoreCase)" (เช่น พิมพ์ cash ก็ให้ผ่าน)
            // พารามิเตอร์ตัวที่ 3: ท่อ out พ่นผลลัพธ์ที่แปลงสำเร็จออกมา
            
            if (Enum.TryParse(userInput, true, out PaymentMethod selectedMethod))
            {
                Console.ForegroundColor = ConsoleColor.Green;
                // คอมพิวเตอร์ฉลาดพอที่รู้ว่า selectedMethod เป็นชนิด PaymentMethod แล้ว!
                Console.WriteLine($"✅ ยืนยันการชำระเงินด้วย: {selectedMethod}");
                Console.ResetColor();

                // สามารถเอาไปเข้า switch ทำงานต่อได้เลย
                if (selectedMethod == PaymentMethod.PromptPay)
                {
                    Console.WriteLine("📱 โปรดสแกน QR Code ที่หน้าจอ");
                }
            }
            else
            {
                // ถ้าผู้ใช้พิมพ์มั่วๆ เช่น "Bitcoin" เครื่องจักรจะหาไม่เจอและคืนค่า false
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("❌ ล้มเหลว: ไม่พบช่องทางชำระเงินที่คุณพิมพ์ในระบบ!");
                Console.ResetColor();
            }
        }
    }
}
```
:::

### ตัวอย่างที่ 2: สุดยอดวิชาคุมสิทธิ์ด้วย Bitwise Flags

นี่คือโค้ดระดับ Advanced Architecture ที่คุณจะได้เจอแน่นอนเมื่อเข้าไปทำระบบ Permission ให้บริษัท!

::: code-group
```csharp [Program.cs]
using System;

namespace BitwiseFlags
{
    // 1. แปะป้ายเวทมนตร์ [Flags] ไว้บนหัว เพื่อให้รู้ว่า Enum ตัวนี้หลอมรวมได้!
    [Flags]
    enum FilePermission
    {
        None = 0,         // ไม่มีสิทธิ์อะไรเลย
        Read = 1,         // สิทธิ์อ่าน (ฐานสอง: 0001)
        Write = 2,        // สิทธิ์เขียน (ฐานสอง: 0010)
        Execute = 4,      // สิทธิ์รันโปรแกรม (ฐานสอง: 0100)
        
        // เราสามารถประกาศตัวแปรคอมโบไว้ล่วงหน้าได้เลยโดยใช้ | รวมกัน
        FullControl = Read | Write | Execute  // สิทธิ์แอดมิน (ฐานสอง: 0111 = 7)
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบจัดการสิทธิ์เข้าถึงไฟล์ระดับ OS ===\n");

            // 2. การ "รวมร่าง" สถานะ (ใช้วิชา | Bitwise OR)
            // ลูกจ้างคนนี้ สามารถทั้ง อ่าน และ เขียน ได้พร้อมกัน!
            FilePermission employeePerm = FilePermission.Read | FilePermission.Write;
            
            Console.WriteLine($"[สิทธิ์ของลูกจ้าง] : {employeePerm}");
            // น่าทึ่งมาก! C# จะพ่นคำว่า "Read, Write" ออกมาให้เองสวยๆ บนหน้าจอเลย (เพราะพลังของป้าย [Flags])

            // 3. การ "ตรวจสอบ" ว่ามีสิทธิ์ย่อยแฝงอยู่ไหม (ใช้วิชา HasFlag)
            Console.WriteLine("\n[ทดสอบการกระทำของลูกจ้าง]");

            if (employeePerm.HasFlag(FilePermission.Read))
            {
                Console.WriteLine("✅ อนุญาตให้อ่านไฟล์ (เปิดดูเอกสารได้)");
            }

            if (employeePerm.HasFlag(FilePermission.Execute))
            {
                Console.WriteLine("✅ อนุญาตรันโปรแกรม");
            }
            else
            {
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("❌ ปฏิเสธ! คุณไม่มีสิทธิ์รันโปรแกรม (Execute)!");
                Console.ResetColor();
            }

            // 4. การ "ลบสิทธิ์ (Revoke)" ออกไปตัวนึง!
            // เช่น ทำโทษลูกจ้างโดยการริบสิทธิ์ Write ทิ้งไป (ใช้วิชา Bitwise AND & คู่กับเครื่องหมายกลับด้าน NOT ~)
            // ลอจิก: เอาสิทธิ์ปัจจุบัน (Read|Write) มาหักล้างกับส่วนกลับของ Write (ลบความสามารถเขียนทิ้ง)
            Console.WriteLine("\n>> ผู้ดูแลระบบทำการริบสิทธิ์เขียน (Write) ของลูกจ้าง!");
            employeePerm = employeePerm & ~FilePermission.Write;

            Console.WriteLine($"[สิทธิ์ที่เหลืออยู่] : {employeePerm}"); 
            // จะเหลือแค่ Read ตัวเดียวโดดๆ
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: อันตรายของคนไม่รู้กฎ 1-2-4-8**
พิจารณาโค้ดหายนะด้านล่างนี้:
```csharp
[Flags]
enum Toppings { Cheese = 1, Bacon = 2, Sausage = 3 }
```
ถ้าคุณสั่ง `Toppings myPizza = Toppings.Cheese | Toppings.Bacon;` แล้วปริ้นตัวแปร `myPizza` ออกหน้าจอ คุณจะได้คำตอบเป็นคำว่า `Cheese, Bacon` หรือไม่? ทำไมจึงเป็นเช่นนั้น และจะแก้อย่างไร?

### **โจทย์ที่ 2: ระบบคัดกรองพนักงาน (The Flags Validator)**
กำหนดให้:
```csharp
[Flags]
enum Skills { Coding = 1, Design = 2, Marketing = 4 }
```
จงเขียนประโยคเงื่อนไข `if` เพื่อตรวจสอบตัวแปร `Skills candidate = Skills.Coding | Skills.Design;` ว่า ผู้สมัครคนนี้ **"มีสกิล Design ผสมอยู่ด้วยหรือไม่?"** (ให้ใช้เมธอด `.HasFlag()`)

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** หายนะแน่นอน! คุณจะไม่ได้ `Cheese, Bacon` แต่คุณจะได้คำว่า **`Sausage`** ออกมาหน้าตาเฉยเลย! เพราะ 1 รวมกับ 2 ทางคณิตศาสตร์มันได้เลข 3 ซึ่งไปตรงกับรหัสของ Sausage พอดีเป๊ะ! (นี่คือเหตุผลที่ [Flags] บังคับให้คุณต้องข้ามเลข 3 ไปใช้เลข 4, 8, 16 ยังไงล่ะครับ!)
- **สำหรับโจทย์ 2:** เรียบง่ายแต่น่าเกรงขาม: `if (candidate.HasFlag(Skills.Design)) { ... }` จบข่าว! สมัยก่อนที่ C# ยังไม่มีเมธอดนี้ โปรแกรมเมอร์ต้องเขียนคณิตศาสตร์ระดับต่ำปวดหัวๆ ว่า `if ((candidate & Skills.Design) == Skills.Design)` โชคดีที่เราเกิดในยุคที่ทุกอย่างสบายขึ้นครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ระบบกรองบทบาทผู้ใช้ (User Role Matrix)**

บริษัทของคุณกำลังพัฒนากระดานสนทนา (Web Forum)
**ข้อมูลเบื้องต้น:**
```csharp
[Flags]
enum Roles {
    Guest = 1,
    User = 2,
    Moderator = 4,
    Admin = 8
}
```

**ภารกิจของคุณ:** 
1. สร้างผู้ใช้ระดับเทพ 1 คน: `Roles superUser = Roles.User | Roles.Moderator;` (เป็นทั้งสมาชิกทั่วไป และคนคุมบอร์ด)
2. เขียนลอจิกจำลองการทำงาน 3 โซนของเว็บไซต์:
   - **โซน A (โพสต์กระทู้):** ต้องมีสิทธิ์ `User` ขึ้นไป ถึงจะทำได้ (Guest ทำไม่ได้)
   - **โซน B (ลบกระทู้ขยะ):** ต้องมีสิทธิ์ `Moderator` หรือ `Admin` คนใดคนหนึ่งเท่านั้น ถึงจะทำได้
   - **โซน C (ลบเซิร์ฟเวอร์ทิ้ง):** ต้องมีสิทธิ์ `Admin` เท่านั้น

จงเขียนเงื่อนไขตรวจสอบ (Authorization) เพื่อให้ `superUser` เดินฝ่าด่านทั้ง 3 โซนนี้ 
- โซน A: ต้องพิมพ์ว่า "อนุญาตให้โพสต์"
- โซน B: ต้องพิมพ์ว่า "อนุญาตให้ลบกระทู้"
- โซน C: ต้องพิมพ์ว่า "ปฏิเสธ! คุณไม่ใช่ Admin"

ถ้าคุณเขียนลอจิกเช็คสิทธิ์แบบ Matrix นี้ผ่าน คุณก็พร้อมไปทำระบบ Security ให้กับ Website Back-End ได้แล้วครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในฟังก์ชัน `Enum.TryParse(text, true, out Result)` ทำไมต้องมีพารามิเตอร์ `true` ตรงกลางด้วย? ถ้าปรับเป็น `false` จะเกิดผลกระทบอะไรกับประสบการณ์ผู้ใช้ (UX)?
**แนวคำตอบ:** ค่า `true` ตรงกลางคือโหมด **"IgnoreCase (ไม่สนตัวพิมพ์เล็กใหญ่)"** ครับ! 
ถ้าคุณตั้งเป็น `false` มันจะโหมดโหดสลัด คือผู้ใช้ต้องพิมพ์คำว่า `"CreditCard"` ตัว C พิมพ์ใหญ่ 2 ตัวเป๊ะๆ เท่านั้น! ถ้าเผลอพิมพ์ `"creditcard"` หรือ `"Creditcard"` โปรแกรมจะถือว่าผิดทันที! การตั้งเป็น `true` จึงช่วยรักษาชีวิตโปรแกรมเมอร์จากการโดนลูกค้าด่าเรื่องความยืดหยุ่นของระบบได้ครับ!

**คำถาม 2:** สมมติ Enum [Flags] ถูกใช้จนครบโควต้าเลข 1, 2, 4, 8, 16... ไปจนถึง 2 พันล้าน (สุดขอบเขตของ int) แล้วเกิดบริษัทต้องการเพิ่มสถานะที่ 33 เข้ามา เราจะทำยังไง?
**แนวคำตอบ:** ย้อนกลับไปวิชาบทที่ 2 และบทที่แล้วครับ! เราสามารถสั่งให้ Enum เก็บเลขที่ใหญ่กว่า `int` ได้!
เพียงแค่เติมโคลอนแล้วขยายร่างมัน: `enum Permissions : long { ... }` 
ชนิดข้อมูล `long` เก็บเลขฐานสองได้ถึง 64 บิต (64 สถานะ!) ซึ่งมากพอที่จะยัดสเตตัสทั้งบริษัทลงไปได้อย่างเหลือเฟือเลยครับ! นี่คือการดีไซน์ระดับสถาปัตยกรรม (Architectural Scalability) ครับ
:::
