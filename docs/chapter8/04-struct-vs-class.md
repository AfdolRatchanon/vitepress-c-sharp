# Struct vs Class (เจาะลึกความลับแห่งหน่วยความจำ)

> 💡 **เป้าหมาย:** ปลดล็อกดวงตาเห็นธรรม (Memory Eyes)! บทนี้คือ "กระดูกสันหลัง" ของสถาปัตยกรรม .NET ทั้งปวง คุณจะได้รู้ว่าทำไมการโยน `struct` ถึงทำงานแบบ "ก๊อปปี้ร่างโคลน" และทำไมการใช้คำว่า `ref` จึงช่วยกอบกู้เอกราชของข้อมูลต้นฉบับเอาไว้ได้ นี่คือความลับที่แยก "โค้ดเด้อ (Coder)" ออกจาก "วิศวกรซอฟต์แวร์ (Software Engineer)"

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. โลกแบ่งเป็นสองฝั่ง: Stack และ Heap (The Memory Divide)

เมื่อโปรแกรม C# ทำงาน แรม (RAM) ในเครื่องคอมพิวเตอร์ของคุณจะถูกแบ่งออกเป็น 2 โซนหลักๆ เสมือนโลกคู่ขนาน:
1. **The Stack (โกดังเก็บของเล็ก รวดเร็วดุจสายฟ้า):**
   - เป็นที่เก็บข้อมูลพื้นฐาน เช่น `int`, `bool`, `double` รวมถึง **`struct`** ทุกตัวบนโลก!
   - โซนนี้ทำงานเป็นระบบ "ซ้อนจาน" (Last-In, First-Out) ฟังก์ชันไหนรันเสร็จ มันระเบิดจานใบบนทิ้งทันที! (เร็วมาก ไม่ทิ้งขยะ)
   - ข้อมูลที่อยู่ในนี้เรียกว่า **Value Types (ชนิดข้อมูลแบบแปะป้ายราคา)**
2. **The Heap (คฤหาสน์หรู ใหญ่โตแต่วิ่งช้า):**
   - เป็นที่เก็บข้อมูลขนาดใหญ่ เช่น `string`, `Array`, `List` รวมถึง **`class`** 
   - โซนนี้เป็นคลังสินค้าไร้รูปแบบ ถ้าฟังก์ชันรันเสร็จ ของในนี้จะไม่ถูกลบกะทันหัน แต่ต้องรอรถขยะ (Garbage Collector) มาตามเก็บกวาดทีหลัง (กินทรัพยากรเครื่อง)
   - ข้อมูลที่อยู่ในนี้เรียกว่า **Reference Types (ชนิดข้อมูลแบบตั๋วฝากของ)**

---

### 2. อาถรรพ์แห่งการก๊อปปี้ (The Pass-By-Value Curse)

เนื่องจาก **`struct` เป็น Value Type** (อยู่ในโกดัง Stack) กฎของจักรวาลระบุไว้ว่า: 
**"ทุกครั้งที่คุณเอา Struct ไปโยนใส่ตัวแปรอื่น หรือโยนเข้า Method... คอมพิวเตอร์จะถ่ายเอกสาร (Copy) สร้างร่างโคลนขึ้นมาใหม่ทั้งหมด 100%!"**

สมมติคุณมีกล่อง `struct Player` ที่มี HP=100
คุณเอา Player คนนี้ โยนเข้าไปใน Method `TakeDamage()` เพื่อลด HP เหลือ 50
ผลลัพธ์คือ: **ร่างโคลนใน Method เลือดลดเหลือ 50... แต่ร่างต้นฉบับใน Main เลือดยังเต็ม 100 เหมือนเดิม!**

---

### 3. อัศวินกู้ภัย: คีย์เวิร์ด `ref` (The Reference Rescuer)

ถ้าเรา "อยากจะแก้เลือดของร่างต้นฉบับจริงๆ" (ไม่อยากให้มันถ่ายเอกสารร่างโคลน) เราต้องใช้วิชาเปิดประตูมิติที่ชื่อว่า **`ref` (Reference)**
การใส่ `ref` นำหน้า `struct` หมายความว่า: 
"เฮ้ยคอมพิวเตอร์! ไม่ต้องถ่ายเอกสารนะ เอาเส้นด้ายมาผูกโยงจาก Method พุ่งกลับไปชี้ที่กล่องต้นฉบับใน Main เลย! ฉันจะผ่าตัดกล่องต้นฉบับ!"

*(นี่คือสาเหตุที่ Challenge ในบทที่แล้ว บังคับให้คุณพิมพ์คำว่า `ref` ในการทำระบบต่อสู้ครับ)*

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: แฉพฤติกรรมร่างโคลน (The Clone Exposure)

โค้ดห้องทดลองนี้จะพิสูจน์ให้คุณเห็นกับตาว่า `struct` มันหักหลังคุณยังไง ถ้าคุณลืมใช้ `ref`!

::: code-group
```csharp [Program.cs]
using System;

namespace StructMutation
{
    struct Player
    {
        public string Name;
        public int HP;
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== การพิสูจน์อาถรรพ์ Struct ===");

            // 1. สร้างร่างต้นฉบับ (Original)
            Player arthur = new Player();
            arthur.Name = "อาร์เธอร์";
            arthur.HP = 100;

            Console.WriteLine($"[Main] เลือดต้นฉบับ: {arthur.HP}");

            // -----------------------------------------------------------
            // การทดลอง A: การถ่ายเอกสารผ่านเครื่องหมายเท่ากับ (=)
            // -----------------------------------------------------------
            Console.WriteLine("\n[ทดลอง A] สั่งโคลนนิ่ง: Player clone = arthur;");
            
            Player clone = arthur; // 💥 วินาทีนี้ เกิดการก๊อปปี้ร่างโคลน 100% ขึ้นในแรม
            clone.HP = 0;          // เราสั่งฆ่าร่างโคลนทิ้ง

            Console.WriteLine($"[Clone] เลือดร่างโคลน: {clone.HP} (ตายสนิท)");
            Console.WriteLine($"[Main] เลือดต้นฉบับยังคง: {arthur.HP} (ไม่สะเทือนเลย!)");

            // -----------------------------------------------------------
            // การทดลอง B: การส่งผ่าน Method (Pass-by-Value vs Pass-by-Ref)
            // -----------------------------------------------------------
            Console.WriteLine("\n[ทดลอง B] เดินเข้าห้องกับดัก (Method โดนโจมตี)");

            // 1. ส่งเข้า Method ธรรมดา (มันก๊อปปี้อีกแล้ว!)
            FakeDamage(arthur); 
            Console.WriteLine($"[Main] หลังโดน FakeDamage เลือดเหลือ: {arthur.HP} (รอดตัวไป!)");

            // 2. ส่งเข้า Method แบบเจาะมิติ (ใช้ ref นำหน้าทั้งคู่)
            RealDamage(ref arthur);
            Console.WriteLine($"[Main] หลังโดน RealDamage เลือดเหลือ: {arthur.HP} (โดนของจริง!)");
        }

        // Method นี้รับ "ร่างโคลน" เข้ามา (Pass-by-Value)
        static void FakeDamage(Player target)
        {
            target.HP -= 50; // หักเลือดร่างโคลนทิ้ง
            Console.WriteLine($"  > ภายใน FakeDamage: เลือดโดนหักเหลือ {target.HP}");
        } // จบ Method ปุ๊บ ร่างโคลนระเบิดทิ้งไปกับอากาศ!

        // Method นี้รับ "เส้นด้ายชี้เป้า" เข้ามา (Pass-by-Reference)
        static void RealDamage(ref Player target)
        {
            target.HP -= 50; // วิ่งตามเส้นด้าย ไปหักเลือดกล่องต้นฉบับใน Main!
            Console.WriteLine($"  > ภายใน RealDamage: เลือดโดนหักเหลือ {target.HP}");
        }
    }
}
```
:::

### ตัวอย่างที่ 2: ความงุนงงเมื่อเทียบกับ Class (The Paradigm Shift)

มาดูความมหัศจรรย์ของ C# กันครับ! ถ้าเราเปลี่ยนพิมพ์เขียวจาก `struct` กลายเป็น `class` พฤติกรรมทุกอย่างจะกลับตาลปัตร 180 องศาเลย!

::: code-group
```csharp [Program.cs]
using System;

namespace ClassVsStruct
{
    // โครงสร้าง A เป็น Struct (Value Type)
    struct StructBox { public int Number; }

    // โครงสร้าง B เป็น Class (Reference Type)
    class ClassBox { public int Number; }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== บททดสอบ Struct ปะทะ Class ===\n");

            // -----------------------------------------------------------
            // ฝั่ง Struct (ถ่ายเอกสาร)
            // -----------------------------------------------------------
            StructBox s1 = new StructBox { Number = 10 };
            StructBox s2 = s1; // สร้างกล่องใหม่ s2 แล้วก๊อปปี้ค่า 10 มาใส่
            
            s2.Number = 999; // แก้กล่อง s2 ไม่มีผลกับ s1
            
            Console.WriteLine($"Struct s1: {s1.Number}"); // ได้ 10
            Console.WriteLine($"Struct s2: {s2.Number}\n"); // ได้ 999

            // -----------------------------------------------------------
            // ฝั่ง Class (ตั๋วฝากของชี้เป้าเดียวกัน)
            // -----------------------------------------------------------
            ClassBox c1 = new ClassBox { Number = 10 };
            // c1 ไม่ใช่กล่อง! มันคือ "ตั๋วฝากของ" ที่ชี้ไปหาคฤหาสน์ Heap

            ClassBox c2 = c1; 
            // 💥 วินาทีนี้ ไม่มีการก๊อปปี้กล่อง! แต่เป็นการ "ก๊อปปี้ตั๋วฝากของ"
            // แปลว่า c1 และ c2 คือตั๋ว 2 ใบ ที่ชี้ไปเปิดคฤหาสน์ "หลังเดียวกัน!"

            c2.Number = 999; // เอาตั๋ว c2 ไปไขกุญแจแก้ของในคฤหาสน์
            
            Console.WriteLine($"Class c1: {c1.Number}"); // ได้ 999 !! (เพราะมันชี้ไปที่เดียวกัน)
            Console.WriteLine($"Class c2: {c2.Number}"); // ได้ 999
        }
    }
}
```
:::

**ความรู้ระดับสถาปัตยกรรม:** 
- ถ้าคุณทำเกม แล้วกระสุนปืน 10,000 นัด (มีพิกัด X,Y) ใช้ `Class` ระบบจะสร้างคฤหาสน์ Heap 10,000 หลัง... แรมเต็ม เกมกระตุกยับเยิน! 
- แต่ถ้ากระสุนเป็น `Struct` ระบบจะแพ็คใส่ Stack ทันที เกิดดับรวดเร็ว เกมวิ่ง 120 FPS ฉลุย! (นี่คือเหตุผลที่โครงสร้างแกนกลางของ Unity Engine ชื่อว่า DOTS บังคับให้ใช้ Struct ล้วนๆ ครับ)

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: คลินิกรับซ่อมรถ (The Mechanic)**
พิจารณาโค้ดนี้:
```csharp
struct Car { public int Fuel; }

class Program {
    static void Main() {
        Car myCar = new Car { Fuel = 10 };
        AddFuel(myCar);
        Console.WriteLine(myCar.Fuel);
    }
    static void AddFuel(Car c) {
        c.Fuel += 50;
    }
}
```
**คำถาม:** เมื่อรันโปรแกรมนี้ ตัวเลขที่จะออกหน้าจอคืออะไร? (ระหว่าง 10 หรือ 60) และคุณต้องแก้โค้ดอย่างไรให้รถยนต์ได้รับการเติมน้ำมันจริงๆ (ได้ผลลัพธ์เป็น 60)?

### **โจทย์ที่ 2: ฝาแฝดคนละโลก (The Twin Types)**
นาย A สร้าง `class Zombie { public int HP = 100; }`
นาย B สร้าง `struct Skeleton { public int HP = 100; }`
ทั้งคู่เอาออบเจกต์ของตัวเองไปใส่โค้ด `obj2 = obj1;` แล้วสั่งแก้เลือด `obj2.HP = 0;`
**คำถาม:** หลังจากรันโค้ด เลือดของ `obj1` ของฝั่งนาย A และฝั่งนาย B จะเหลือเท่าไหร่? (จงจับคู่ตัวเลขให้ถูกต้อง)

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** ผลลัพธ์คือ **`10`** ครับ! เพราะรถโดนก๊อปปี้ไปเติมน้ำมัน (เติมให้รถโคลนนิ่ง!) วิธีแก้คือต้องใส่ `ref` ทั้งที่พารามิเตอร์ `static void AddFuel(ref Car c)` และตอนเรียกใช้งาน `AddFuel(ref myCar);`
- **สำหรับโจทย์ 2:** 
  ฝั่งนาย A (Class): ได้ **`0`**! เพราะ Class เป็นตั๋วฝากของชี้จุดเดียวกัน แก้แฝดน้อง แฝดพี่ตายด้วย!
  ฝั่งนาย B (Struct): ได้ **`100`**! เพราะ Struct เป็นการถ่ายเอกสาร แก้แฝดน้อง แฝดพี่ยังรอดปลอดภัย!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: เครื่องจำลองการจัดสรรหน่วยความจำ (The Memory Allocator Simulator)**

คุณคือวิศวกรวิเคราะห์หน่วยความจำ! จงอ่านโค้ดด้านล่างนี้ โดยห้ามก๊อปปี้ไปรันใน Visual Studio 
แล้วจดคำตอบลงบนกระดาษว่า หน้าจอจะพิมพ์ผลลัพธ์อะไรออกมาบ้างเรียงตามลำดับ 1-2-3-4!

```csharp
struct BankAccount { public int Money; }
class Vault { public int Money; }

class Program
{
    static void Main()
    {
        BankAccount a1 = new BankAccount { Money = 50 };
        BankAccount a2 = a1;
        a2.Money += 20;
        
        Vault v1 = new Vault { Money = 50 };
        Vault v2 = v1;
        v2.Money += 20;

        Console.WriteLine(a1.Money); // บรรทัดที่ 1
        Console.WriteLine(a2.Money); // บรรทัดที่ 2
        Console.WriteLine(v1.Money); // บรรทัดที่ 3
        Console.WriteLine(v2.Money); // บรรทัดที่ 4
    }
}
```

ถ้าคุณหลับตาแล้วนึกภาพการ "ถ่ายเอกสาร 2 แผ่น" คู่กับการ "ถือตั๋ว 2 ใบชี้ไปตู้เซฟใบเดียวกัน" ออก... คุณตอบข้อนี้ถูก 100% แน่นอนครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในเมื่อการใช้ Struct เวลาโยนข้าม Method มันต้อง "ก๊อปปี้ข้อมูล" ทั้งดุ้น... แปลว่าถ้าเรามี Struct ขนาดยักษ์ (มีตัวแปรซ่อนอยู่ข้างใน 100 ตัว) เวลาเรียก Method โปรแกรมเราจะแครชตายเพราะมัวแต่ถ่ายเอกสารใช่ไหม?
**แนวคำตอบ:** **ถูกต้องที่สุดครับ! สถาปนิกซอฟต์แวร์ประเมินไว้เฉียบขาดมาก!**
ไมโครซอฟต์มี "กฎเหล็ก (Best Practice)" ระบุไว้ชัดเจนว่า: **"จงใช้ Struct กับข้อมูลที่มีขนาดรวมกันไม่เกิน 16 Bytes เท่านั้น!"** (พูดง่ายๆ คือ มีตัวแปร int ยัดอยู่ข้างในไม่เกิน 4 ตัว)
ถ้ายัดของใหญ่กว่านั้น การก๊อปปี้ร่างโคลนจะกินพลังงาน CPU หนักมาก จนทำให้ Class (ที่แค่ก๊อปปี้ตั๋วชี้เป้า 8 Bytes) ทำงานเร็วกว่า! ดังนั้น Struct จึงเหมาะกับของเล็กๆ รวดเร็วๆ อย่าง Point, Color, Quaternion เท่านั้นครับ!

**คำถาม 2:** คำว่า `ref` ดูเจ๋งมาก! ทำไมเราไม่แปะ `ref` ใส่พารามิเตอร์ทุกตัวในจักรวาลไปเลยล่ะ? จะได้ไม่ต้องเปลืองแรมถ่ายเอกสารไง?
**แนวคำตอบ:** **นั่นคือฝันร้ายของการเขียนโค้ดทีมเวิร์คครับ!**
ลองนึกภาพคุณโยน Struct ให้เพื่อนร่วมงานเอาไปคำนวณสถิติเฉยๆ ถ้าเพื่อนคุณเป็นคนซุ่มซ่าม ดันเผลอเขียนโค้ดทับค่าใน Struct ของคุณ... ถ้าคุณเผลอใส่ `ref` ปุ๊บ โค้ดฝั่งคุณพังพินาศตามไปด้วยทันที! (เรียกว่าเกิด Side Effect ระดับรุนแรง) 
การที่ C# ออกแบบให้ก๊อปปี้ร่างโคลน (Pass-by-Value) เป็น Default ก็เพื่อความ "ปลอดภัย" ของข้อมูลต้นฉบับครับ เราจะใช้ `ref` เฉพาะตอนที่ "ตกลงทำสัญญาใจกันแล้วว่าจะยอมให้ Method นี้มีสิทธิ์ในการผ่าตัดข้อมูลต้นฉบับจริงๆ" เท่านั้นครับ!
:::
