# Protected Modifier (ยามประจำตระกูล)

> 💡 **เป้าหมาย:** ทำความรู้จักกับ Access Modifier ตัวที่ 3 ของภาษา C# (`protected`) ซึ่งเป็นรหัสลับที่ถูกสร้างมาเพื่อใช้ในโลกของการสืบทอด (Inheritance) โดยเฉพาะ! มันคือยามที่โหดร้ายกับคนนอก แต่ใจดีกับคนในสายเลือดเดียวกัน

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. จุดบอดของ Private ในโลกครอบครัว

จากบทที่แล้ว เราเรียนรู้ว่าการห่อหุ้มข้อมูลที่ดี (Encapsulation) ตัวแปร Field ทั้งหมดต้องเป็น `private`
แต่เมื่อเราก้าวเข้าสู่โลกของการสืบทอด (Inheritance) ความ `private` กลับกลายเป็นหอกข้างแคร่!

```csharp
class Parent 
{
    private string _familySecret = "สูตรลูกชิ้นปิ้ง";
}

class Child : Parent 
{
    public void Cook() 
    {
        // 💥 พังพินาศ! คอมไพเลอร์จะด่าว่าคุณไม่มีสิทธิ์เข้าถึง _familySecret
        Console.WriteLine($"ทำอาหารด้วย: {_familySecret}"); 
    }
}
```
**เกิดอะไรขึ้น?** 
`private` แปลว่า "ความลับสุดยอดของฉันคนเดียว" ขนาดลูกแท้ๆ ที่สืบทอดสายเลือดมา ก็ยังไม่มีสิทธิ์เอื้อมมือเข้าไปหยิบสูตรนี้ได้! 
ถ้าเราเปลี่ยนมันเป็น `public` ล่ะ? ลูกหยิบได้จริง... แต่คนข้างบ้าน (Main Method) ก็ดันเดินมาขโมยสูตรลูกชิ้นปิ้งไปได้ด้วย! (เสียระบบ Encapsulation อีก)

---

### 2. กำเนิด `protected` (ความลับระดับวงศ์ตระกูล)

เพื่อแก้ปัญหากลืนไม่เข้าคายไม่ออกนี้ ไมโครซอฟต์จึงสร้าง Access Modifier ตัวใหม่ขึ้นมาชื่อว่า **`protected`**

**`protected` แปลว่า "คุ้มครองไว้ให้เฉพาะคนในตระกูล"**
- สำหรับ **คลาสลูก (Derived Class)**: มันจะทำตัวเหมือน `public` (ลูกสามารถหยิบมาใช้ได้หน้าตาเฉย)
- สำหรับ **โลกภายนอก (เช่น Main)**: มันจะทำตัวเหมือน `private` (ซ่อนตัวมิดชิด คนนอกมองไม่เห็น)

```csharp
class Parent 
{
    protected string _familySecret = "สูตรลูกชิ้นปิ้ง"; // เปลี่ยนเป็น protected!
}

class Child : Parent 
{
    public void Cook() 
    {
        // ✅ ผ่านฉลุย! เพราะฉันคือลูกแท้ๆ
        Console.WriteLine($"ทำอาหารด้วย: {_familySecret}"); 
    }
}
// แต่ถ้าใน Main สั่ง: Child c = new Child(); Console.WriteLine(c._familySecret); 
// 💥 พังพินาศ! ยามจะเอาไม้กระบองตีหัวคุณ โทษฐานคนนอกแอบดูความลับตระกูล!
```

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: ระบบคลังอาวุธและตู้เซฟประจำตระกูล (The Family Vault)

โค้ดชุดนี้โชว์ให้เห็นการผสมผสานระหว่าง `private` (เก็บเรื่องส่วนตัวของแม่), `protected` (ของใช้ร่วมกันในตระกูล), และ `public` (ปุ่มให้คนนอกกด)

::: code-group
```csharp [Program.cs]
using System;

namespace ProtectedConcept
{
    // =======================================================
    // BASE CLASS (คลาสแม่)
    // =======================================================
    class BaseCharacter
    {
        public string Name { get; set; }

        // 1. ตัวแปร private: ความลับส่วนตัวของแม่ ลูกห้ามยุ่ง!
        private string _diary = "แอบชอบอัศวินข้างบ้าน"; 

        // 2. ตัวแปร protected: มรดกตระกูล ยอมให้ลูกหลานเบิกไปใช้ได้
        protected int _goldVault = 1000;

        public void ShowDiary()
        {
            // เมธอดนี้อยู่ข้างในคลาสแม่ จึงเอื้อมไปหยิบ _diary มาอ่านได้
            Console.WriteLine($"[ความลับ] บันทึก: {_diary}");
        }
    }

    // =======================================================
    // DERIVED CLASS (คลาสลูก)
    // =======================================================
    class Hero : BaseCharacter
    {
        public void BuySword()
        {
            Console.WriteLine($"\n>> {Name} เดินไปร้านอาวุธเพื่อซื้อดาบราคา 300 ทอง...");

            // 💥 พังพินาศ!: Console.WriteLine(_diary); 
            // ^ ลองเอาคอมเมนต์ออกดูครับ ลูกแอบอ่านไดอารี่แม่ไม่ได้! (Inaccessible)

            // ✅ ลูกสามารถเอื้อมมือเข้าไปหยิบเงินในตู้เซฟตระกูลได้สบายๆ เพราะมันเป็น protected!
            if (_goldVault >= 300)
            {
                _goldVault -= 300;
                Console.ForegroundColor = ConsoleColor.Green;
                Console.WriteLine($"[สำเร็จ] {Name} ซื้อดาบแล้ว! (เงินกงสีตระกูลเหลือ: {_goldVault} ทอง)");
                Console.ResetColor();
            }
        }
    }

    // =======================================================
    // WORLD / EXTERNAL (โลกภายนอก)
    // =======================================================
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบตู้เซฟประจำตระกูล (Protected Modifier) ===\n");

            Hero arthur = new Hero();
            arthur.Name = "อาร์เธอร์";

            // 💥 พังพินาศ!: arthur._goldVault = 99999;
            // ^ คนนอก (Main) พยายามจะเสกเงินเข้าตู้เซฟ! ยาม protected จะทำงาน และฟันหน้าคุณ!

            // การกระทำที่ถูกต้อง ต้องผ่าน Public Method
            arthur.BuySword();
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: นักสืบสายเลือด (Bloodline Detective)**
กำหนดให้มีโค้ด:
```csharp
class Machine {
    protected int Fuel = 100;
}
class Car : Machine {
    public void Drive() { Console.WriteLine("Fuel: " + Fuel); }
}
class Mechanic {
    public void Fix(Car c) {
        c.Fuel = 100; // บรรทัดนี้มีปัญหาอะไรไหม?
    }
}
```
**คำถาม:** ในคลาส `Mechanic` เราสามารถทะลวงเข้าไปเติม `Fuel` ให้กับรถยนต์ตรงๆ ได้หรือไม่? ทำไม? 

### **โจทย์ที่ 2: รหัสลับสี่ตัว (The Access Modifiers)**
ณ ตอนนี้ คุณได้เรียนรู้ Access Modifier มาแล้วถึง 4 ตัว! (ใช่ครับ 4 ตัว ถ้านับค่า Default ด้วย)
จงจับคู่ Modifier เหล่านี้ กับ พฤติกรรมการคุ้มกัน ให้ถูกต้อง:
- [A] `public`
- [B] `private`
- [C] `protected`
- [D] (ไม่ใส่คำนำหน้าอะไรเลย)

1. () มองเห็นแค่ในคลาสตัวเองเท่านั้น ลูกก็มองไม่เห็น
2. () มองเห็นได้ทุกคนในจักรวาล สดใสซาบซ่า
3. () คอมไพเลอร์จะมองว่ามันคือ `private` อัตโนมัติ
4. () มองเห็นในคลาสตัวเอง และลูกหลานที่สืบทอดสายเลือดไป

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **ไม่ได้เด็ดขาดครับ แดงเถือก!**
  เพราะ `Mechanic` ไม่ได้มีเครื่องหมาย `: Machine` (ไม่ได้สืบทอดสายเลือดมา) ดังนั้น Mechanic คือ "คนนอกตระกูล" สำหรับตัวแปร `Fuel` ที่เป็น `protected` จะทำตัวเสมือน `private` บล็อกคนนอกทันทีครับ!
- **สำหรับโจทย์ 2:** 
  1. [B] `private` (เก็บงำความลับส่วนตัว)
  2. [A] `public` (สาธารณะชน)
  3. [D] ลืมใส่คำนำหน้า (C# ปิดประตูตีแมว เป็น `private` ให้เสมอ)
  4. [C] `protected` (สายเลือดเดียวกันเท่านั้น)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ระบบบัญชีเงินเดือนฉบับผู้จัดการ (The Payroll Manager)**

**ภารกิจของคุณ:** จงสร้างระบบจัดการเงินเดือนที่มีการปกป้องข้อมูลยอดเยี่ยม!

1. **คลาสแม่สุด: `Employee` (พนักงานทั่วไป)**
   - มี Property `public string Name { get; set; }`
   - มี Field **`protected decimal _baseSalary;`** (เงินเดือนพื้นฐาน ป้องกันไม่ให้ Main โกงเงินเดือนได้ แต่ยอมให้ลูกหลานเอาไปคำนวณโบนัสต่อได้)
   - มี Constructor รับค่า Name และ Salary มาตั้งค่า
   - มี Method `public void PrintSlip()` ปริ้นข้อความ "[Name] รับเงินเดือนสุทธิ: [_baseSalary] บาท"

2. **คลาสลูก: `Manager` (ผู้จัดการ)** สืบทอดจาก `Employee`
   - มี Constructor รับค่า Name และ Salary (แต่เดี๋ยวก่อน! ตรงนี้คุณต้องใช้วิชาใหม่ที่ยังไม่เคยเรียน... ลองมั่วดู หรือข้ามไปรอเรียนในหัวข้อหน้านะครับ!) *ถ้าไม่อยากงง ให้ลบ Constructor ทิ้ง แล้วปล่อยให้เซ็ตค่าผ่าน Property/Method เอาครับสำหรับข้อนี้*
   - ให้คลาส `Manager` สร้าง Method ยอดฮิต: `public void CalculateBonus()`
   - ใน Method นี้ ให้เอา `_baseSalary` (ซึ่งเป็นมรดกจากแม่) มาบวกเพิ่มไปอีก 50,000 บาท! แล้วปริ้นว่า "ผู้จัดการ [Name] ได้รับโบนัส! ยอดเงินใหม่คือ: [_baseSalary]"

3. **การทดสอบใน Main:**
   - (เนื่องจากข้อนี้เราไม่ได้ทำ Constructor) ให้เสก `Manager m1 = new Manager();`
   - เซ็ตชื่อเป็น "Boss"
   - สร้างฟังก์ชันในคลาสแม่ `public void SetSalary(decimal money)` ไว้เซ็ตเงินตั้งต้นเป็น 30,000
   - เรียก `m1.PrintSlip()` (ควรได้ 30,000)
   - เรียก `m1.CalculateBonus()` (ควรบวกเพิ่มเป็น 80,000)
   - เรียก `m1.PrintSlip()` (ควรได้ 80,000)

**จุดประสงค์ของข้อนี้:** ให้คุณเห็นภาพว่า ถ้า `_baseSalary` ถูกตั้งเป็น `private` ผู้จัดการจะบวกโบนัสให้ตัวเองไม่ได้เลย! (เอื้อมไม่ถึงตัวแปรแม่) `protected` จึงเกิดมาเพื่อสิ่งนี้ครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในโลกของการเขียนโปรแกรมจริง (Real World) เราควรใช้ `protected` กับ Field (ตัวแปร) บ่อยแค่ไหน? 
**แนวคำตอบ:** **สถาปนิกระดับโลกจะเตือนว่า "ใช้ให้น้อยที่สุดเท่าที่จำเป็น!" ครับ**
แม้ `protected` จะดีกว่า `public` แต่การเปิดให้คลาสลูกเอื้อมมือมาล้วงตู้เซฟคลาสแม่ได้ตรงๆ... มันก็แอบทำลายกฎการป้องกันข้อมูล (Encapsulation) ไปกลายๆ ครับ
สมมติแม่มี `protected int _hp` แล้วจู่ๆ มีลูกหลานนรกรุ่นที่ 8 แอบเขียนโค้ด `_hp = -999;`... แม่ก็ตายอเนจอนาถโดยที่ไม่รู้ตัวเลยว่าลูกคนไหนเป็นคนฆ่า! (หาบั๊กยากมาก)
**วิธีปฏิบัติที่ดีที่สุด (Best Practice):** สถาปนิกมักจะให้ Field เป็น `private` ไว้เหมือนเดิม แล้วไปเปิด `protected Property` { get; private set; } เพื่อให้ลูกดึงไปอ่านได้อย่างเดียว แต่ห้ามเขียนทับครับ! (โหดไหมล่ะ!)

**คำถาม 2:** โค้ดด้านล่างนี้ เกิดข้อผิดพลาดอะไร?
```csharp
class Animal { protected void Breath() { } }
class Dog : Animal { }
class Program {
    static void Main() {
        Dog d = new Dog();
        d.Breath(); 
    }
}
```
**แนวคำตอบ:** **แดงเถือกที่บรรทัด d.Breath(); ครับ!**
หลายคนชอบคิดว่า `protected` ใช้ได้เฉพาะกับตัวแปร... ผิดครับ! `protected` ใช้กับ Method ได้ด้วย! 
คลาสแม่สร้าง Method ไว้ว่า "การหายใจเป็นเคล็ดวิชาลับตระกูล" ลูก (`Dog`) สามารถเรียกหายใจได้สบายๆ ภายในคลาสของตัวเอง...
แต่ใน Main คือโลกภายนอก (คนนอก) การที่คนนอกพยายามเดินมาสั่งให้หมาหายใจ... ยาม `protected` จะบล็อกการมองเห็นทันที ราวกับว่า Method นี้ไม่มีอยู่จริงบนโลกครับ!
:::
