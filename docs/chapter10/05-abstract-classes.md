# Abstract Classes (พิมพ์เขียววิญญาณ)

> 💡 **เป้าหมาย:** สัมผัสวิชาขั้นสุดยอดแห่งการบังคับบัญชา! เรียนรู้วิธีสร้าง "คลาสวิญญาณ" ที่ไม่มีตัวตนจริงบนโลก (ห้ามคน `new` เด็ดขาด) แต่มีหน้าที่เป็นพระคัมภีร์ศักดิ์สิทธิ์ ที่บังคับจดจ่อให้ลูกหลานทุกตัวที่เกิดมา ต้องเอาหน้าที่ในคัมภีร์ไปสานต่อให้เสร็จสมบูรณ์!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ปัญหาของคลาสแม่ที่คลุมเครือ (The Vague Base Class Problem)

ในบทที่ผ่านมา เราสร้างคลาส `Animal` ขึ้นมาเป็นคลาสแม่ เพื่อให้นก หมา แมว สืบทอด
คุณเคยคิดไหมครับว่า ถ้าจู่ๆ มีโปรแกรมเมอร์ฝึกงาน พิมพ์คำสั่ง `Animal a = new Animal();` ใน Main... สิ่งที่จะเกิดมาคืออะไร?
**มันคือสัตว์ประหลาดที่ไม่มีอยู่จริงบนโลกครับ!** มันไม่มีรูปร่าง ไม่รู้สายพันธุ์ (คลุมเครือสุดๆ)

แถมถ้าในคลาส `Animal` คุณสร้างเมธอด `virtual void Speak() { Console.WriteLine("???"); }` 
แล้วสมมติว่าคลาส `Cat` ดันขี้เกียจและ "ลืม" เขียน `override`... แมวตัวนั้นก็จะคลอดออกมาและร้อง "???" เป็นสัตว์ประหลาดเอเลี่ยนทันที! ซึ่งระบบที่แข็งแกร่งไม่ควรปล่อยให้เกิดช่องโหว่นี้ครับ!

---

### 2. กำเนิด `abstract` (คลาสวิญญาณ และ กฎเหล็กบังคับ)

วิธีแก้ปัญหาอย่างเด็ดขาดระดับสถาปนิก คือการแปะยันต์ **`abstract`** ไว้หน้าคลาสแม่ครับ!

**คุณสมบัติของ Abstract Class:**
1. **ห้ามจุติเป็นตัวตน (Cannot Instantiate):** ถ้าใครพยายามพิมพ์ `new Animal()` คอมไพเลอร์จะตบหน้าและแจ้ง Error ทันที! (มันมีไว้เพื่อเป็นแม่พิมพ์ถ่ายทอดพันธุกรรมเท่านั้น)
2. **คัมภีร์บังคับลูก (Abstract Methods):** ภายในคลาสวิญญาณ คุณสามารถเขียนเมธอดแบบ "มีแต่ชื่อ ไม่มีไส้ใน" ได้! (จบด้วยเครื่องหมายเซมิโคลอน `;`)
   - มันคือการประกาศกร้าวว่า: *"สัตว์ทุกตัวต้องร้องได้นะ แต่ฉันไม่รู้หรอกว่าร้องยังไง... ใครที่สืบทอดสายเลือดฉันไป **บังคับ** ต้องไปเขียนวิธีร้องเอาเองให้เสร็จ ถ้าไม่เขียน คอมไพเลอร์จะไม่ยอมให้ลูกเกิดเด็ดขาด!"*

**ไวยากรณ์ (Syntax):**
```csharp
// 1. แปะยันต์วิญญาณที่คลาสแม่
abstract class Animal 
{
    // 2. สามารถมีตัวแปรหรือเมธอดปกติที่มีไส้ในปะปนอยู่ได้นะ (ลูกเอาไปใช้ฟรี)
    public void Sleep() { Console.WriteLine("Zzz"); }

    // 3. 🌟 ไฮไลต์: เมธอดไร้ไส้ใน! (บังคับให้ลูกไปเขียนต่อ)
    public abstract void Speak(); 
}

class Dog : Animal 
{
    // 4. ลูก "ถูกบังคับ" ให้ต้อง override ไส้ในให้เสร็จ ไม่งั้น Error แดงเถือก!
    public override void Speak() 
    { 
        Console.WriteLine("โฮ่ง!"); 
    }
}
```

### 3. Abstract VS Virtual (ความต่างของประตู 2 บาน)

เพื่อป้องกันความสับสนตอนสัมภาษณ์งาน จำคติพจน์นี้ไว้ครับ:
- **`virtual` (ประตูบานเปิด):** คลาสแม่ **"มีไส้ใน (Default Behavior)"** เตรียมไว้ให้แล้ว... ลูกจะใช้ของเดิมก็ได้ หรือจะเกรียนเขียน `override` ทับใหม่ก็ได้ (อนุญาตแต่ไม่บังคับ)
- **`abstract` (คัมภีร์เปล่า):** คลาสแม่ **"ไม่มีไส้ในอะไรเลย"**... ลูก **ถูกบังคับคอหอย** ให้ต้องเขียน `override` เติมไส้ในด้วยตัวเองเสมอ 100%! ไม่งั้นห้ามเกิด!

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สถาปัตยกรรมการคำนวณพื้นที่ (The Shape Engine)

นี่คือโปรแกรมวาดภาพรูปทรงเรขาคณิต ซึ่งเป็นตัวอย่างคลาสสิกที่ดีที่สุดของการใช้ Abstract Class เพราะรูปทรงนามธรรม (Shape) มันไม่มีสูตรคำนวณตายตัวครับ! วงกลม สี่เหลี่ยม สูตรมันต่างกันฟ้ากับเหว!

::: code-group
```csharp [Program.cs]
using System;

namespace AbstractConcept
{
    // =======================================================
    // ABSTRACT BASE CLASS (พิมพ์เขียวนามธรรม - เกิดจริงไม่ได้)
    // =======================================================
    abstract class Shape
    {
        public string Color { get; set; }

        public Shape(string color)
        {
            this.Color = color;
        }

        // 🌟 กฎหมายบังคับ: ลูกทุกคนต้องมีฟังก์ชันคำนวณพื้นที่นะ! 
        // (แม่ไม่มีปัญญาเขียนสูตรให้ เพราะไม่รู้ว่าลูกเป็นรูปทรงอะไร)
        public abstract double GetArea();

        // ของฟรี: ฟังก์ชันวาดสี ที่ลูกทุกคนใช้ร่วมกันได้
        public void PrintColor()
        {
            Console.WriteLine($"[ระบบ] ระบายสีพื้นที่ด้วยสี: {Color}");
        }
    }

    // =======================================================
    // DERIVED CLASSES (คลาสลูกรูปธรรม - เอาคัมภีร์มาสานต่อ)
    // =======================================================
    
    // ลูกคนที่ 1: สี่เหลี่ยมผืนผ้า
    class Rectangle : Shape
    {
        public double Width { get; set; }
        public double Height { get; set; }

        public Rectangle(string color, double w, double h) : base(color)
        {
            this.Width = w;
            this.Height = h;
        }

        // 🌟 จำนนต่อกฎหมาย: บังคับ Override สูตรคำนวณพื้นที่
        public override double GetArea()
        {
            return Width * Height; // กว้าง x ยาว
        }
    }

    // ลูกคนที่ 2: วงกลม
    class Circle : Shape
    {
        public double Radius { get; set; }

        public Circle(string color, double r) : base(color)
        {
            this.Radius = r;
        }

        // 🌟 จำนนต่อกฎหมาย: บังคับ Override สูตรคำนวณพื้นที่ (ใช้คณิตศาสตร์ต่างกันสิ้นเชิง)
        public override double GetArea()
        {
            // พาย x r^2
            return Math.PI * Math.Pow(Radius, 2); 
        }
    }

    // =======================================================
    // EXECUTION
    // =======================================================
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบคำนวณพื้นที่เรขาคณิตอัจฉริยะ ===\n");

            // 💥 พังพินาศ!: Shape s = new Shape("Red");
            // แดงเถือกแน่นอน! โลกนี้ไม่มีหรอกรูปทรงโง่ๆ ที่ไม่มีกว้างยาวรัศมีอะไรเลย!

            // การใช้งานจริง (บรรจุลงใน Array Polymorphism!)
            Shape[] drawingBoard = new Shape[2];
            drawingBoard[0] = new Rectangle("สีน้ำเงิน", 5, 10);
            drawingBoard[1] = new Circle("สีแดง", 7);

            // วงลูปคำนวณ!
            foreach (Shape s in drawingBoard)
            {
                s.PrintColor(); // ของฟรีจากแม่
                
                // C# จะวิ่งลงไปดึงสูตร (กว้างxยาว หรือ พายอาร์กำลังสอง) มาให้เองอัตโนมัติ!
                Console.WriteLine($" -> คำนวณพื้นที่ได้: {s.GetArea():F2} ตารางหน่วย\n");
            }
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: คลาสครึ่งผีครึ่งคน (The Mixed State)**
หลายคนสับสนว่า "ถ้าคลาสมีคำว่า `abstract` นำหน้า แปลว่า Method ข้างใน **ทุกตัว** ต้องเป็น `abstract` ไร้ไส้ในทั้งหมดเลยใช่ไหม?"
**คำถาม:** คำกล่าวนี้จริงหรือไม่? เราสามารถเอา Method แบบปกติที่มีปีกกา `{ }` สมบูรณ์ๆ ไปใส่ไว้ใน `abstract class` ได้ไหม? 

### **โจทย์ที่ 2: โซ่ตรวนที่ไม่สมบูรณ์ (The Unfinished Child)**
พิจารณาโค้ดนี้:
```csharp
abstract class Document {
    public abstract void Print();
}
// ลูกคนที่ 1 รับช่วงต่อ แต่ขี้เกียจเขียน override
class PdfDocument : Document {
}
```
**คำถาม:** เกิดอะไรขึ้นกับคลาส `PdfDocument`? คอมไพเลอร์จะยอมให้โค้ดนี้ผ่านหรือไม่? แล้วถ้าเรายืนยันหัวเด็ดตีนขาดว่า "ฉันไม่อยากเขียนไส้ใน Print() ตรงนี้อ่ะ อยากให้ลูกหลานของ PDF ไปเขียนแทน!" เราต้องเติมคำว่าอะไรลงไปหน้าคลาส `PdfDocument`?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **ไม่จำเป็นเลยครับ!**
  Abstract Class อนุญาตให้คุณเขียนผสมกันได้เลย! (ตามตัวอย่างด้านบน) คุณสามารถมีตัวแปรปกติ มี Method ปกติที่ทำงานได้ 100% เอาไว้เป็น "เครื่องมือส่วนกลาง" แจกจ่ายให้ลูกๆ นำไปใช้ฟรีๆ ได้อย่างเต็มที่! มีเพียงแค่บรรทัดที่คุณเติมคำว่า `abstract` เท่านั้นแหละ ที่ลูกถูกบังคับให้ไปแต่งเติมเอาเองครับ (นี่คือข้อแตกต่างสำคัญระหว่าง Abstract Class กับ Interface ที่คุณจะได้เจอในหัวข้อต่อๆ ไป)
- **สำหรับโจทย์ 2:** **แดงเถือกครับ ไม่รอดแน่นอน!**
  คอมไพเลอร์จะด่าว่า `PdfDocument does not implement inherited abstract member Document.Print()` (แกทิ้งภาระไว้ให้ฉันเหรอ!)
  วิธีเดียวที่จะให้โค้ดนี้ผ่านโดยไม่ต้องเขียน `override` คือ... คุณต้อง "สละสิทธิ์การเป็นรูปธรรม" แล้วแปะยันต์ `abstract` ไว้ที่คลาสลูก `PdfDocument` ด้วยครับ! (ยอมกลายเป็นวิญญาณตามแม่ไป) เพื่อผลักภาระไปบังคับลูกหลานรุ่นเหลนให้รับกรรมเขียน `override` ต่อไปครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ระบบจัดการการแจ้งเตือน (The Notification Factory)**

**ภารกิจของคุณ:** บริษัทต้องการระบบยิงข้อความแจ้งเตือน (Notification) หลากหลายช่องทาง โดยห้ามโปรแกรมเมอร์สร้างการแจ้งเตือนแปลกๆ ที่ส่งไม่ได้เด็ดขาด!

1. **คลาสแม่ (วิญญาณ): `Notification`**
   - เป็น `abstract`
   - มี Auto-Property `public string Recipient { get; set; }` (ผู้รับ)
   - มี Constructor รับค่า `recipient`
   - มี **Abstract Method** 1 ตัว: `public abstract void SendMessage(string content);` (คัมภีร์บังคับลูก)

2. **คลาสลูกสายที่ 1: `EmailNotification` (ส่งอีเมล)**
   - สืบทอดจาก `Notification` และใช้ `: base` จัดการผู้รับ
   - 🌟 ภารกิจ: `override` การทำงานให้เสร็จ โดยปริ้นหน้าจอว่า: `"📧 กำลังส่ง Email ไปที่ [Recipient]..."` ตามด้วยเนื้อหา `"ข้อความ: [content]"`

3. **คลาสลูกสายที่ 2: `SmsNotification` (ส่ง SMS)**
   - สืบทอดเหมือนกัน
   - 🌟 ภารกิจ: `override` การทำงาน โดยดักจับก่อนว่า:
     - "ถ้ายาวเกิน 20 ตัวอักษร ให้ปริ้นเตือนว่า ❌ ล้มเหลว SMS ยาวเกินไป!"
     - "ถ้าไม่เกิน ให้ปริ้นว่า 📱 กำลังยิง SMS ไปที่เบอร์ [Recipient]... เนื้อหา: [content]"

4. **การทดสอบใน Main:**
   - สร้าง Array ของ `Notification` จำนวน 2 ช่อง ยัด Email กับ SMS ลงไป
   - ใช้ลูป `foreach` ดึงมาทีละอัน แล้วสั่งเมธอด `SendMessage("ยินดีต้อนรับเข้าสู่ระบบจ้า!")`
   - ลองแกล้งส่งข้อความยาวๆ ให้ SMS ตัวแตกดูด้วยนะ!

ถ้าคุณสามารถสร้างคัมภีร์แม่ และบังคับลูกๆ ให้เขียนช่องทางการยิง Network 2 รูปแบบที่ต่างกันอย่างสุดขั้วได้สำเร็จ... คุณก็พร้อมแล้วสำหรับการทำระบบ Plugin Architecture ระดับเทพแล้วล่ะครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ผมเริ่มสับสนระหว่างเรื่อง `virtual/override` (บทที่แล้ว) กับ `abstract/override` (บทนี้) ครับ... สรุปว่ามันทำหน้าที่แปลงกาย (Polymorphism) ดึงกระบวนท่าของลูกมาใช้ได้เหมือนกันทั้งคู่เลยใช่ไหมครับ?
**แนวคำตอบ:** **ถูกต้อง 100% ครับ! มันคือ Polymorphism ทั้งคู่เลย!**
กลไกการทำงานเบื้องหลัง (Late Binding) เวลาที่รันในลูป `foreach` นั้นทำงานเหมือนกันเป๊ะๆ คอมพิวเตอร์ไม่สนเลยว่าแม่จะเป็น Virtual หรือ Abstract มันขอแค่อ่านเจอคำว่า `override` ที่ตัวลูก มันก็จะดึงของลูกมาใช้ทันที!
เพียงแต่ **จุดประสงค์ด้านสถาปัตยกรรม (Architectural Intent)** มันต่างกันเท่านั้นเอง:
- ใช้ `virtual` เมื่อคุณอยากมี "แผนสำรอง (Default Behavior)" ให้ลูก
- ใช้ `abstract` เมื่อคุณ "ไม่มีแผนสำรองอะไรเลย (No Behavior)" และอยาก "บังคับขู่เข็ญ (Force Implementation)" ให้ลูกรับผิดชอบตัวเองครับ!

**คำถาม 2:** แล้วถ้าคลาส `Abstract` มีไว้ให้คนสืบทอดอย่างเดียว... มันจะสามารถสร้าง Constructor ไว้ข้างใน (เหมือนที่ทำในคลาส Shape) ไปเพื่ออะไรครับ ในเมื่อมันห้ามคนสั่ง `new Shape()` เด็ดขาดอยู่แล้ว? ใครจะมาเป็นคนโทรเรียกมัน?
**แนวคำตอบ:** คำถามคมกริบระดับซีเนียร์ครับ!
คนนอกไม่มีสิทธิ์โทรเรียก `new Shape("Red")` ก็จริง... **แต่ "ลูกแท้ๆ" มีสิทธิ์โทรเรียกครับ!**
จำคีย์เวิร์ด `: base(color)` ได้ไหมครับ? นั่นแหละครับ! Constructor ของคลาส Abstract ถูกสร้างมาเพื่อ "รอคอยให้ลูกๆ โทรมาหาผ่านท่อ base" เพื่อที่แม่จะได้รับพารามิเตอร์มา แล้วเอาไปยัดใส่ตัวแปรส่วนกลาง (เช่น `this.Color = color`) ช่วยลดภาระการเขียนโค้ดซ้ำซ้อนให้ลูกหลานทุกคนครับ! หน้าที่ของแม่ยังคงยิ่งใหญ่เสมอแม้จะกลายเป็นวิญญาณไปแล้วก็ตาม!
:::
