# Inheritance Basics (พันธุกรรมและการสืบทอดสายเลือด)

> 💡 **เป้าหมาย:** ทำความรู้จักกับเสาหลักต้นที่ 3 ของ OOP (Inheritance) เรียนรู้วิธีการลดโค้ดขยะ (Code Reusability) โดยการย้ายจุดร่วมทั้งหมดไปไว้ใน "คลาสแม่ (Base Class)" และปล่อยให้ "คลาสลูก (Derived Class)" ดูดซับความสามารถนั้นมาใช้งานเสมือนเป็นของตัวเอง

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ความสัมพันธ์แบบ "IS-A" (มันคือ... ชนิดหนึ่ง)

ก่อนที่คุณจะสืบทอดสายเลือดอะไร คุณต้องทดสอบตรรกะในโลกความจริงก่อนด้วยคำว่า **"IS-A" (เป็น...ชนิดหนึ่ง)**
- หมา (Dog) สืบทอดจาก สัตว์ (Animal) ได้ไหม? -> **Dog IS-A Animal** (หมาเป็นสัตว์ชนิดหนึ่ง) ✅ รอด!
- รถยนต์ (Car) สืบทอดจาก สัตว์ (Animal) ได้ไหม? -> **Car IS-A Animal** (รถเป็นสัตว์ชนิดหนึ่ง) ❌ ผิดตรรกะโลก!

ในภาษา C# เราเรียกคลาสตัวตั้งต้นว่า **Base Class (คลาสแม่/บรรพบุรุษ)**
และเรียกคลาสที่สืบทอดมาว่า **Derived Class (คลาสลูก/ผู้สืบทอด)**

---

### 2. ไวยากรณ์แห่งการสืบทอด (The Colon Syntax)

วิชาการสืบทอดใน C# นั้นเรียบง่ายและทรงพลังมาก เพียงแค่คุณใช้เครื่องหมายโคลอน `:` หลังชื่อคลาสลูก แล้วตามด้วยชื่อคลาสแม่!

```csharp
// 1. คลาสแม่ (ผู้มีพระคุณ)
class Animal 
{
    public string Name;
    public void Eat() { Console.WriteLine("ง่ำๆๆ"); }
}

// 2. คลาสลูก (ผู้สืบทอด)
// แปลว่า: ฉันคือ Dog ที่มีสายเลือดของ Animal ไหลเวียนอยู่!
class Dog : Animal 
{
    public void Bark() { Console.WriteLine("โฮ่ง!"); }
}
```

เมื่อคุณทำแบบนี้... คลาส `Dog` จะมี Data และ Method ของตัวเองคือ `Bark()` ผสมรวมกับของบรรพบุรุษคือ `Name` และ `Eat()` ทันที!

---

### 3. กฎเหล็ก "มีพ่อได้คนเดียว" (Single Inheritance)

ภาษาคอมพิวเตอร์แต่ละภาษา มีปรัชญาที่ต่างกัน:
- ภาษาอย่าง C++ หรือ Python ยอมให้คุณสืบทอดจากคลาสแม่หลายตัวพร้อมกันได้ (Multiple Inheritance) ซึ่งมักจะก่อให้เกิดปัญหาดราม่าที่เรียกว่า "The Diamond Problem" (สมมติสืบทอดจากพ่อและแม่ ถ้าพ่อและแม่มี Method หายใจเหมือนกัน... ตกลงลูกจะหายใจด้วยจมูกพ่อหรือจมูกแม่ล่ะ!?)
- **แต่ในภาษา C# และ Java...** สถาปนิกไมโครซอฟต์ตัดปัญหานี้ทิ้งตั้งแต่ต้นขั้ว โดยตั้งกฎหมายว่า: **"คลาส 1 คลาส สามารถสืบทอดจากคลาสแม่ได้เพียงคลาสเดียวเท่านั้น!"** (มีบรรพบุรุษสายตรงได้เส้นเดียว)

*(ถ้าคุณอยากได้ความสามารถจากหลายๆ แหล่ง C# จะมีเครื่องมือชื่อ `Interface` มาให้ใช้แทน ซึ่งเราจะได้เจอในอนาคตครับ)*

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สวนสัตว์จำลอง (The Zoo Simulator)

เรามาดูแสนยานุภาพของการเขียนโค้ดแบบ DRY (Don't Repeat Yourself) ด้วยการแยกสัตว์ 2 ชนิดออกจากกัน

::: code-group
```csharp [Program.cs]
using System;

namespace InheritanceConcept
{
    // =======================================================
    // BASE CLASS (คลาสแม่)
    // เก็บเฉพาะสิ่งที่ "สัตว์ทุกตัวบนโลก" ต้องมีเหมือนกัน
    // =======================================================
    class Animal
    {
        public string Name { get; set; }
        
        public void Eat()
        {
            Console.WriteLine($"[ระบบ] {Name} กำลังกินอาหาร...");
        }

        public void Sleep()
        {
            Console.WriteLine($"[ระบบ] {Name} นอนหลับ Zzz...");
        }
    }

    // =======================================================
    // DERIVED CLASS 1 (คลาสลูก: นก)
    // =======================================================
    class Bird : Animal // สืบทอดจาก Animal
    {
        public double WingSpan { get; set; } // ความยาวปีก (นกมี แต่สัตว์อื่นอาจจะไม่มี)

        public void Fly()
        {
            Console.WriteLine($"🦅 {Name} กางปีกกว้าง {WingSpan} ซม. โผบินขึ้นสู่ท้องฟ้า!");
        }
    }

    // =======================================================
    // DERIVED CLASS 2 (คลาสลูก: แมว)
    // =======================================================
    class Cat : Animal // สืบทอดจาก Animal
    {
        public bool IsNinja { get; set; } // แมวเป็นนินจาไหม? (เงียบเชียบ)

        public void Meow()
        {
            Console.WriteLine($"🐈 {Name} ร้องเหมียวๆ อ้อนขอปลาทู");
        }
    }

    // =======================================================
    // PROGRAM EXECUTION
    // =======================================================
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบจัดการสวนสัตว์จำลอง ===\n");

            // 1. สร้างออบเจกต์ นกอินทรี
            Bird eagle = new Bird();
            
            // 🌟 เวทมนตร์: eagle สามารถเข้าถึง Property Name ได้ (ทั้งๆ ที่คลาส Bird ไม่เคยเขียนตัวแปร Name เลย!)
            eagle.Name = "พญาอินทรี";
            eagle.WingSpan = 150.5;

            // 2. สร้างออบเจกต์ แมวนินจา
            Cat garfield = new Cat();
            garfield.Name = "การ์ฟิลด์";
            garfield.IsNinja = false;

            // 3. ทดสอบการใช้ความสามารถของบรรพบุรุษ (Shared Behavior)
            Console.WriteLine(">> ถึงเวลาให้อาหารสัตว์:");
            eagle.Eat();    // ดึงเมธอดจากคลาสแม่มาใช้
            garfield.Eat(); // ดึงเมธอดจากคลาสแม่มาใช้

            // 4. ทดสอบความสามารถเฉพาะสายพันธุ์ (Specific Behavior)
            Console.WriteLine("\n>> แสดงพฤติกรรมเฉพาะตัว:");
            eagle.Fly();
            garfield.Meow();

            // 💥 พังพินาศ!: garfield.Fly();
            // คอมไพเลอร์ฉลาดพอที่จะรู้ว่า "แมวบินไม่ได้!" 
            // เพราะ Fly() เป็นสมบัติส่วนตัวของตระกูล Bird ไม่ได้อยู่ในคลาสแม่ Animal ครับ
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: ตรรกะวิบัติ (The Broken IS-A)**
มีโปรแกรมเมอร์คนนึงเขียนโค้ดนี้:
```csharp
class Engine { public void Start() { } }
class Car : Engine { public string Brand; }
```
เขาหวังว่ารถยนต์ของเขาจะใช้ความสามารถ `Start()` ของเครื่องยนต์ได้
**คำถาม:** โค้ดนี้ Compile ผ่านหรือไม่? แล้วในเชิง "สถาปัตยกรรมซอฟต์แวร์ (IS-A Relationship)" โค้ดนี้ถูกต้องหรือไม่? 

### **โจทย์ที่ 2: ห่วงโซ่อาหาร (The Inheritance Chain)**
สมมติว่าคุณมีคลาสเรียงต่อกันเป็นทอดๆ:
```csharp
class Shape { public int Area; }
class Rectangle : Shape { public int Width; }
class Square : Rectangle { public int Edge; }
```
**คำถาม:** ถ้าใน Main คุณสั่ง `Square s = new Square();` ตัวแปรออบเจกต์ `s` จะมี Property ให้เรียกใช้งานทั้งหมดกี่ตัว? อะไรบ้าง?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  **Compile ผ่าน 100% ครับ! คอมพิวเตอร์ไม่ได้ฉลาดเรื่องตรรกะมนุษย์** 
  **แต่ในเชิงสถาปัตยกรรม นี่คือความพินาศครับ!** เพราะรถยนต์ (Car) ไม่ใช่ (IS-NOT-A) เครื่องยนต์ (Engine) เป็นต้น! รถยนต์ไม่ได้ "สืบทอด" จากเครื่องยนต์ แต่รถยนต์ "มี" เครื่องยนต์อยู่ข้างใน (เรียกว่าความสัมพันธ์แบบ HAS-A หรือ Composition) วิธีที่ถูกคือ ต้องเอาคลาส Engine ไปประกาศเป็น Property ตัวนึงอยู่ข้างในคลาส Car ครับ!
- **สำหรับโจทย์ 2:** 
  มี **3 ตัว** ครับ! คือ `Area`, `Width`, และ `Edge` 
  นี่คือจุดเด่นของ C# พันธุกรรมจะถูกส่งต่อเป็นทอดๆ (Chain) เหมือนปู่ -> พ่อ -> ลูก ความสามารถจะถูกควบรวม (Accumulate) ก้อนใหญ่ขึ้นเรื่อยๆ ไปตามสายเลือดครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: วิศวกรโรงงานผลิตรถยนต์ (The Vehicle Factory)**

**ภารกิจของคุณ:** จงสร้างผังครอบครัวยานพาหนะตามลำดับชั้นดังนี้:

1. **คลาสแม่สุด (Base Class): `Vehicle` (ยานพาหนะ)**
   - มี `public string Brand { get; set; }`
   - มี `public int Speed { get; set; }`
   - มี `public void Honk()` ปริ้นว่า "[Brand] บีบแตร ปี๊นๆ!"

2. **คลาสลูกสายที่ 1: `Car` (รถยนต์)** สืบทอดจาก `Vehicle`
   - มี `public int NumberOfDoors { get; set; }`
   - มี `public void TurnOnAC()` ปริ้นว่า "เปิดแอร์เย็นฉ่ำในรถ [NumberOfDoors] ประตู"

3. **คลาสลูกสายที่ 2: `Motorcycle` (มอเตอร์ไซค์)** สืบทอดจาก `Vehicle`
   - มี `public bool HasSidecar { get; set; }` (มีพ่วงข้างไหม)
   - มี `public void Wheelie()` ปริ้นว่า "ยกล้อโชว์สาว! ซิ่งด้วยความเร็ว [Speed] กม./ชม."

4. **การทดสอบใน Main:**
   - สร้างออบเจกต์ Car ยี่ห้อ "Honda" มี 4 ประตู ความเร็ว 120
   - สร้างออบเจกต์ Motorcycle ยี่ห้อ "Yamaha" ไม่มีพ่วงข้าง ความเร็ว 150
   - สั่งให้รถยนต์ "บีบแตร" และ "เปิดแอร์"
   - สั่งให้มอเตอร์ไซค์ "บีบแตร" และ "ยกล้อ"

ถ้าคุณทำได้แปลว่าคุณสามารถแบ่งหมวดหมู่ (Taxonomy) ของ Data ในโลกแห่งความจริงออกมาเป็นแบบจำลองเชิงวัตถุได้อย่างสมบูรณ์แบบครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในอดีตมีโปรแกรมเมอร์หัวใสคนนึง เห็นว่าคลาส `DatabaseConnection` มันมีเมธอด `Connect()` ที่ดีมาก เขาเลยให้คลาส `Customer` ไปสืบทอด (Inherit) มาจากคลาส `DatabaseConnection` ซะเลย... เพื่อที่ลูกค้าจะได้มีเมธอด `Connect()` ติดตัวไปใช้เชื่อมดาต้าเบสได้ฟรีๆ การทำแบบนี้ฉลาดหรือไม่?
**แนวคำตอบ:** **เป็นความคิดที่โง่เง่าที่สุดในวงการสถาปัตยกรรมซอฟต์แวร์ครับ! (Antipattern)**
จำกฎ "IS-A" ได้ไหมครับ? ลูกค้า (Customer) "เป็น...ชนิดหนึ่ง" ของ ฐานข้อมูล (DatabaseConnection) หรือไม่? คำตอบคือไม่ใช่เลย! 
การฉวยโอกาสสืบทอดพันธุกรรมเพียงเพราะ "อยากได้เมธอดฟรีๆ มาใช้" จะทำให้ผังครอบครัวพินาศ (Spaghetti Code) และทำให้คนที่มาอ่านโค้ดต่อปวดหัวมาก เพราะจู่ๆ ลูกค้าก็มีความสามารถในการยิงคิวรี่ SQL ได้ซะงั้น!
วิธีที่ถูกต้องคือ การประกอบร่าง (Composition) โดยการยัดออบเจกต์ Database เข้าไปเป็นตัวแปรหนึ่งในคลาส Customer ครับ! 

**คำถาม 2:** โค้ดคลาสของ C# ทุกๆ ตัวบนโลก (แม้แต่คลาสที่คุณเพิ่งสร้างเมื่อกี้) ล้วนสืบทอดมาจากบรรพบุรุษต้นตระกูลเดียวกันทั้งหมด คุณรู้หรือไม่ว่าบรรพบุรุษผู้นั้นชื่ออะไร?
**แนวคำตอบ:** บรรพบุรุษสูงสุดของจักรวาล C# คือคลาสที่ชื่อว่า **`System.Object`** ครับ!
แม้ว่าคุณจะเขียนแค่ `class Dog { }` โดยไม่ได้ใส่เครื่องหมาย `:` ตามด้วยอะไรเลย... เบื้องหลัง (Compiler) มันจะแอบเติม `: Object` ให้คุณโดยอัตโนมัติ!
นี่คือสาเหตุที่ทำไมไม่ว่าคุณจะสร้างคลาสอะไรขึ้นมาก็ตาม พอคุณพิมพ์จุด (`.`) มันจะมีเมธอดโง่ๆ 4 ตัวโผล่มาให้ใช้เสมอ (เช่น `.ToString()`, `.Equals()`, `.GetHashCode()`) เพราะคุณได้รับมรดกชิ้นนี้มาจากบรรพบุรุษ `Object` นั่นเองครับ!
:::
