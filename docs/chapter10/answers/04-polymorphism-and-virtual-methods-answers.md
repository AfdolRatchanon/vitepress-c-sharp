# เฉลย: Polymorphism & Virtual (เวทมนตร์การแปลงกายและกบฏ)

## 🎯 เฉลย Mini Exercise

### **โจทย์ที่ 1: อันตรายของคำว่า `new` Method (The Shadow Hiding)**
**เป้าหมาย:** สัมผัสความสยดสยองของการเขียนโค้ดทับแบบปลอมๆ (Shadowing) ซึ่งจะทำให้ระบบ Polymorphism พังพินาศ

**เฉลยคำตอบระดับสถาปนิก:**
ผลลัพธ์ที่พิมพ์ออกมาคือ **`I am your father.`** (คำสั่งของพ่อทำงานครับ!)
**เหตุผล:** 
- ในโลกของ C# ถ้าคุณเอาออบเจกต์ยัดใส่กล่องชนิดไหน (บรรทัด `Parent p = ...` แปลว่าจับยัดใส่กล่อง `Parent`) คอมไพเลอร์จะมองออบเจกต์ชิ้นนั้นเป็น "พ่อ" ทันที (Early Binding)
- การที่คุณใช้คีย์เวิร์ด `new` หน้าเมธอด มันเป็นเพียงแค่วิชาลวงตา (Shadowing)... พอมันอยู่ในกล่องของพ่อ คอมพิวเตอร์ก็จะเรียกเมธอดของพ่อมาทำงานหน้าตาเฉยเลย! โค้ดที่ลูกพยายามเขียนมาทั้งหมดสูญเปล่าทันที!
**วิธีแก้เด็ดขาด:** ต้องลบ `new` ทิ้ง! แล้วเปลี่ยนพ่อให้เป็น `virtual` จากนั้นให้ลูกใช้ `override` ครับ! (เมื่อใช้ override ต่อให้คุณจับยัดใส่กล่องของปู่ย่าตาทวด คอมพิวเตอร์ก็จะมุดกล่องลงไปหา "ตัวตนที่แท้จริง" ของลูกขึ้นมาทำงานได้สำเร็จเสมอ! นี่คือ Late Binding ของจริงครับ!)

### **โจทย์ที่ 2: ผ่าตัดกระบวนท่าเสริม (The Base Expansion)**
**เป้าหมาย:** ประยุกต์ใช้วิชากระโดดข้ามมิติร่วมกับการก่อกบฏ

**เฉลย:**
วิชาที่ต้องใช้คือคีย์เวิร์ด **`base.`** ครับ! 
(จำเอาไว้ว่า: `this` คือเรียกหาของตัวเอง, ส่วน `base` คือเรียกหาเวอร์ชันเก่าดั้งเดิมของบรรพบุรุษ)
```csharp
public override void Speak() 
{
    // แอบไปเรียกเสียงเก่าของบรรพบุรุษมาทำงานให้เสร็จก่อน
    base.Speak(); 
    
    // ค่อยแถมพฤติกรรมเสริมของตัวเองต่อท้าย
    Console.WriteLine("และฉันคือผู้กอบกู้จักรวาล!"); 
}
```

## 🔥 เฉลย Challenge (ระบบจ่ายเงินพนักงาน 3 ระดับ The Polymorphic Payroll)

**เป้าหมาย:** กำจัด `if/else` นรกที่ใช้เช็คประเภทพนักงานทิ้งให้เกลี้ยง! แล้วปล่อยให้ความขี้เกียจของ Polymorphism ทำงานจัดการเงินเดือนให้เอง!

**กระบวนการคิดระดับสถาปนิก (Open-Closed Principle):**
นี่คือเสาหลักของลัทธิ SOLID ครับ! ถ้าระบบของคุณเป็น Polymorphism... วันหน้าถ้าบริษัทมี "พนักงานรับเหมา" เพิ่มขึ้นมา คุณแค่สร้างคลาสใหม่มาสืบทอดแค่นั้นจบเลย! โค้ดส่วนคำนวณเงินเดือนใน Main ไม่ต้องแก้แม้แต่ตัวอักษรเดียว!

```csharp
using System;

namespace PolymorphicPayroll
{
    // =======================================================
    // 1. BASE CLASS (คลาสพนักงานพื้นฐาน)
    // =======================================================
    class Employee
    {
        public string Name { get; set; }
        public decimal BaseSalary { get; set; }

        public Employee(string name, decimal salary)
        {
            Name = name;
            BaseSalary = salary;
        }

        // 🌟 ประตู VIRTUAL: คืนค่าเงินเดือนพื้นฐาน (เผื่อใครขี้เกียจแก้ ก็จะได้ยอดนี้เลย)
        public virtual decimal CalculatePay()
        {
            return BaseSalary;
        }
    }

    // =======================================================
    // 2. DERIVED CLASS (คลาสเซลล์แมน)
    // =======================================================
    class Salesman : Employee
    {
        public decimal Commission { get; set; }

        // โยน 2 ค่าแรกให้แม่จัดการ, เก็บ 1 ค่าไว้ของตัวเอง
        public Salesman(string n, decimal s, decimal comm) : base(n, s)
        {
            Commission = comm;
        }

        // 🌟 OVERRIDE: แปลงร่างสูตรเงินเดือน (เงินก้อนหลัก + คอมมิชชัน)
        public override decimal CalculatePay()
        {
            return BaseSalary + Commission;
        }
    }

    // =======================================================
    // 3. DERIVED CLASS (คลาสผู้จัดการ)
    // =======================================================
    class Manager : Employee
    {
        public decimal Bonus { get; set; }

        public Manager(string n, decimal s, decimal bonus) : base(n, s)
        {
            Bonus = bonus;
        }

        // 🌟 OVERRIDE: แปลงร่างสูตรเงินเดือน (เงินก้อนหลัก + โบนัสก้อนโต)
        public override decimal CalculatePay()
        {
            return BaseSalary + Bonus;
        }
    }

    // =======================================================
    // EXECUTION
    // =======================================================
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบคำนวณรายจ่ายบริษัท (Polymorphism) ===\n");

            // 1. นำพนักงานทุกระดับ มายัดใส่รวมกันใน "กล่องของคลาสแม่ (Employee Array)" (Upcasting)
            Employee[] staff = new Employee[3];
            
            staff[0] = new Employee("จอห์น (ภารโรง)", 15000);                // ไม่มีออปชันเสริม
            staff[1] = new Salesman("อลิซ (นักขายทองคำ)", 20000, 50000);   // มีค่าคอม 5 หมื่น
            staff[2] = new Manager("บอส (หัวหน้าใหญ่)", 50000, 100000);      // มีโบนัส 1 แสน

            // 2. ตัวสะสมยอดรวม
            decimal totalCompanyExpense = 0;

            // 3. 🌟 ไฮไลต์ของโลก OOP: วาทยกรสั่งการด้วยลูปเดียว!
            foreach (Employee emp in staff)
            {
                // C# Runtime จะมุดลงไปดูไส้ใน ว่าแต่ละคนสืบทอดมาจากอะไร 
                // แล้วหยิบสูตรคณิตศาสตร์ที่ถูกต้องของคนนั้นมาคำนวณให้ทันที! (Late Binding)
                decimal pay = emp.CalculatePay(); 
                
                Console.WriteLine($"- โอนเงินให้คุณ {emp.Name}: {pay:N0} บาท");

                // เอาไปบวกสะสมในยอดรวมบริษัท
                totalCompanyExpense += pay;
            }

            Console.WriteLine("---------------------------------------------");
            Console.WriteLine($"💰 สรุปรายจ่ายบริษัทเดือนนี้รวม: {totalCompanyExpense:N0} บาท!");
            Console.WriteLine("---------------------------------------------");
        }
    }
}
```

**สิ่งที่ได้เรียนรู้ (Architecture Lesson):**
ถ้าเป็นโค้ดของมือใหม่ (Procedural) ลูป `foreach` ด้านบนจะกลายสภาพเป็นหน้าตาแบบนี้ครับ:
```csharp
if (emp is Salesman) {
    // เอาฐานบวกค่าคอม
} else if (emp is Manager) {
    // เอาฐานบวกโบนัส
} else { ... }
```
เห็นความอุบาทว์ไหมครับ? ถ้าบริษัทเรามีพนักงาน 20 ประเภท (ช่างเทคนิค, ฟรีแลนซ์, พาร์ทไทม์...) เราต้องมานั่งไล่แก้ `if/else` 20 ชั้นในทุกจุดที่มีการจ่ายเงิน! นี่คือเหตุผลว่าทำไมบริษัทชั้นนำถึงบังคับให้ทุกคนต้องใช้ Polymorphism ครับ!
