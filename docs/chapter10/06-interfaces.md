# Interfaces (สัญญารับเหมาสุดโหด)

> 💡 **เป้าหมาย:** สัมผัสจิ๊กซอว์ชิ้นสุดท้ายของ OOP! เรียนรู้วิธีปลดล็อกข้อจำกัด "สืบทอดสายเลือดได้แค่เส้นเดียว (Single Inheritance)" ของ C# โดยใช้ **Interface (อินเทอร์เฟซ)** ซึ่งเปรียบเสมือนใบสัญญารับเหมา ที่บังคับให้ผู้รับเหมาทุกคนต้องมีคุณสมบัติตามที่ระบุไว้เป๊ะๆ ถึงจะเซ็นสัญญาได้!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. วิกฤติพันธุกรรม (The Diamond Problem)

จากที่เรียนมา C# มีกฎเหล็กว่า: **"คลาส 1 คลาส มีแม่ได้แค่ 1 คนเท่านั้น"**
สมมติคุณมีคลาส `Machine` (เครื่องจักร) และคลาส `Animal` (สัตว์) 
ถ้าคุณอยากสร้างออบเจกต์ **`Cyborg` (ไซบอร์กครึ่งคนครึ่งจักรกล)** คุณจะเขียน `: Animal, Machine` ไม่ได้เด็ดขาด! คอมไพเลอร์จะตบหน้าคุณทันที!

แต่ในโลกความจริง สิ่งของหนึ่งชิ้นมันมักจะมีความสามารถหลายอย่างปะปนกัน (เช่น โดรน เป็นทั้งยานพาหนะ, เป็นทั้งกล้องถ่ายรูป, และเป็นอุปกรณ์บินได้) เราจะแก้ปัญหานี้ยังไง?

### 2. กำเนิด Interface (สัญญารับเหมา)

**Interface** ถูกสร้างมาเพื่ออุดช่องโหว่นี้ครับ! 
มันไม่ใช่การสืบทอดสายเลือด (IS-A) แต่มันคือ **"พฤติกรรม (Behavior) หรือ สัญญา (Contract)"** ที่ออบเจกต์บอกโลกว่า "ฉันทำแบบนี้ได้นะ!"

**คุณสมบัติของ Interface:**
1. **ชื่อต้องนำหน้าด้วยอักษรตัวไอ `I` เสมอ** (เช่น `IFlyable`, `IDriveable`, `ICamera`) เพื่อให้โปรแกรมเมอร์ดูปุ๊บรู้ปั๊บว่านี่ไม่ใช่คลาสปกติ!
2. **ห้ามมีไส้ในเด็ดขาด! (กะโหลกล้วนๆ 100%)** มันคล้ายกับ `abstract class` มาก แต่โหดกว่า! ใน Interface คุณห้ามเขียนตัวแปรปกติ ห้ามเขียนปีกกา `{}` ใส่ลอจิกเด็ดขาด มีได้แค่ชื่อฟังก์ชันเปล่าๆ เท่านั้น
3. **1 คลาส สามารถเซ็นสัญญากี่ใบก็ได้! (Multiple Implementation)**

**ไวยากรณ์ (Syntax):**
```csharp
// 1. ประกาศใบสัญญา (ใช้คีย์เวิร์ด interface แทน class)
interface IFlyable 
{
    // ไม่ต้องใส่ public ไม่ต้องใส่ abstract เพราะมันบังคับเป็นของมันเองอยู่แล้ว!
    void Fly(); 
    void Land();
}

interface ICamera 
{
    void TakePhoto();
}

// 2. การเซ็นสัญญารับเหมา (คลาสสืบทอดแม่ได้ 1 คน, แต่เซ็นสัญญาได้ไม่จำกัด!)
class Drone : Vehicle, IFlyable, ICamera
{
    // 3. จำนนต่อสัญญา: ต้องเขียนไส้ในให้ครบทุกฟังก์ชันที่ระบุไว้ในสัญญา!
    public void Fly() { Console.WriteLine("บินขึ้น!"); }
    public void Land() { Console.WriteLine("ลงจอด!"); }
    public void TakePhoto() { Console.WriteLine("แชะ!"); }
}
```

### 3. พลังที่แท้จริง: สวมหน้ากาก Interface (Polymorphism)

ความน่ากลัว (ในทางที่ดี) ของ Interface คือ **มันสามารถใช้เป็นกล่องรับออบเจกต์ (Polymorphism)** ได้เหมือนคลาสแม่เลยครับ!

สมมติคุณมี `Bird` และ `Drone` ซึ่ง 2 คลาสนี้อยู่คนละสายเลือดกันอย่างสิ้นเชิง (นกสืบทอดจากสัตว์, โดรนสืบทอดจากเครื่องจักร)
แต่ถ้าทั้งคู่เซ็นสัญญา **`IFlyable`** ไว้... คุณสามารถจับมันยัดลงกล่อง `IFlyable` ใบเดียวกันได้!

```csharp
// ไม่แคร์สายเลือด ขอแค่บินได้ ก็จับยัดลง Array นี้ได้หมด!
IFlyable[] skyObjects = new IFlyable[2];
skyObjects[0] = new Bird();
skyObjects[1] = new Drone();

foreach (IFlyable obj in skyObjects) {
    obj.Fly(); // วาทยกรสั่งลูปเดียว นกกับโดรนบินขึ้นฟ้าพร้อมกัน!
}
```

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สถาปัตยกรรมเกมแอคชัน (The Damage System)

นี่คือระบบเกมของจริง! ในเกมแอคชัน เวลาระเบิดตกลงพื้น ทุกอย่างในรัศมีจะพังหมด ไม่ว่าจะเป็น ผู้เล่น, ศัตรู, หรือแม้กระทั่ง "ถังน้ำมัน" 
ถ้าเราใช้ Class สืบทอด เราคงไม่ได้จับถังน้ำมันไปสืบทอดจากคลาส `Character` แน่ๆ... ทางออกที่ดีที่สุดคือการใช้ **Interface `IDamageable`** ครับ!

::: code-group
```csharp [Program.cs]
using System;

namespace InterfaceConcept
{
    // =======================================================
    // INTERFACE (สัญญารับเหมา: ใครก็ตามที่สามารถ "รับความเสียหาย" ได้)
    // =======================================================
    interface IDamageable
    {
        // บังคับว่าต้องมีค่าพลังชีวิต
        int HP { get; set; } 
        
        // บังคับว่าต้องมีท่ารับดาเมจ
        void TakeDamage(int amount); 
    }

    // =======================================================
    // CONCRETE CLASSES (คนเซ็นสัญญา - สายเลือดต่างกันสุดขั้ว)
    // =======================================================
    
    // 1. ผู้เล่น (สิงมีชีวิต)
    class Player : IDamageable
    {
        public int HP { get; set; } = 100;
        
        public void TakeDamage(int amount)
        {
            HP -= amount;
            Console.WriteLine($"[ผู้เล่น] ร้องอ๊าก! โดนดาเมจ {amount} (เลือดเหลือ {HP})");
        }
    }

    // 2. ถังน้ำมัน (สิ่งของ)
    class ExplosiveBarrel : IDamageable
    {
        public int HP { get; set; } = 30; // ถังน้ำมันมีเลือด 30

        public void TakeDamage(int amount)
        {
            HP -= amount;
            Console.WriteLine($"[ถังน้ำมัน] บุบสลาย! (ความทนทานเหลือ {HP})");

            if (HP <= 0)
            {
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("💥 ตูมมมม! ถังน้ำมันระเบิดเป็นจุณ!!");
                Console.ResetColor();
            }
        }
    }

    // =======================================================
    // EXECUTION
    // =======================================================
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ทดสอบระบบฟิสิกส์ในเกม (Explosion Area) ===\n");

            // 🌟 เวทมนตร์: สร้างระเบิดวงกว้าง (AOE)
            // อะไรก็ตามที่อยู่ในรัศมี และ "เซ็นสัญญา IDamageable" จะถูกยัดลง Array นี้
            IDamageable[] targetsInBlastRadius = new IDamageable[3];
            
            targetsInBlastRadius[0] = new Player();
            targetsInBlastRadius[1] = new ExplosiveBarrel(); // ถังใบที่ 1
            targetsInBlastRadius[2] = new ExplosiveBarrel(); // ถังใบที่ 2

            Console.WriteLine(">> 💣 ขว้างระเบิดมือ!! (ดาเมจกระจาย 40 หน่วย)");

            // วงลูปเดียวสะเทือนทั้งแมพ!
            foreach (IDamageable target in targetsInBlastRadius)
            {
                // C# ไม่แคร์ว่าเป้าหมายคือคนหรือสิ่งของ มันรู้แค่ว่าเป้าหมายนี้ "มีเมธอด TakeDamage()" ชัวร์ๆ
                target.TakeDamage(40);
            }
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: วิญญาณ ปะทะ สัญญา (Abstract VS Interface)**
ในการสัมภาษณ์งานโปรแกรมเมอร์ คำถามยอดฮิต 100% คือ: **"Abstract Class กับ Interface ต่างกันอย่างไร ในเมื่อมันก็บังคับให้ลูกเขียนโค้ดเหมือนกัน?"**
**คำถาม:** จงหาความแตกต่างที่เป็น "ข้อจำกัด" ที่สำคัญที่สุด 2 ข้อ ที่ทำให้ 2 อย่างนี้ต่างกันสิ้นเชิง

### **โจทย์ที่ 2: ด่านตรวจคนเข้าเมืองสุดโหด (The Stubborn Contractor)**
พิจารณาใบสัญญานี้:
```csharp
interface IWorker {
    void Work();
    void Rest();
}

class Robot : IWorker {
    public void Work() { Console.WriteLine("ทำงาน 24 ชม."); }
}
```
**คำถาม:** โค้ดด้านบน Compile ผ่านหรือไม่? เกิดข้อผิดพลาดอะไรขึ้น? แล้วถ้าหุ่นยนต์ไม่จำเป็นต้องพักผ่อน เราจะแก้ปัญหานี้ให้มัน Compile ผ่านได้อย่างไร?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  1. **สิทธิ์การสืบทอด:** คลาสลูกสืบทอด `Abstract class` ได้ **คนเดียว** แต่สามารถเซ็นสัญญา `Interface` ได้ **ไม่จำกัดจำนวน!**
  2. **ไส้ในโค้ด:** `Abstract class` อนุญาตให้คุณเขียนตัวแปรปกติ หรือ Method ทั่วไปผสมลงไปได้ (แม่มีไส้ใน) แต่ `Interface` เป็นสัญญาแห้งๆ **ห้ามมีลอจิกไส้ในใดๆ ทั้งสิ้น 100%**
- **สำหรับโจทย์ 2:** **แดงเถือกครับ ไม่รอด!**
  คอมไพเลอร์จะฟ้องว่า `Robot does not implement interface member 'IWorker.Rest()'`
  กฎของการเซ็นสัญญาคือ **"เมื่อเซ็นแล้ว ต้องรับเหมาทำทุกข้อ!"** ห้ามทำครึ่งๆ กลางๆ เด็ดขาด! 
  วิธีแก้: ต่อให้หุ่นยนต์ไม่ต้องพัก คุณก็ต้องเขียนเมธอด `public void Rest() { }` เปล่าๆ ทิ้งไว้ให้มันครบตามสัญญาครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ระบบบันทึกข้อมูลแบบปลั๊กอิน (The Data Logger System)**

**ภารกิจของคุณ:** บริษัทต้องการระบบเก็บประวัติการทำงาน (Logs) โดยที่บางลูกค้าอยากให้ปริ้นลงหน้าจอ (Console) และบางลูกค้าอยากให้เซฟลงไฟล์ทิ้งไว้

1. **สร้าง Interface: `ILogger`**
   - มีเมธอดเดียวเพียวๆ: `void Log(string message);`

2. **สร้างผู้รับเหมาคนที่ 1: `ConsoleLogger`**
   - สืบทอด (Implement) จาก `ILogger`
   - ไส้ใน: ให้ปริ้นข้อความลงหน้าจอว่า `[จอภาพ]: {message}`

3. **สร้างผู้รับเหมาคนที่ 2: `FileLogger`**
   - สืบทอดจาก `ILogger`
   - ไส้ใน: ให้ปริ้นข้อความลงหน้าจอว่า `[เซฟลงไฟล์]: {message}` (สมมติว่าเซฟลงไฟล์แล้ว)

4. **สร้างระบบจัดการหลัก (Dependency Injection!): `Application`**
   - นี่คือคลาสปกติ มี Property `private ILogger _logger;` 
   - มี Constructor รับค่า `(ILogger logger)` โยนยัดเข้าไปใน Property (การฉีดปลั๊กอิน)
   - มีเมธอด `public void DoWork()` เมื่อทำงานเสร็จ ให้สั่ง `_logger.Log("ประมวลผลฐานข้อมูลเสร็จสิ้น!");`

5. **การทดสอบใน Main:**
   - แบบที่ 1: เสก `Application app1 = new Application(new ConsoleLogger());` แล้วสั่ง `.DoWork()`
   - แบบที่ 2: เสก `Application app2 = new Application(new FileLogger());` แล้วสั่ง `.DoWork()`

ถ้าคุณเขียนโค้ดข้อนี้ได้... คุณกำลังใช้สุดยอดคัมภีร์สถาปัตยกรรมระดับสากลที่ชื่อว่า **"Dependency Injection (DI)"** ซึ่งเป็นหัวใจสำคัญ 100% ของการเขียน ASP.NET Core MVC เลยครับ! (เปลี่ยนปลั๊กอินหน้างานได้โดยไม่ต้องแก้ลอจิกแกนกลาง!)

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ใน C# รุ่นใหม่ๆ (ตั้งแต่ C# 8.0 เป็นต้นมา) ผมได้ข่าวว่า Interface สามารถมี "ไส้ใน (Default Implementation)" ได้แล้ว... แบบนี้มันก็กลายพันธุ์เป็น Abstract Class ไปแล้วสิครับ?
**แนวคำตอบ:** **ใช่ครับ เป็นเรื่องที่ถกเถียงกันในวงการหนักมาก!**
ไมโครซอฟต์เพิ่มฟีเจอร์นี้เข้ามาเพื่อแก้ปัญหาที่เรียกว่า "API Evolution" ครับ (สมมติสัญญามี 2 ข้อ คนเซ็นไปแล้วล้านคนทั่วโลก วันนึงเราอยากเพิ่มข้อ 3... โค้ดพังกันทั่วโลกเลยทีเดียว!) เขาเลยอนุญาตให้เติมข้อ 3 พร้อมใส่ไส้ใน Default ไว้ได้ เพื่อกันโค้ดคนอื่นพัง...
แต่อย่างไรก็ตาม **"สถาปนิกที่ดี จะยังคงถือศีลบริสุทธิ์"** ห้ามเขียนไส้ในลง Interface เด็ดขาดครับ! ถ้าอยากมีไส้ใน ให้กลับไปใช้ Abstract Class เถอะครับ โค้ดจะอ่านง่ายและเป็นระเบียบตามมาตรฐานดั้งเดิมกว่าเยอะครับ!
:::
