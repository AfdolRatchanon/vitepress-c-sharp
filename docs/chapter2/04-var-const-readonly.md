# var, const และ readonly (ความฉลาดของ Compiler และค่าคงที่)

> 💡 **เป้าหมาย:** เรียนรู้เทคนิคการเขียนโค้ดให้สั้นกระชับและสะอาดตาขึ้นด้วย `var` (Implicit Typing) พร้อมทั้งทำความเข้าใจกลไกการล็อคข้อมูลไม่ให้ถูกแก้ไขเพื่อความปลอดภัยระดับสูงสุดด้วย `const` และ `readonly`

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. การลบความซ้ำซ้อนด้วย `var` (Implicitly Typed Local Variables)

ในการเขียนภาษา C# แบบดั้งเดิม (Explicit Typing) เรามักจะต้องพิมพ์ชนิดข้อมูลซ้ำซ้อนทั้งฝั่งซ้ายและขวาของเครื่องหมายเท่ากับ
เช่น `Dictionary<int, string> myDict = new Dictionary<int, string>();` ซึ่งยาวและน่ารำคาญมาก!

ไมโครซอฟต์จึงแนะนำคีย์เวิร์ด **`var`** ขึ้นมาใน C# 3.0 โดยมีกลไกการทำงานคือ **"ให้ Compiler ไปแอบดูข้อมูลทางฝั่งขวามือ แล้วเดาเอาเองว่าฝั่งซ้ายควรเป็น Data Type อะไร"** (เรียกว่า Type Inference)

```text
แผนภาพจำลองกระบวนการ Type Inference (ตอน Compile)

[Source Code]
  var score = 100;
  var name = "Alice";
  var isDone = false;

        │
        ▼ (คอมไพเลอร์วิ่งเข้ามาตรวจสอบฝั่งขวา)
        
[Compiler's Brain]
  100 เป็นจำนวนเต็มไม่มีทศนิยม ---> งั้น 'score' ต้องเป็น 'int' แน่นอน!
  "Alice" มีเครื่องหมาย Double Quote หุ้ม ---> งั้น 'name' ต้องเป็น 'string'!
  false เป็นค่าความจริง ---> งั้น 'isDone' ต้องเป็น 'bool'!

        │
        ▼ (สิ่งที่ถูกแปลงลงไปใน Bytecode จริงๆ)
        
[IL Code (ผลลัพธ์)]
  int score = 100;
  string name = "Alice";
  bool isDone = false;
```

**⚠️ กฎเหล็ก 3 ข้อของการใช้ `var`:**
1. **ไม่ใช่ Dynamic Type:** `var` ไม่ได้แปลว่ากล่องนี้จะเปลี่ยนชนิดข้อมูลไปมาได้เหมือนภาษา JavaScript หรือ Python! เมื่อคอมไพเลอร์ตัดสินใจแล้วว่ามันคือ `int` มันก็คือ `int` ไปตลอดชีวิต ห้ามเอา `string` มาใส่เด็ดขาด (C# ยังคงเป็น Strongly Typed 100%)
2. **ต้องกำหนดค่าทันที:** คุณจะพิมพ์แค่ `var x;` ไม่ได้เด็ดขาด เพราะถ้าไม่มีฝั่งขวา คอมไพเลอร์ก็ตรัสรู้ไม่ได้ว่าฝั่งซ้ายควรเป็นอะไร (ต้องเป็น `var x = 10;` เสมอ)
3. **ใช้ได้เฉพาะตัวแปร Local เท่านั้น:** `var` อนุญาตให้ใช้สร้างตัวแปรในขอบเขตปีกกาของ Method (เช่น ใน `Main`) เท่านั้น ห้ามนำไปสร้างตัวแปรระดับ Class หรือ Method Parameter เด็ดขาด

### 2. Magic Numbers (ตัวเลขอาถรรพ์ที่คนเขียนโปรแกรมเกลียด)

ในวงการ Software Engineering มีคำศัพท์ที่เรียกว่า **"Magic Number"** (หรือ Magic String) ซึ่งหมายถึงตัวเลขหรือข้อความที่ถูกโผล่พรวดขึ้นมากลางโค้ด โดยไม่มีใครรู้ความหมายของมัน

```csharp
// โค้ดที่แย่ (มี Magic Number)
double total = price + (price * 0.07); // 0.07 คืออะไร? ภาษี? ดอกเบี้ย? ส่วนลด?
if (status == 2) { ... }               // สถานะ 2 คืออะไร? แบน? VIP? ปกติ?
```

วิธีการกำจัดความอาถรรพ์นี้ คือการตั้งชื่อให้กับตัวเลขเหล่านั้นด้วยตัวแปร "ค่าคงที่"

### 3. การล็อคข้อมูลด้วย `const` (Compile-time Constant)

**`const`** (ย่อมาจาก Constant) คือคำสั่งสำหรับสร้าง "ค่าคงที่ถาวร" ที่ **ห้ามถูกแก้ไขเปลี่ยนแปลงอีกเลยตลอดกาล** (Read-only) เมื่อโปรแกรมถูกสร้างขึ้น

**คุณสมบัติของ `const`:**
1. ต้องกำหนดค่าทันทีตอนประกาศ (เหมือน var)
2. ค่าที่นำมาใส่ **ต้องเป็นสิ่งที่รู้ค่าแน่นอนตั้งแต่ตอนเขียนโค้ด (Compile-time)** เท่านั้น (เช่น ตัวเลขตายตัว, ข้อความตายตัว) จะเอาผลลัพธ์จากการเรียกฟังก์ชันหรือสิ่งที่ให้ผู้ใช้กรอกมาใส่ไม่ได้เด็ดขาด
3. ถ้าฝ่าฝืนเอาตัวแปร `const` ไปแก้ไขค่า โปรแกรมจะ Compile Error สีแดงทันที

```csharp
const double VatRate = 0.07; 
// VatRate = 0.10;  <-- ❌ Error: ไม่สามารถเปลี่ยนค่า const ได้
double total = price + (price * VatRate); // ✅ โค้ดอ่านง่ายขึ้น 10 เท่า! (รู้ทันทีว่าคือ Vat)
```

### 4. ก้าวข้ามข้อจำกัดด้วย `readonly` (Runtime Constant)

หาก `const` มันตึงเครียดเกินไป เพราะบังคับให้ระบุค่าตั้งแต่ตอนเขียนโค้ด 
C# ก็มีเครื่องมือพี่น้องที่ชื่อว่า **`readonly`** ซึ่งยืดหยุ่นกว่ามาก:

- `readonly` เป็นค่าคงที่เหมือนกัน แก้ไขไม่ได้เหมือนกัน (หลังจากตั้งค่าเสร็จแล้ว)
- แต่จุดเด่นคือ **มันอนุญาตให้คุณไปดึงข้อมูลระหว่างรันโปรแกรม (Runtime)** หรือดึงเวลาปัจจุบัน มากำหนดค่าเริ่มต้นให้มันได้ในครั้งแรก (ผ่าน Constructor)
- *(หมายเหตุ: `readonly` มักจะใช้ในระดับ Class Fields ซึ่งคุณจะได้เรียนอย่างละเอียดในบทของ Object-Oriented Programming แต่นี่เป็นการปูพื้นฐานให้เห็นความต่างจาก `const`)*

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: การใช้ var อย่างเหมาะสมและไม่เหมาะสม

โค้ดนี้จะแสดงให้เห็นว่าเมื่อใดควรใช้ `var` และเมื่อใดที่การใช้ `var` ทำให้เพื่อนร่วมงานปวดหัว

::: code-group
```csharp [Program.cs]
using System;

namespace VarDemonstration
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== การใช้งาน var ===");

            // 1. กรณีที่ควรใช้ var อย่างยิ่ง (ชัดเจนว่าฝั่งขวาคืออะไร)
            var playerName = "Knight"; // เห็นชัดๆ ว่าเป็น string
            var playerLevel = 99;      // เห็นชัดๆ ว่าเป็น int
            var isPremium = true;      // เห็นชัดๆ ว่าเป็น bool

            // 2. การทำงานเบื้องหลัง (Type Safety ยังคงอยู่)
            Console.WriteLine($"ผู้เล่น: {playerName}, เลเวล: {playerLevel}");

            // ทดลองแก้ค่า (Re-assign)
            playerName = "Paladin"; // ✅ ทำได้ เพราะ "Paladin" เป็น string เหมือนกัน
            
            // playerLevel = "Max"; // ❌ Error! คอมไพเลอร์ล็อค playerLevel เป็น int ไปแล้ว 
                                    // ห้ามเอา string ไปยัดใส่กล่อง int

            // 3. กรณีที่ไม่ควรใช้ var (ความหมายคลุมเครือ)
            // ตัวอย่าง: var result = CalculateDamage(); 
            // - ถ้าเขียนแบบนี้ คนมาอ่านโค้ดทีหลังจะไม่รู้เลยว่า result เป็นทศนิยม, หรือจำนวนเต็ม
            // - แบบนี้ควรเขียน explicit ชัดๆ ไปเลยว่า double result = CalculateDamage();

            // 4. ข้อระวังเรื่องตัวเลขทศนิยม
            var salary = 25000.50;  
            // เดาซิว่า salary เป็นอะไร? คำตอบคือ double! (เพราะเป็นค่าเริ่มต้นของ C#)
            // ถ้าอยากให้ var มองเป็น decimal ต้องใส่สัญลักษณ์กำกับ (m) ด้วย
            var vatAmount = 0.07m;  // คอมไพเลอร์รู้ทันทีว่านี่คือ decimal
        }
    }
}
```
:::

### ตัวอย่างที่ 2: ระบบคำนวณทรงกลมด้วยค่าคงที่ (Math Constants)

โค้ดนี้สาธิตการใช้ `const` ในการกำจัด Magic Numbers ทางคณิตศาสตร์

::: code-group
```csharp [Program.cs]
using System;

namespace ConstantsDemo
{
    class Program
    {
        // สร้างค่าคงที่ระดับ Class (เข้าถึงได้จากทุกที่)
        // Naming Convention สำหรับ const: นิยมใช้ PascalCase หรือ UPPER_CASE แบบ C/C++
        const double PI = 3.14159265359;
        const string AppName = "Sphere Calculator V.1";

        static void Main(string[] args)
        {
            Console.WriteLine($"=== ยินดีต้อนรับสู่ {AppName} ===");

            Console.Write("กรุณากรอกรัศมีของวงกลม (cm): ");
            double radius = double.Parse(Console.ReadLine());

            // สูตรหาพื้นที่วงกลม: Area = π * r^2
            double area = PI * (radius * radius);

            // สูตรหาเส้นรอบวง: Circumference = 2 * π * r
            double circumference = 2 * PI * radius;

            Console.WriteLine("\n[ผลลัพธ์การคำนวณ]");
            Console.WriteLine($"รัศมี       : {radius} cm");
            Console.WriteLine($"พื้นที่      : {area:F2} sq.cm");       // :F2 บังคับทศนิยม 2 ตำแหน่ง
            Console.WriteLine($"เส้นรอบวง   : {circumference:F2} cm");

            // PI = 3.0; // ❌ ลองเอาคอมเมนต์ออกดู โปรแกรมจะพังทันที เพราะค่า PI ถูกล็อคไว้แล้ว!
        }
    }
}
```
:::

**Expected Output:**
```
=== ยินดีต้อนรับสู่ Sphere Calculator V.1 ===
กรุณากรอกรัศมีของวงกลม (cm): 5

[ผลลัพธ์การคำนวณ]
รัศมี       : 5 cm
พื้นที่      : 78.54 sq.cm
เส้นรอบวง   : 31.42 cm
```

### **โจทย์ที่ 2: ผ่าตัดโค้ดที่ไร้ระเบียบ (Refactoring Magic Numbers)**
คุณได้รับโค้ดมรดกมาจากพนักงานคนก่อน ซึ่งเขียนระบบคำนวณเงินเดือนสำหรับโปรแกรมเมอร์ที่ทำโอที (Overtime) ในวันหยุด
```csharp
double baseSalary = 30000;
int otHours = 5;

// คำนวณเงินเดือนรวม = เงินเดือน + (ชั่วโมง OT * 450.50) หักประกันสังคม 5%
double totalPay = baseSalary + (otHours * 450.50);
totalPay = totalPay - (totalPay * 0.05);

Console.WriteLine("รายรับสุทธิ: " + totalPay);
```
**ภารกิจ:** จง Refactor (ปรับปรุง) โค้ดนี้ใหม่ โดยดึงตัวเลขปริศนา `450.50` และ `0.05` ออกมาตั้งเป็นตัวแปร `const` พร้อมตั้งชื่อให้เหมาะสมและสื่อความหมายชัดเจนที่สุด แล้วนำตัวแปรค่าคงที่นั้นกลับไปแทนที่ในสูตรคำนวณ

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: มหกรรมสับสน var VS dynamic**

ในภาษา C# มีคีย์เวิร์ดระดับสูงอีกตัวที่ชื่อว่า **`dynamic`** ซึ่งทำหน้าที่คล้ายๆ ตัวแปรใน JavaScript คือแปลงร่างไปมาได้ตลอดเวลา

**ภารกิจของคุณ (การวิจัยเชิงทดลอง):**
ให้คุณสร้างไฟล์ใหม่ แล้วคัดลอกโค้ดนี้ไปทดลองรัน:

```csharp
// การทดลองฝั่ง var (Strongly Typed)
var box1 = 10;
// box1 = "Hello"; // บรรทัดนี้จะ Error ตัวแดงก่อนรันเลยใช่ไหม?

// การทดลองฝั่ง dynamic (Weakly Typed)
dynamic box2 = 10;
box2 = "Hello";    // บรรทัดนี้ Error ไหม? ลองรันดู!
Console.WriteLine(box2);
```

**คำถามท้าทาย:** 
1. `dynamic` ยอมให้คุณเปลี่ยนชนิดข้อมูล (จากตัวเลขเป็นข้อความ) ได้สำเร็จหรือไม่?
2. แล้วถ้ามันสะดวกขนาดนั้น... ทำไมบริษัทยักษ์ใหญ่และโปรแกรมเมอร์ C# ทั่วโลกถึงพร่ำสอนกันว่า **"จงหลีกเลี่ยงการใช้ dynamic ให้มากที่สุด และจงใช้ var แทน"** ? (ลองค้นหาข้อเสียของ dynamic เกี่ยวกับ Performance และ Intellisense ในเน็ตดู!)

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** คีย์เวิร์ด `var` มีผลทำให้โปรแกรมทำงาน "ช้าลง" หรือไม่? เพราะมันดูเหมือนคอมพิวเตอร์ต้องมานั่งเดาตอนโปรแกรมทำงาน?
**แนวคำตอบ:** **ไม่ช้าลงเลยแม้แต่นิดเดียว! (0% Performance Impact)**
นี่คือความเข้าใจผิดยอดฮิต! กระบวนการ "เดา (Type Inference)" ของ `var` เกิดขึ้น **"ตอน Compile (ตอนที่แปลงโค้ดจากข้อความอังกฤษเป็น Bytecode)"** ไม่ใช่ตอนที่ผู้ใช้นำโปรแกรมไปรัน 
ดังนั้น ไม่ว่าคุณจะเขียน `int x = 5;` หรือ `var x = 5;` เมื่อมันถูก Compile เสร็จ รหัสเครื่อง (Machine Code) เบื้องหลังจะหน้าตา **เหมือนกันเป๊ะ 100%** การใช้ `var` จึงเป็นแค่ Syntax Sugar (ลูกอมเคลือบไวยากรณ์) ที่ช่วยประหยัดเวลาพิมพ์ให้โปรแกรมเมอร์เท่านั้น

**คำถาม 2:** ระหว่างการใช้ `readonly` และ `const` คุณควรเลือกใช้อันไหนเมื่อต้องการเก็บ "หมายเลขเวอร์ชันของแอปพลิเคชัน (เช่น 1.0.5)" และ "เวลาที่ผู้ใช้เริ่มรันแอปพลิเคชัน (Start Time)"
**แนวคำตอบ:**
- หมายเลขเวอร์ชัน: ควรใช้ **`const`** (เช่น `const string AppVersion = "1.0.5";`) เพราะมันเป็นข้อมูลที่เรา "รู้ล่วงหน้า" ตั้งแต่ตอนนั่งพิมพ์โค้ดแล้ว และไม่มีทางเปลี่ยนไปเปลี่ยนมาได้
- เวลาที่เริ่มรัน: ต้องใช้ **`readonly`** เท่านั้น! เพราะเวลาตอนที่กดรันโปรแกรม (Runtime) มันหมุนไปเรื่อยๆ เราไม่สามารถเดาได้ล่วงหน้าตอนเขียนโค้ด (Compile-time) การใช้ `readonly` จะทำให้ระบบดึงเวลาตอนที่เป๊ะๆ นั้นมากำหนดค่า และล็อคเอาไว้ไม่ให้เปลี่ยนแปลงตลอดการทำงานของแอป
