# Numeric Types (ชนิดข้อมูลตัวเลขเชิงลึก)

> 💡 **เป้าหมาย:** ทะลวงลึกถึงการเก็บตัวเลขในระดับหน่วยความจำ (Bits & Bytes) ทำความเข้าใจปัญหาคลาสสิกของนักเขียนโปรแกรมอย่าง Integer Overflow และ Floating-Point Inaccuracy และเหตุผลที่ทำไมธนาคารทั่วโลกถึงบังคับใช้ `decimal` ในการคำนวณ

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. โครงสร้างตัวเลขระดับบิต (Bits and Bytes)

ก่อนที่เราจะเลือกใช้ตัวแปรตัวเลข เราต้องเข้าใจว่าคอมพิวเตอร์มองทุกอย่างเป็นเลข ฐาน 2 (0 และ 1)
- **1 Bit** (บิต) = ข้อมูล 1 ตำแหน่ง (เป็นได้แค่ 0 หรือ 1)
- **8 Bits** = **1 Byte** (ไบต์)

ข้อมูลประเภทต่างๆ จะกินพื้นที่ Memory (RAM) แตกต่างกัน ยิ่งกินพื้นที่มาก ยิ่งเก็บเลขได้เยอะ:

| ชนิดข้อมูล (Type) | ขนาดใน RAM | ช่วงตัวเลขที่เก็บได้ (Range) | การใช้งานที่เหมาะสม |
| :--- | :---: | :--- | :--- |
| `byte` | 1 Byte (8 bits) | `0` ถึง `255` | อายุ, สี (RGB), ข้อมูลระดับฮาร์ดแวร์ |
| `short` | 2 Bytes (16 bits) | `-32,768` ถึง `32,767` | ตัวเลขขนาดเล็กที่ติดลบได้ (ไม่ค่อยนิยมใช้แล้ว) |
| **`int`** | **4 Bytes (32 bits)**| `-2,147,483,648` ถึง `2,147,483,647` (ประมาณ 2 พันล้าน) | **ค่าเริ่มต้นของระบบ** ใช้เกือบทุกสถานการณ์ |
| `long` | 8 Bytes (64 bits) | หลักล้านล้านล้าน... | ยอดวิว YouTube โลก, การนับวินาทีระดับจักรวาล |

**กลไก Two's Complement (ทำไมถึงมีเลขติดลบ?):**
ในขนาดพื้นที่เท่ากัน (เช่น 32 bits ของ int) บิตซ้ายสุด (Most Significant Bit) จะถูกยึดไปทำหน้าที่เป็น **"ป้ายบอกเครื่องหมาย" (Sign bit)** ว่าเลขนี้เป็นบวกหรือลบ นี่คือสาเหตุว่าทำไม `byte` (ที่ไม่มีบิตเครื่องหมาย) เก็บได้ 0-255 แต่ถ้าเป็นชนิดที่มีบิตเครื่องหมาย (เช่น `sbyte`) จะเก็บได้แค่ -128 ถึง 127 แทน

### 2. มหันตภัย Integer Overflow (ตัวเลขล้น)

เมื่อเราพยายามเติมข้อมูลที่ขนาดใหญ่เกินกว่าที่กล่อง (Data Type) จะรับได้ จะเกิดปรากฏการณ์ที่เรียกว่า **Overflow**
จงจินตนาการถึงหน้าปัดเลขไมล์ของรถยนต์รุ่นเก่า ที่มีตัวเลขสุดแค่ `999,999`
เมื่อรถวิ่งไปอีก 1 กิโลเมตร หน้าปัดจะไม่แสดง `1,000,000` แต่มันจะหมุนตีกลับไปเป็น `000,000` ใหม่!

คอมพิวเตอร์ก็เช่นกัน แต่สิ่งที่น่ากลัวกว่าคือ **เมื่อค่าฝั่งบวกของ `int` ทะลุขีดจำกัด (2 พันล้าน) มันจะตีกลับไปเป็นค่าติดลบ (-2 พันล้าน) ทันที!** ซึ่งเคยทำให้ระบบซอฟต์แวร์ของจรวดอวกาศพังมาแล้ว

```text
แผนภาพจำลอง Integer Overflow

[ ค่าบวกสูงสุด: 2,147,483,647 ]  ----บวกเพิ่มอีก 1----> [ ค่าลบสุดขั้ว: -2,147,483,648 ]
       ▲                                                          │
       │                                                          │
       └────────────────────────(ตีกลับวงกลม)──────────────────────┘
```

### 3. ตัวเลขทศนิยม (Floating-Point Types)

ตัวเลขที่มีจุดทศนิยมใน C# แบ่งออกเป็น 3 ชนิดหลักตามความแม่นยำ (Precision) และสถาปัตยกรรม:

| ชนิด (Type) | ขนาด | ความแม่นยำ (จำนวนหลัก) | Suffix (ตัวต่อท้าย) | เหมาะกับงาน |
| :--- | :---: | :---: | :---: | :--- |
| `float` | 4 Bytes | ~6-9 หลัก | `f` หรือ `F` | เกม 3D (Unity), กราฟิกที่ต้องการประมวลผลเร็ว |
| **`double`** | 8 Bytes | ~15-17 หลัก | `d` (ปกติไม่ต้องใส่) | **ค่าเริ่มต้นของ C#**, การคำนวณคณิตศาสตร์ทั่วไป วิทยาศาสตร์ |
| **`decimal`** | 16 Bytes| ~28-29 หลัก | `m` หรือ `M` | **ระบบการเงิน ธนาคาร**, ระบบบัญชีที่ห้ามผิดพลาด |

**⚠️ คำเตือนเรื่อง IEEE 754 (ปัญหาทศนิยมระดับฮาร์ดแวร์):**
`double` และ `float` จัดเก็บข้อมูลแบบเศษส่วนของฐาน 2 (Base-2 fraction) คล้ายๆ กับการที่เราเขียนเศษ $1/3$ ในฐาน 10 ว่า `0.33333...` (ไม่สิ้นสุด)
ในฐาน 2 การเก็บเลข `0.1` จะกลายเป็นเลขทศนิยมไม่รู้จบ คอมพิวเตอร์จึงต้องตัดทอน (Round) ทำให้ค่าเพี้ยน!

```csharp
double a = 0.1;
double b = 0.2;
Console.WriteLine(a + b); 
// คุณคาดหวัง 0.3 ใช่ไหม? แต่คอมพิวเตอร์จะพิมพ์: 0.30000000000000004
```
นี่คือสาเหตุที่เรา **ห้ามใช้ `double` หรือ `float` กับการคำนวณเงินเด็ดขาด** (จินตนาการว่าดอกเบี้ยธนาคารคลาดเคลื่อนไป 0.00000000000000004 ทุกครั้งที่โอนเงิน คนนับล้านคนจะทำให้เกิดความเสียหายมหาศาล)

**วิธีแก้ปัญหา:** ต้องใช้ **`decimal`** ซึ่งจัดเก็บข้อมูลแบบฐาน 10 (Base-10) ทำให้ไม่มีปัญหาการคลาดเคลื่อนนี้

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: การจำลอง Integer Overflow และวิธีรับมือ

โค้ดนี้จะทำให้คุณเห็นภาพการพังทลายของตัวเลขแบบสดๆ

::: code-group
```csharp [Program.cs]
using System;

namespace OverflowDemo
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== การสาธิต Integer Overflow ===");

            // 1. ดึงค่าสูงสุดที่เป็นไปได้ของประเภท int (ใช้ Properties MaxValue)
            int maxScore = int.MaxValue; 
            
            // แสดงผลค่าเริ่มต้น (ควรจะเป็น 2,147,483,647)
            Console.WriteLine($"คะแนนสูงสุดที่เป็นไปได้ของ int คือ: {maxScore:N0}"); 
            // หมายเหตุ: :N0 คือการใส่ลูกน้ำ (Comma) แบ่งหลักพัน เพื่อให้อ่านง่าย

            // 2. จำลองการได้คะแนนเพิ่มอีก 1 คะแนน
            Console.WriteLine("\n[ผู้เล่นเก็บไอเทมพิเศษ ได้รับคะแนนเพิ่ม +1]");
            
            // การบวกเพิ่ม 1 ครั้งนี้จะทำให้ขีดจำกัดพังทลาย
            maxScore = maxScore + 1;

            // 3. ดูผลลัพธ์ของความผิดพลาด
            Console.WriteLine($"คะแนนปัจจุบันของผู้เล่นกลายเป็น: {maxScore:N0}");
            Console.WriteLine("เห็นไหมว่าคะแนนกลายเป็นติดลบ! ผู้เล่นหัวร้อนแน่นอน");

            // ============================================
            // แนวทางแก้ไข: หากรู้ว่าตัวเลขจะเยอะขนาดนี้ ควรเปลี่ยนไปใช้ long
            // ============================================
            Console.WriteLine("\n=== แนวทางแก้ไขด้วย long ===");
            long betterScore = int.MaxValue; // นำค่าเดียวกันมาใส่กล่องใหญ่ขึ้น (long)
            betterScore = betterScore + 1;
            Console.WriteLine($"ถ้าใช้กล่อง long คะแนนจะกลายเป็น: {betterScore:N0} (ถูกต้องตามปกติ)");
        }
    }
}
```
:::

**Expected Output:**
```
=== การสาธิต Integer Overflow ===
คะแนนสูงสุดที่เป็นไปได้ของ int คือ: 2,147,483,647

[ผู้เล่นเก็บไอเทมพิเศษ ได้รับคะแนนเพิ่ม +1]
คะแนนปัจจุบันของผู้เล่นกลายเป็น: -2,147,483,648
เห็นไหมว่าคะแนนกลายเป็นติดลบ! ผู้เล่นหัวร้อนแน่นอน

=== แนวทางแก้ไขด้วย long ===
ถ้าใช้กล่อง long คะแนนจะกลายเป็น: 2,147,483,648 (ถูกต้องตามปกติ)
```

### ตัวอย่างที่ 2: ความผิดพลาดของทศนิยม (Double vs Decimal)

โค้ดนี้จำลองระบบคิดดอกเบี้ยเงินฝาก เพื่อแสดงความสำคัญของ `decimal`

::: code-group
```csharp [Program.cs]
using System;

namespace FinancialDemo
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ทดสอบการคำนวณทางบัญชี (Double vs Decimal) ===");

            // สถานการณ์: มีเงิน 0.1 บาท 3 บัญชี ต้องการนำมารวมกัน (ผลรวมควรเป็น 0.3)

            // 1. การคำนวณด้วย double (วิธีที่ผิด)
            double d1 = 0.1;
            double d2 = 0.1;
            double d3 = 0.1;
            double doubleResult = d1 + d2 + d3;

            Console.WriteLine($"ผลรวมด้วยกล่อง double  : {doubleResult}");
            
            // เช็คว่ามันเท่ากับ 0.3 จริงไหม?
            Console.WriteLine($"เช็ค doubleResult == 0.3 : {doubleResult == 0.3}"); 
            // จะได้ค่า False! เพราะมันคือ 0.30000000000000004 ไม่ใช่ 0.3 เป๊ะๆ

            Console.WriteLine("-------------------------------------------------");

            // 2. การคำนวณด้วย decimal (วิธีที่ถูกต้อง)
            // ข้อบังคับ: ต้องห้อยตัว 'm' หรือ 'M' ท้ายตัวเลขเสมอเพื่อบอก Compiler
            decimal m1 = 0.1m;
            decimal m2 = 0.1m;
            decimal m3 = 0.1m;
            decimal decimalResult = m1 + m2 + m3;

            Console.WriteLine($"ผลรวมด้วยกล่อง decimal : {decimalResult}");
            
            // เช็คว่ามันเท่ากับ 0.3 จริงไหม?
            Console.WriteLine($"เช็ค decimalResult == 0.3m : {decimalResult == 0.3m}");
            // จะได้ค่า True ถูกต้องตามหลักคณิตศาสตร์ร้อยเปอร์เซ็นต์
        }
    }
}
```
:::

**Expected Output:**
```
=== ทดสอบการคำนวณทางบัญชี (Double vs Decimal) ===
ผลรวมด้วยกล่อง double  : 0.30000000000000004
เช็ค doubleResult == 0.3 : False
-------------------------------------------------
ผลรวมด้วยกล่อง decimal : 0.3
เช็ค decimalResult == 0.3m : True
```

### **โจทย์ที่ 1: สถาปนิกโครงสร้างข้อมูล (Data Architect)**
ในฐานะผู้เขียนโปรแกรม คุณมีหน้าที่ออกแบบฐานข้อมูลสำหรับ **"ระบบบริหารจัดการโรงพยาบาล"** 
จงเลือก Data Type ชนิดตัวเลขที่เหมาะสมที่สุด (`byte`, `short`, `int`, `long`, `double`, `decimal`) ให้กับตัวแปรเหล่านี้ พร้อมบอกเหตุผลสั้นๆ:
1. `bloodPressure` (ค่าความดันโลหิต เช่น 120)
2. `patientHeight` (ส่วนสูงของคนไข้ในหน่วยเซนติเมตร เช่น 175.5)
3. `hospitalBudget` (งบประมาณโรงพยาบาลรายปี เช่น 5,000,000,000 บาท)
4. `roomRate` (ค่าเช่าห้องพักผู้ป่วยต่อคืน เช่น 2500.50 บาท)
5. `heartRate` (อัตราการเต้นของหัวใจต่อนาที เช่น 85)

### **โจทย์ที่ 2: บั๊กตัวอักษร M (Suffix Bug)**
คุณได้รับมอบหมายให้แก้ไขโค้ดเก่าของบริษัท ซึ่งขึ้นแถบ Error สีแดงตอน Build รันไม่ผ่าน:
```csharp
decimal productPrice = 999.99;
float discountRate = 0.15;
```
จงอธิบายว่าทำไมถึงรันไม่ผ่าน และคุณจะแก้โค้ดสองบรรทัดนี้อย่างไรให้ถูกต้องตามหลักของ C#

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  - งบประมาณประเทศหลักหลายพันล้าน... `int` จะเอาอยู่ไหม?
  - ค่าห้องพักผู้ป่วย... เป็นเรื่องของ "เงิน" ต้องใช้อะไร?
  - อัตราการเต้นหัวใจคนเรา... เป็นจำนวนเต็ม และคนเราหัวใจไม่เต้นเกิน 255 ครั้งต่อนาทีแน่ๆ (ถ้าใช้กล่องเล็กสุดได้ก็จะประหยัด RAM)
- **สำหรับโจทย์ 2:** ตัวเลขทศนิยมที่ไม่มีสัญลักษณ์ต่อท้าย (Suffix) จะถูก C# มองว่าเป็นลูกหลานชั้นประเสริฐ (นั่นคือ `double`) เสมอ การจะเอาของขนาดยักษ์อย่าง `double` ไปยัดใส่กล่องเฉพาะทางอย่าง `decimal` หรือ `float` เลยตรงๆ ไม่ได้ ต้องมีการแปะป้ายสัญลักษณ์ (M และ F)

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: การควบคุม Overflow ด้วยคำสั่ง `checked`**

เราเรียนรู้ว่าเมื่อเกิด Overflow ตัวแปรจะตีกลับไปเป็นค่าลบโดยที่โปรแกรมไม่ร้องเตือน (Silent Bug) ซึ่งอันตรายมาก 
แต่ C# มีอาวุธลับเรียกว่า **`checked block`** ซึ่งถ้ามี Overflow เกิดขึ้นในบล็อกนี้ โปรแกรมจะพัง (Crash) และพ่น Error ออกมาทันที ซึ่งดีกว่าปล่อยให้ตัวเลขคลาดเคลื่อน

**ภารกิจของคุณ:** 
ให้หาข้อมูลเกี่ยวกับวิธีการเขียนบล็อก `checked { ... }` ใน C# 
แล้วนำโค้ดในตัวอย่างที่ 1 (Integer Overflow) ไปจับใส่ในบล็อก `checked` 
จากนั้นรันโปรแกรมดูว่า มันแสดง Error Exception ตัวแดงที่มีชื่อว่าอะไร? (เช่น FormatException, OverflowException, เป็นต้น)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในเกม Minecraft สมัยโบราณ มีบั๊กอันโด่งดังที่เรียกว่า "Far Lands" ซึ่งเมื่อผู้เล่นเดินไปถึงพิกัด 12,550,820 (ตามหลักคณิตศาสตร์ของโค้ดที่คูณกัน) ภูมิประเทศของเกมจะบิดเบี้ยวและแตกสลายพังทลาย คุณคิดว่านี่คือผลลัพธ์ของปรากฏการณ์ใดในหัวข้อนี้?
**แนวคำตอบ:** ปรากฏการณ์ **Integer Overflow** (หรือการสูญเสีย Precision ของ Floating-point ในบางบริบท) เมื่อตำแหน่งแกนพิกัด (X, Z) ถูกบวกลบคูณหารจนทะลุขีดจำกัดสูงสุดของตัวแปรประเภท 32-bit กราฟิกเครื่องยนต์ (Engine) จะคำนวณตัวเลขมั่วและพังทลายลง ซึ่งต่อมาเกมยุคใหม่ๆ แก้ปัญหานี้โดยการเปลี่ยนตัวแปรพิกัดทั้งหมดให้กลายเป็น `long` (64-bit) ซึ่งวิ่งได้ไกลกว่าเดิมหลายล้านเท่า

**คำถาม 2:** สมมติคุณกำลังเขียนโปรแกรมเครื่องคิดเลขวิทยาศาสตร์ธรรมดาๆ ที่มีการคำนวณค่า Pi, หาพื้นที่วงกลม, คำนวณความเร็วแสง ทำไมคุณถึงควรเลือกใช้ `double` มากกว่าที่จะใช้ `decimal` ทั้งที่ `decimal` แม่นยำกว่า?
**แนวคำตอบ:** มีเหตุผลหลัก 2 ประการ:
1. **ประสิทธิภาพ (Performance):** การคำนวณด้วย `double` (IEEE 754) ได้รับการสนับสนุนโดยตรงจากชิปประมวลผล (FPU ใน CPU) ทำให้บวกลบคูณหารได้ **เร็วกว่า `decimal` เป็นสิบๆ เท่า**
2. **ขอบเขต (Range):** `double` อาจจะแม่นยำแค่ 15 หลัก แต่ช่วงตัวเลขที่มันเก็บได้ทั้งกว้างและยาวมหาศาลมาก (หลัก 10^308) เหมาะกับตัวเลขทางฟิสิกส์ ดาราศาสตร์ หรือวิทยาศาสตร์ที่ค่ามากๆ หรือเล็กมากๆ ต่างจาก `decimal` ที่โฟกัสไปที่ความแม่นยำ 28 หลักแต่ช่วงตัวเลขจำกัดกว่ามาก
สรุปกฎเหล็ก: "วิทยาศาสตร์ใช้ Double, การเงินและบัญชีใช้ Decimal"
