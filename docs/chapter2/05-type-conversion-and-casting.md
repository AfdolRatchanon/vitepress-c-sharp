# Type Conversion & Casting (การแปลงชนิดข้อมูล)

> 💡 **เป้าหมาย:** ทำความเข้าใจกฎของการเปลี่ยนรูปแบบข้อมูลข้ามสายพันธุ์ (เช่น จากตัวเลขไปข้อความ, จากข้อความไปตัวเลข) เรียนรู้ความแตกต่างของกลไก Implicit/Explicit Casting พร้อมทั้งเทคนิคขั้นสูงในการสกัดข้อมูลที่ปลอดภัยด้วย `TryParse`

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ทำไมเราต้องแปลงข้อมูล? (The Need for Conversion)

ในภาษาที่เป็น **Strongly Typed** อย่าง C# กฎเหล็กที่แข็งแกร่งที่สุดคือ "คุณไม่สามารถนำข้อมูลต่างสายพันธุ์มาผสมหรือจัดเก็บร่วมกันได้โดยพลการ" 
เช่น การเอาข้อความไปใส่ในกล่องตัวเลข หรือเอาตัวเลขที่มีทศนิยมไปใส่ในกล่องจำนวนเต็ม
แต่ในโลกความเป็นจริง ข้อมูลทุกอย่างที่ไหลเข้ามาในโปรแกรมผ่านแป้นพิมพ์ผู้ใช้ (Console) หรือผ่านเว็บไซต์ มักจะมาในรูปแบบของ **"ข้อความ (`string`)"** เสมอ ดังนั้น การแปลงข้อความเป็นตัวเลขเพื่อนำไปคำนวณต่อ จึงเป็นกระบวนการที่ถูกใช้มากที่สุดในวงการโปรแกรมมิ่ง

---

### 2. Implicit Casting (การแปลงแบบอัตโนมัติ / ปลอดภัย)

เมื่อคุณต้องการนำข้อมูลจาก **"กล่องใบเล็ก"** ไปเทใส่ **"กล่องใบใหญ่"** (ที่มีประเภทเดียวกัน เช่น ตัวเลขไปตัวเลข) คอมไพเลอร์ของ C# จะมองเห็นว่าเรื่องนี้ปลอดภัยร้อยเปอร์เซ็นต์ (Safe Widening Conversion) มันจึงยอมให้คุณทำได้เลยโดยอัตโนมัติ ไม่ต้องมีคำสั่งพิเศษใดๆ

```text
แผนภาพ Implicit Casting (พื้นที่รับข้อมูลกว้างขึ้น ไม่มีทางล้น)

  [ int: 4 Bytes ]   ======เทข้อมูล======>   [ double: 8 Bytes ]
  
  (ข้อมูล 100)                             (ข้อมูลถูกแปลงเป็น 100.0 อย่างปลอดภัย)
```

**ตัวอย่างในโค้ด:**
```csharp
int myInt = 15;
double myDouble = myInt; // ✅ คอมพิวเตอร์ทำให้เองอัตโนมัติ เพราะ 8 bytes ย่อมจุ 4 bytes ได้สบายๆ
```

---

### 3. Explicit Casting (การแปลงแบบบังคับ / อันตราย)

ในทางกลับกัน เมื่อคุณพยายามนำข้อมูลจาก **"กล่องใบใหญ่"** ไปยัดใส่ **"กล่องใบเล็ก"** คอมไพเลอร์จะร้องเตือนและบล็อกคุณทันที (Error) เพราะมันกังวลว่าข้อมูลบางส่วนอาจจะ "ล้น" หรือ "ถูกตัดทิ้ง" (Narrowing Conversion)

ถ้าคุณยืนยันที่จะทำจริงๆ คุณต้องรับผิดชอบความเสี่ยงเอง โดยการใส่ **สัญลักษณ์วงเล็บชนิดข้อมูล (Cast Operator)** แปะไว้หน้าตัวแปรเพื่อ "บังคับ (Force)" คอมพิวเตอร์

```text
แผนภาพ Explicit Casting (ยัดของใหญ่ลงกล่องเล็ก ข้อมูลอาจสูญหาย)

  [ double: 8 Bytes ]   ======ยัดข้อมูล======>   [ int: 4 Bytes ]
  
  (ข้อมูล 9.99)                              (ข้อมูล 9.99 ถูกหั่นทิ้ง! เหลือแค่ 9)
```

**ตัวอย่างในโค้ด:**
```csharp
double myDouble = 9.99;
// int myInt = myDouble;    // ❌ Error! คอมพิวเตอร์ไม่ยอมให้ทำ
int myInt = (int)myDouble;  // ✅ บังคับทำ (Explicit Cast) ผลลัพธ์: myInt จะมีค่า 9 (เศษทศนิยม .99 ถูกสับทิ้งอย่างเหี้ยมโหด)
```

### 4. การแปลงระหว่าง String และ Number (Parsing & Convert)

การใช้ตรรกะ "กล่องเล็ก กล่องใหญ่" ใช้ได้แค่กับสายพันธุ์ตัวเลขด้วยกันเท่านั้น
แต่ถ้าคุณจะข้ามสายพันธุ์ไปมาระหว่าง `string` กับตัวเลข คุณต้องพึ่งพากระบวนการพิเศษที่เรียกว่า **Parsing** หรือ **Converting**

#### การแปลงจาก "ข้อความ" ไปเป็น "ตัวเลข"
มี 3 วิธีหลักที่นิยมใช้ใน C#:

1. **`int.Parse(text)`**
   - เป็นวิธีที่ตรงไปตรงมา
   - **ข้อเสีย:** ถ้าข้อความนั้นไม่ใช่ตัวเลข (เช่น พิมพ์ว่า "สิบ" แทน "10" หรือพิมพ์ตัวอักษร A) โปรแกรมจะเกิดการ Crash บึ้มสนั่นทันที (FormatException)

2. **`Convert.ToInt32(text)`**
   - คล้าย `Parse` แต่ฉลาดกว่านิดหน่อยตรงที่ ถ้าข้อมูลต้นทางเป็นค่าว่างเปล่า (`null`) มันจะคืนค่า 0 ให้แทนที่จะพัง (แต่ถ้าเป็นตัวอักษรก็พังเหมือนเดิม)

3. **`int.TryParse(text, out variable)` (พระเอกของงาน!)**
   - นี่คือเครื่องมือที่มืออาชีพใช้! เป็นวิธีที่ **"ปลอดภัย 100%"** 
   - ระบบจะ "พยายาม" แปลงดูเงียบๆ 
   - ถ้าแปลงสำเร็จ มันจะโยนผลลัพธ์ใส่ตัวแปรที่อยู่หลังคำว่า `out` และตอบกลับมาเป็น `true`
   - ถ้าแปลงไม่สำเร็จ (เพราะกรอกข้อความขยะมา) โปรแกรมจะ **ไม่พัง** แต่จะตอบกลับเป็น `false` และตั้งค่าตัวแปรเป็น 0 ให้แทน

#### การแปลงจาก "ตัวเลข" ไปเป็น "ข้อความ"
ง่ายที่สุดในโลก! ทุกตัวแปรใน C# มีความสามารถพิเศษ (Method) ที่ชื่อว่า **`.ToString()`** ติดตัวมาตั้งแต่เกิด

```csharp
int score = 100;
string textScore = score.ToString(); // ตอนนี้กลายเป็นคำว่า "100" (string) เรียบร้อยแล้ว
```

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: มหกรรมสับทศนิยม (Casting Data Loss)

โค้ดนี้สาธิตให้เห็นถึงอันตรายของการใช้ Explicit Casting ว่าข้อมูลถูกทำลายทิ้งได้อย่างไร

::: code-group
```csharp [Program.cs]
using System;

namespace CastingDemo
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== การบังคับแปลงข้อมูล (Explicit Casting) ===");

            double originalPrice = 1299.99;
            Console.WriteLine($"ราคาสินค้าตั้งต้น (double): {originalPrice}");

            // การบังคับแปลงเป็น int จะส่งผลให้ระบบ "ตัดเศษทิ้ง (Truncate)" ไม่ใช่การปัดเศษ (Round)
            int integerPrice = (int)originalPrice;
            Console.WriteLine($"ราคาหลังถูกบีบใส่กล่อง (int): {integerPrice}");
            Console.WriteLine("เห็นไหมว่า .99 หายวับไปกับตา! (Data Loss)");

            // ถ้าต้องการปัดเศษจริงๆ ต้องพึ่งพาคลาสคณิตศาสตร์ (Math)
            int roundedPrice = (int)Math.Round(originalPrice);
            Console.WriteLine($"\nถ้านำไปผ่านกระบวนการ Math.Round() ก่อน จะได้ราคา: {roundedPrice}");
        }
    }
}
```
:::

**Expected Output:**
```
=== การบังคับแปลงข้อมูล (Explicit Casting) ===
ราคาสินค้าตั้งต้น (double): 1299.99
ราคาหลังถูกบีบใส่กล่อง (int): 1299
เห็นไหมว่า .99 หายวับไปกับตา! (Data Loss)

ถ้านำไปผ่านกระบวนการ Math.Round() ก่อน จะได้ราคา: 1300
```

### ตัวอย่างที่ 2: ระบบรับข้อมูลที่ไม่มีวันพังด้วย `TryParse`

นี่คือระบบเครื่องคิดเลขอายุ ที่ผู้ใช้จะไม่สามารถทำให้โปรแกรมค้างได้แม้จะพยายามก่อกวนพิมพ์ข้อความแปลกๆ ลงไปก็ตาม (นี่คือการทำ Input Validation แบบเริ่มต้น)

::: code-group
```csharp [Program.cs]
using System;

namespace SafeParsingDemo
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบตรวจสอบอายุเข้าผับ (Crash-Proof) ===");

            Console.Write("กรุณากรอกอายุของคุณ: ");
            string input = Console.ReadLine();

            // สร้างกล่อง int ว่างๆ มารอรับข้อมูล
            int age;

            // พระเอกออกโรง! ใช้ TryParse
            // มันจะเอาข้อความ input ไปลองแปลง ถ้าแปลงได้ จะยัดใส่กล่อง age แล้วคืนค่า True 
            // ถ้าผู้ใช้กวนโอ๊ยพิมพ์ตัวอักษรมา มันจะคืนค่า False และโปรแกรมเดินหน้าต่อไปได้โดยไม่พัง
            bool isSuccess = int.TryParse(input, out age);

            Console.WriteLine("\n[ผลลัพธ์การตรวจสอบ]");
            Console.WriteLine($"โปรแกรมสามารถแปลงข้อความเป็นตัวเลขได้หรือไม่?: {isSuccess}");
            
            // นำตัวแปร bool ไปประยุกต์ร่วมกับ Ternary Operator เพื่อการตัดสินใจ
            string status = isSuccess 
                            ? $"การวิเคราะห์เสร็จสิ้น: คุณมีอายุ {age} ปี" 
                            : "❌ ข้อผิดพลาด: สิ่งที่คุณกรอกไม่ใช่ตัวเลข กรุณาลองใหม่!";

            Console.WriteLine(status);
        }
    }
}
```
:::

**Expected Output (กรณีผู้ใช้กรอกตัวเลขปกติ):**
```
=== ระบบตรวจสอบอายุเข้าผับ (Crash-Proof) ===
กรุณากรอกอายุของคุณ: 25

[ผลลัพธ์การตรวจสอบ]
โปรแกรมสามารถแปลงข้อความเป็นตัวเลขได้หรือไม่?: True
การวิเคราะห์เสร็จสิ้น: คุณมีอายุ 25 ปี
```

**Expected Output (กรณีผู้ใช้ก่อกวน พิมพ์ข้อความ):**
```
=== ระบบตรวจสอบอายุเข้าผับ (Crash-Proof) ===
กรุณากรอกอายุของคุณ: ยี่สิบห้า

[ผลลัพธ์การตรวจสอบ]
โปรแกรมสามารถแปลงข้อความเป็นตัวเลขได้หรือไม่?: False
❌ ข้อผิดพลาด: สิ่งที่คุณกรอกไม่ใช่ตัวเลข กรุณาลองใหม่!
```

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: เส้นทางแห่งการสูญเสียข้อมูล**
จากโค้ดด้านล่างนี้ จงเดาผลลัพธ์สุดท้ายที่ปรากฏบนหน้าจอ โดยการทดทดเลขและทำกระบวนการ Casting ไว้ในใจทีละบรรทัด (บอกเลยว่าหลอกตาสุดๆ)
```csharp
double startingPoint = 5.8;
int step1 = (int)startingPoint;
double step2 = step1;
int finalResult = (int)step2 + 2;

Console.WriteLine(finalResult);
```

### **โจทย์ที่ 2: แปลงโค้ดมรณะ (The Crash Fixer)**
คุณเพื่อนบ้านกำลังเขียนโปรแกรมสุ่มแจกรางวัล แต่เค้าใช้โค้ดแบบนี้:
```csharp
Console.Write("กรุณาป้อนรหัสลุ้นรางวัล (ตัวเลขเท่านั้น): ");
string userInput = Console.ReadLine();
int luckyNumber = int.Parse(userInput);
Console.WriteLine("เลขที่คุณส่งชิงโชคคือ " + luckyNumber);
```
ปัญหาก็คือ เมื่อมีคนพิมพ์ช่องว่าง ` ` หรือพิมพ์คำว่า `ไม่มี` โปรแกรมมันก็จะ Crash หน้าจอแดงเดือดทันที
**ภารกิจ:** จงปรับปรุงโค้ดช่วงบรรทัดที่ 3 และ 4 ให้ปลอดภัย โดยเปลี่ยนไปใช้กระบวนการ **`int.TryParse`** แทน 

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** ระลึกไว้เสมอว่า การคาสต์จาก `double` เป็น `int` คือกระบวนการ "หั่นเศษทิ้ง" อย่างไร้เยื่อใย ไม่มีการปัดขึ้นใดๆ ทั้งสิ้น ดังนั้น 5.8 จะกลายเป็นอะไร? และพอกลับเป็น `double` อีกครั้ง เศษที่โดนหั่นทิ้งไปแล้วจะฟื้นคืนชีพกลับมาได้หรือไม่?
- **สำหรับโจทย์ 2:** คุณต้องสร้างตัวแปรรอรับผลลัพธ์ เช่น `int resultNumber;` จากนั้นเรียกใช้ `bool isValid = int.TryParse(userInput, out resultNumber);` เพื่อให้โปรแกรมปลอดภัย
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: มนตร์ตราแห่งการจัดรูปแบบ (String Formatting Magic)**

นอกจากการแปลงตัวเลขกลับเป็นข้อความด้วย `.ToString()` ธรรมดาๆ แล้ว เรายังสามารถใส่ **"รหัสลับ"** ลงไปในวงเล็บของ ToString เพื่อตกแต่งข้อความได้ด้วย (เรียกว่า Standard Numeric Format Strings)

**ภารกิจ:**
1. ให้กำหนดตัวแปร `double money = 1250000.55;`
2. จงค้นหาข้อมูลใน Google ด้วยคีย์เวิร์ด `C# ToString format specifiers`
3. เขียนโค้ดเพื่อพิมพ์ตัวแปร `money` ออกมาใน **3 รูปแบบ** นี้ให้ได้ โดยการใช้เครื่องมือ `.ToString("รหัส")` เท่านั้น ห้ามมานั่งเอา String มาต่อบวกกันเอง:
   - **แบบที่ 1:** แสดงผลลัพธ์เป็นรูปแบบสกุลเงิน (มีเครื่องหมายสกุลเงินและลูกน้ำ: `฿1,250,000.55`) *[คำใบ้: ตัวอักษร C]*
   - **แบบที่ 2:** แสดงผลลัพธ์เป็นรูปแบบตัวเลขสวยงามที่มีลูกน้ำแบ่งหลักพัน ไม่มีจุดทศนิยม (`1,250,001` - สังเกตว่ามันปัดเศษขึ้นด้วย!) *[คำใบ้: ตัวอักษร N]*
   - **แบบที่ 3:** แสดงผลลัพธ์ในรูปแบบเปอร์เซ็นต์ คูณร้อยอัตโนมัติ (`125,000,055.00 %`) *[คำใบ้: ตัวอักษร P]*

ลองทำดู แล้วคุณจะพบว่ากระบวนการแปลงข้อมูลจากตัวเลขสู่หน้าจอนั้นทรงพลังเพียงใด!

**แนวคำตอบ:** เป็นกลไกการรักษาความปลอดภัยของข้อมูล (Data Integrity) 
- `int` ใช้หน่วยความจำ 4 Bytes ไม่มีทศนิยม
- `double` ใช้หน่วยความจำ 8 Bytes และรองรับทศนิยม
การเอากล่อง 4 Bytes ไปยัดในกล่อง 8 Bytes (Widening) ข้อมูลจะไม่มีวันตกหล่นหายไปไหน คอมพิวเตอร์จึงทำให้เงียบๆ
แต่การเอา 8 Bytes กลับไปยัดใส่กล่อง 4 Bytes แถมกล่องนั้นไม่มีคุณสมบัติเก็บทศนิยมได้ (Narrowing) มันย่อมเกิดการ "หั่นและเฉือน" ข้อมูลทิ้งอย่างแน่นอน คอมไพเลอร์จึงมองว่านี่คือเรื่องคอขาดบาดตาย และบังคับให้มนุษย์โปรแกรมเมอร์ "เซ็นใบยินยอม (ใส่วงเล็บ Explicit Cast)" รับความเสี่ยงเรื่องข้อมูลสูญหายด้วยตัวเอง

**คำถาม 2:** ถ้าเรามีคำสั่ง `Convert.ToInt32()` แล้ว ทำไมวิศวกรซอฟต์แวร์ทั่วโลกถึงยังพร่ำบอกให้ใช้ `int.TryParse()` เป็นหลักในการรับค่าจากผู้ใช้?
**แนวคำตอบ:** 
เพราะ `Convert.ToInt32()` ถูกออกแบบมาให้ "พัง (Throw Exception)" เมื่อเจอข้อมูลขยะ (เช่น ตัวอักษร) การที่โปรแกรมพัง (Crash) กลางอากาศหมายความว่าแอปเด้งหลุด ผู้ใช้หัวเสีย 
และในแง่ของสถาปัตยกรรมคอมพิวเตอร์ การเกิด Exception (การปา Error) แต่ละครั้ง กินทรัพยากร CPU และ Memory อย่างมหาศาลเพื่อดึง Stack Trace
ในขณะที่ `int.TryParse()` ถูกออกแบบมาด้วยแนวคิด "Fail-Safe" คือเมื่อเจอข้อมูลขยะ มันก็แค่ยักไหล่ ตอบกลับมาแบบเรียบง่ายว่า `false` โดยไม่มีการปา Exception ใดๆ โปรแกรมจึงไหลลื่น ปลอดภัย และกินทรัพยากรน้อยกว่ามากในกรณีที่เกิดข้อผิดพลาด
