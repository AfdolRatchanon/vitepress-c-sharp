# เฉลย: var, const และ readonly

## 🎯 เฉลย Mini Exercise

### **โจทย์ที่ 1: นักสืบประเภทข้อมูล (Type Detective)**
เป้าหมายคือการฝึกฝนวิชา "อ่านใจ C# Compiler" เมื่อเห็นฝั่งขวามือ คุณต้องเดาได้ 100% ว่าฝั่งซ้ายของ `var` จะกลายเป็นชนิดข้อมูลอะไร

**วิเคราะห์และเฉลยรายข้อ:**
1. `var a = 42;`
   - **คำตอบ:** กลายเป็น **`int`**
   - **เหตุผล:** ตัวเลขจำนวนเต็มล้วนๆ ไม่มีจุดทศนิยม C# จะเหมาว่าเป็นค่ามาตรฐานคือ int เสมอ
2. `var b = 42.0;`
   - **คำตอบ:** กลายเป็น **`double`**
   - **เหตุผล:** ตัวเลขที่มีจุดทศนิยมแต่ไม่มีตัวอักษรกำกับด้านหลัง (Suffix) C# จะเหมาว่าเป็นสายพันธุ์ที่ใหญ่สุดคือ double ไว้ก่อนเพื่อกันข้อมูลสูญหาย
3. `var c = 'X';`
   - **คำตอบ:** กลายเป็น **`char`**
   - **เหตุผล:** สังเกตจากเครื่องหมาย Single Quote (`' '`) ที่ห่อหุ้มอักขระตัวเดียวไว้
4. `var d = 100m;`
   - **คำตอบ:** กลายเป็น **`decimal`**
   - **เหตุผล:** ตัวอักษร `m` ที่ห้อยท้าย เป็นตัวชี้ขาดที่บังคับให้คอมไพเลอร์รู้ว่านี่คือกล่องเก็บตัวเลขทางการเงิน
5. `var e = (5 > 3);`
   - **คำตอบ:** กลายเป็น **`bool`**
   - **เหตุผล:** ในวงเล็บเป็นสมการเชิงเปรียบเทียบตรรกศาสตร์ (5 มากกว่า 3 จริงหรือไม่?) ซึ่งผลลัพธ์ของสมการนี้มีค่าเป็นความจริง (true) ดังนั้นมันจึงกลายเป็นตรรกะ bool โดยอัตโนมัติ

### **โจทย์ที่ 2: ผ่าตัดโค้ดที่ไร้ระเบียบ (Refactoring Magic Numbers)**
**โจทย์ที่ให้มา (มรดกตกทอดที่มี Magic Number):**
```csharp
double baseSalary = 30000;
int otHours = 5;

double totalPay = baseSalary + (otHours * 450.50);
totalPay = totalPay - (totalPay * 0.05);

Console.WriteLine("รายรับสุทธิ: " + totalPay);
```

**กระบวนการแก้ปัญหา (Refactoring Process):**
ตัวเลข `450.50` และ `0.05` คือ Magic Numbers ที่แฝงตัวอยู่ ทำให้คนมาอ่านทีหลังอาจเข้าใจผิดได้ การเอาตัวเลขพวกนี้ไปฝังในตัวแปรค่าคงที่ (`const`) จะช่วยล็อคความปลอดภัยไม่ให้ใครเผลอไปแก้ไขโดยไม่ตั้งใจ และทำให้สูตรคำนวณอ่านง่ายราวกับภาษาอังกฤษ

**โค้ดที่ได้รับการผ่าตัดปรับปรุงแล้ว (Clean Code):**
```csharp
Console.WriteLine("=== ระบบคำนวณรายได้สุทธิ (Payroll System) ===");

// 1. ดึง Magic Number มาตั้งชื่อและล็อคเป็นค่าคงที่ไว้ด้านบนสุด (เพื่อให้อัปเดตง่ายในอนาคต)
// นิยมใช้ PascalCase สำหรับตัวแปรคงที่ระดับนี้
const double OvertimeHourlyRate = 450.50;
const double SocialSecurityTaxRate = 0.05; // 5%

// 2. ตัวแปรที่เป็น Input ปกติ (อาจจะรับมาจากผู้ใช้หรือดึงจากฐานข้อมูล)
double baseSalary = 30000;
int otHours = 5;

// 3. ลงมือคำนวณโดยใช้ชื่อตัวแปรที่สื่อความหมายแทนตัวเลขเปล่าๆ
// อ่านสูตรนี้แล้วเข้าใจทันทีโดยไม่ต้องพึ่งพาคอมเมนต์เลย!
double overtimeBonus = otHours * OvertimeHourlyRate;
double grossIncome = baseSalary + overtimeBonus;

double taxDeduction = grossIncome * SocialSecurityTaxRate;
double netIncome = grossIncome - taxDeduction;

// 4. แสดงผลลัพธ์
Console.WriteLine($"เงินเดือนฐาน : {baseSalary:N2} บาท");
Console.WriteLine($"ค่าล่วงเวลา  : {overtimeBonus:N2} บาท");
Console.WriteLine($"หักภาษี 5%  : -{taxDeduction:N2} บาท");
Console.WriteLine("--------------------------------");
Console.WriteLine($"รายได้สุทธิ  : {netIncome:N2} บาท");
```
*(เคล็ดลับ: การแยกการคำนวณออกมาทีละบรรทัด (เช่น แยก `overtimeBonus`) แทนที่จะยัดทุกอย่างลงไปในสูตรเดียว จะช่วยให้การหาบั๊ก (Debugging) ทำได้ง่ายขึ้นมาก!)*

## 🔥 เฉลย Challenge (มหกรรมสับสน var VS dynamic)

**ผลการวิจัยเชิงทดลอง:**
หากคุณนำโค้ดที่ให้ไปรันใน Visual Studio หรือ VS Code คุณจะพบเหตุการณ์ต่อไปนี้:

1. **ฝั่ง `var` (Strongly Typed):**
   ```csharp
   var box1 = 10;
   box1 = "Hello"; // ❌ บรรทัดนี้จะขึ้นแถบแดง (Compile Error)
   ```
   **ผลลัพธ์:** พังตั้งแต่ยังไม่เริ่มกดปุ่ม Run! คอมไพเลอร์บอกว่าไม่สามารถเอา string ใส่ใน int ได้ (ชี้ให้เห็นว่า C# ยังเข้มงวด 100% var แค่เดาให้ตอนแรกเฉยๆ)

2. **ฝั่ง `dynamic` (Weakly Typed):**
   ```csharp
   dynamic box2 = 10;
   box2 = "Hello";    // ✅ บรรทัดนี้ไร้ซึ่งขีดแดงใดๆ
   Console.WriteLine(box2);
   ```
   **ผลลัพธ์:** โปรแกรมรันผ่านฉลุย และพิมพ์คำว่า `Hello` ออกมาหน้าตาเฉย! มันยอมเปลี่ยนรูปร่างจากตัวเลขกลายเป็นข้อความได้ตามใจชอบเหมือนภาษา Python

**คำถามท้าทาย: แล้วถ้า `dynamic` สะดวกขนาดนี้ ทำไมคนถึงห้ามใช้?**
**เฉลยและเหตุผลทางวิศวกรรม (The Trade-offs):**
การใช้ `dynamic` เป็นการ "ปิดระบบป้องกันตัวเองของ C#" (Bypassing the Type Checker) สิ่งที่แลกมากับความสะดวกสบายนี้ มีราคาแพงมหาศาล:
1. **พังตอนทำงาน (Runtime Crash):** สมมติคุณเผลอไปเรียก `box2.ToUpper()` ทั้งที่มันมีตัวเลข 10 อยู่ข้างใน โปรแกรมจะไม่ขึ้น Error สีแดงเตือนคุณเลย (เพราะมันปิดระบบป้องกันไปแล้ว) แต่มันจะไปพังบึ้มต่อหน้าต่อตาลูกค้าตอนที่กำลังใช้งานโปรแกรมแทน (ซึ่งเป็นฝันร้ายของโปรแกรมเมอร์)
2. **ระบบช่วยเหลือตาบอด (Loss of IntelliSense):** ปกติเวลาคุณพิมพ์ชื่อตัวแปรแล้วตามด้วยจุด (`.`) VS Code จะมีกล่องรายชื่อคำสั่ง (Autocomplete) เด้งขึ้นมาช่วยคุณ แต่พอเป็น `dynamic` ตัว VS Code จะมืดบอดทันที มันไม่รู้จะช่วยเดาอะไรให้คุณ คุณต้องพิมพ์ทุกฟังก์ชันให้ถูกเป๊ะแบบสะกดคำเองทั้งหมด
3. **กินทรัพยากรเครื่อง (Performance Overhead):** การใช้ `dynamic` บังคับให้ C# ต้องฝังระบบประเมินผลจิ๋ว (DLR - Dynamic Language Runtime) ตามตัวแปรนั้นไปด้วยทุกที่ ทำให้คอมพิวเตอร์ต้องทำงานหนักขึ้นและเปลืองหน่วยความจำกว่าการใช้ `var` หลายสิบเท่า

*(บทสรุป: `dynamic` มีไว้เพื่อเชื่อมต่อข้ามภาษา หรือใช้ทำไฟล์โครงสร้าง JSON ยืดหยุ่นๆ เท่านั้น ในการเขียนโค้ด C# ทั่วไป **จงกอด `var` เอาไว้ให้แน่นที่สุด!**)*
