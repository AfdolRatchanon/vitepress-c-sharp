# เฉลย: Numeric Types (ชนิดข้อมูลตัวเลขเชิงลึก)

## 🎯 เฉลย Mini Exercise

### **โจทย์ที่ 1: สถาปนิกโครงสร้างข้อมูล (Data Architect)**
เป้าหมายของโจทย์นี้คือการฝึกประเมิน "ขนาด" (Range) และ "ความละเอียด" (Precision) ของข้อมูลในชีวิตจริง ว่าควรแมปเข้ากับ Data Type ตัวไหนเพื่อความประหยัดและถูกต้องมากที่สุด

**วิเคราะห์และเฉลยรายข้อ:**
1. **`bloodPressure` (ความดันโลหิต เช่น 120)**
   - **คำตอบที่เหมาะสมที่สุด: `short` หรือ `int` (หรือ `byte` ถ้ามั่นใจ)**
   - **เหตุผล:** ความดันมนุษย์เป็นตัวเลขจำนวนเต็ม ไม่มีทศนิยม และไม่เกิน 300 แน่นอน (เกินนี้คือเสียชีวิตแล้ว) การใช้ `int` (4 Bytes) เป็นค่าเริ่มต้นก็เหมาะสม แต่ถ้าอยากรีดประสิทธิภาพหน่วยความจำบนอุปกรณ์สวมใส่ทางการแพทย์ (Wearable) การใช้ `short` (2 Bytes) หรือ `byte` (0-255) จะเป็นการออกแบบที่ชาญฉลาดมาก
2. **`patientHeight` (ส่วนสูงของคนไข้ เช่น 175.5)**
   - **คำตอบที่เหมาะสมที่สุด: `float` หรือ `double`**
   - **เหตุผล:** ต้องมีทศนิยมแน่นอน ส่วนสูงไม่ต้องการความแม่นยำระดับบัญชีธนาคาร (ต่างกัน 0.0001 ซม. ไม่ใช่ปัญหา) ดังนั้นจึงไม่จำเป็นต้องใช้ `decimal` ให้หนักเครื่อง การใช้ `double` หรือ `float` ถือว่าตรงโจทย์ที่สุด
3. **`hospitalBudget` (งบประมาณโรงพยาบาลรายปี เช่น 5,000,000,000)**
   - **คำตอบที่เหมาะสมที่สุด: `long` หรือ `decimal`**
   - **เหตุผล:** งบประมาณ 5 พันล้าน ทะลุเพดาน `int` ไปเรียบร้อยแล้ว (Overflow ที่ 2.1 พันล้าน) ถ้าเป็นตัวเลขจำนวนเต็มลอยๆ ก็ต้องใช้ `long` แต่เนื่องจากนี่คือ "เรื่องการเงิน" ตามหลักสถาปัตยกรรมที่ดี ควรเก็บด้วย `decimal` เพื่อป้องกันปัญหาการคำนวณภาษีหรือดอกเบี้ยคลาดเคลื่อนในอนาคต
4. **`roomRate` (ค่าเช่าห้องพักผู้ป่วย เช่น 2500.50)**
   - **คำตอบที่เหมาะสมที่สุด: `decimal`**
   - **เหตุผล:** กฎเหล็กของการเขียนโปรแกรมคือ "ถ้าเป็นเรื่องของเงินที่มีทศนิยม ต้องใช้ `decimal` เท่านั้น"
5. **`heartRate` (อัตราการเต้นของหัวใจต่อนาที เช่น 85)**
   - **คำตอบที่เหมาะสมที่สุด: `byte` หรือ `short`**
   - **เหตุผล:** เหมือนกับความดันโลหิต อัตราการเต้นของหัวใจไม่มีทศนิยม และไม่มีทางติดลบ (ไม่มีใครหัวใจเต้น -5 ครั้ง) และไม่น่าจะเกิน 255 ครั้งต่อนาที การใช้ `byte` จึงเป็นคำตอบที่โชว์ให้เห็นว่าสถาปนิกคนนี้เข้าใจขีดจำกัดมนุษย์และฮาร์ดแวร์

### **โจทย์ที่ 2: บั๊กตัวอักษร M (Suffix Bug)**
**โจทย์ที่ให้มา:**
```csharp
decimal productPrice = 999.99;
float discountRate = 0.15;
```

**วิเคราะห์สาเหตุ (Why it fails):**
ในภาษา C# ตัวเลขทศนิยมทุกตัวที่คุณพิมพ์ลอยๆ เข้าไปในโค้ด (Literal value) เช่น `999.99` คอมไพเลอร์จะเหมารวม (Default) ว่ามันเป็นชนิด **`double`** เสมอ! 
- บรรทัดแรก: คุณกำลังพยายามเอากล่อง `double` ไปยัดใส่กล่อง `decimal`
- บรรทัดที่สอง: คุณกำลังพยายามเอากล่อง `double` ไปยัดใส่กล่อง `float`
คอมไพเลอร์ของ C# ไม่อนุญาตให้เอาของข้ามสายพันธุ์ (ที่กลไกทศนิยมต่างกัน) มายัดใส่กันตรงๆ โดยไม่มีสัญลักษณ์บอกกำกับ มันจึงระเบิด Error Compile-time ออกมา

**การแก้ไขที่ถูกต้อง (The Fix):**
ต้องเติมอักขระระบุชนิดตัวเลข (Suffix) ต่อท้าย เพื่อบอก Compiler ให้ชัดเจนว่าตัวเลขนี้ตั้งใจสร้างมาเพื่อกล่องไหน
- `m` หรือ `M` สำหรับ `decimal` (จำง่ายๆ ว่า m คือ Money)
- `f` หรือ `F` สำหรับ `float` (จำง่ายๆ ว่า f คือ Float)

```csharp
decimal productPrice = 999.99m;  // ✅ มีตัว m (พิมพ์เล็กพิมพ์ใหญ่ก็ได้)
float discountRate = 0.15f;      // ✅ มีตัว f
```

## 🔥 เฉลย Challenge (การควบคุม Overflow ด้วยคำสั่ง `checked`)

**โจทย์:** หาข้อมูลเรื่องการใช้คำสั่งบล็อก `checked` เพื่อป้องกันไม่ให้ตัวเลขตีกลับไปเป็นค่าติดลบเงียบๆ (Silent Bug)
**เฉลยและวิธีทำ:**

เรานำโค้ดในตัวอย่างที่ 1 มาจับใส่ในบล็อกที่ล้อมรอบด้วยปีกกาของ `checked { }`
เมื่อใดก็ตามที่มีการบวกลบคูณหารจนทะลุขีดจำกัดภายในกล่องนี้ C# จะไม่ยอมให้เลขตีกลับเป็นค่าติดลบ แต่มันจะ "ฆ่าโปรแกรมทิ้ง (Crash)" ทันที ซึ่งในโลกวิศวกรรม การยอมให้โปรแกรมแครช ดีกว่าการปล่อยให้ตรรกะทำงานผิดพลาดโดยที่ไม่มีใครรู้!

```csharp
Console.WriteLine("=== ทดสอบระบบตรวจสอบ Overflow (Checked Context) ===");

int maxScore = int.MaxValue; 
Console.WriteLine($"คะแนนเริ่มต้น: {maxScore}");

try
{
    // โยนโค้ดที่สุ่มเสี่ยงเข้าไปในเขตตรวจตราของ checked
    checked
    {
        Console.WriteLine("กำลังบวกคะแนนเพิ่ม...");
        // บรรทัดนี้จะระเบิดทันที!
        maxScore = maxScore + 1;
        Console.WriteLine("ถ้าเห็นข้อความนี้แปลว่าไม่ระเบิด (ซึ่งเป็นไปไม่ได้)");
    }
}
catch (OverflowException ex)
{
    // catch คือเทคนิคขั้นสูงในการดักจับระเบิดไม่ให้โปรแกรมพัง (จะเรียนในบท Exception Handling)
    Console.WriteLine("💥 บึ้มมม! โปรแกรมตรวจพบการล้นของข้อมูล (OverflowException)!");
    Console.WriteLine($"คำอธิบายจากระบบ: {ex.Message}");
}

Console.WriteLine("คะแนนยังคงปลอดภัย ไม่ตีกลับเป็นเลขติดลบ");
```

**ผลลัพธ์ที่ได้จากการรัน:**
```text
=== ทดสอบระบบตรวจสอบ Overflow (Checked Context) ===
คะแนนเริ่มต้น: 2147483647
กำลังบวกคะแนนเพิ่ม...
💥 บึ้มมม! โปรแกรมตรวจพบการล้นของข้อมูล (OverflowException)!
คำอธิบายจากระบบ: Arithmetic operation resulted in an overflow.
คะแนนยังคงปลอดภัย ไม่ตีกลับเป็นเลขติดลบ
```

**สาระสำคัญ:** โดยปกติ C# จะอยู่ในสถานะ `unchecked` เพื่อความรวดเร็วในการประมวลผล (Performance) แต่ในระบบการเงินระดับสูง โปรแกรมเมอร์มักจะคลุมโค้ดทุกส่วนที่เกี่ยวกับการคิดดอกเบี้ยด้วย `checked` เสมอ เพื่อเป็นตาข่ายนิรภัยดักความผิดพลาด!
