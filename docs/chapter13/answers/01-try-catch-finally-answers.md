# เฉลย: try / catch / finally (ตาข่ายนิรภัยกันโปรแกรมร่วง)

## 🎯 เฉลย Mini Exercise

### **โจทย์ที่ 1: ชะตากรรมของตัวแปร (Variable Scope Curse)**
**เป้าหมาย:** สลักกฎเหล็กเรื่อง Scope ให้ฝังลึกถึงกระดูก เพราะมือใหม่ตกม้าตายเรื่องนี้เยอะมาก!

**เฉลยคำตอบระดับสถาปนิก:**
มันพังเพราะ **กฎกรงขัง (Scope Rule)** ของปีกกา `{ }` ครับ!
ตัวแปร `score` ถูกให้กำเนิด "ข้างใน" ปีกกาของ `try`... มันจึงถูกสาปให้มีชีวิตอยู่แค่ในกรงนั้นเท่านั้น ทันทีที่โค้ดวิ่งทะลุปีกกาออกมาเพื่อจะรันบรรทัดล่างสุด... ตัวแปร `score` จะถูกทำลายล้างระเหยหายไปในอากาศเรียบร้อยแล้ว (Garbage Collected)!
คอมไพเลอร์จึงแหกปากด่าคุณว่า *"บรรทัดสุดท้าย ตัวแปร `score` คือผี! ฉันไม่รู้จักมัน!"*

**วิธีแก้ทางสถาปัตยกรรม:** คุณต้องคลอดมันไว้ "นอกกรง" ก่อนครับ!
```csharp
int score = 0; // คลอดไว้ข้างนอก
try {
    score = 100; // เข้าไปปรับค่าในกรง
} catch (Exception ex) {
    Console.WriteLine("พัง!");
}
Console.WriteLine(score); // รอดตาย! ปริ้น 100 ได้สวยงาม!
```

### **โจทย์ที่ 2: วิกฤติกระสุนด้าน (The Swallowed Exception)**
**เป้าหมาย:** ทำลายค่านิยมผิดๆ ที่คิดว่า "แอปไม่แครช = โค้ดดี"

**เฉลย:**
วงการซอฟต์แวร์เรียกพฤติกรรมนี้ว่า **"การซุกขยะไว้ใต้พรม (Swallowing Exceptions)"** ซึ่งเป็นความผิดร้ายแรงขั้นสูงสุด (Anti-Pattern)!
ถึงแอปจะไม่แครช... แต่มันทำให้ความจริงบิดเบี้ยวครับ!
สมมติคุณสั่งโอนเงิน 1 ล้านบาท... แล้วเกิดเน็ตตัดกลางคัน (Exception)
โค้ดกระโดดเข้า `catch` ว่างๆ... แล้วก็เงียบกริบ ทำเหมือนไม่มีอะไรเกิดขึ้น! เดินหน้ารันโปรแกรมต่อไปหน้าตาเฉย!
ผลคือ: หน้าจอผู้ใช้ขึ้นว่า "โอนเงินสำเร็จ" แต่ในดาต้าเบสธนาคาร "เงินไม่ไป!"
พอผู้ใช้ฟ้องร้อง... ทีม Developer เปิดดู Log ไฟล์... อ้าว ว่างเปล่า! ตามรอยสืบหาต้นตอไม่ได้เลยตลอดกาล! (นี่แหละครับ ความหายนะระดับองค์กร จงจำไว้ว่า `catch` ว่างเปล่า คือฆาตกรเงียบ!)

## 🔥 เฉลย Challenge (บอทเดินหมากรุกตาบอด The Blind Chess Bot)

**เป้าหมาย:** โชว์ความพลิกแพลงของการเอา Exception มาประยุกต์ใช้เป็น "เบรกเกอร์ (Control Flow)" ในเกมหรือบ็อท

**กระบวนการคิดระดับสถาปนิก (Exception-based Flow):**
จริงๆ เราใช้ `if (position >= board.Length)` เพื่อกันหลุดกระดานก็ได้ครับ มันจะเร็วกว่า
แต่ในสถานการณ์บางอย่าง (เช่น การทดสอบเจาะระบบ หรือหาช่องโหว่) เรายอมปล่อยให้มันวิ่งไปชนกำแพงแรงๆ เพื่อให้เกิด Exception กระแทกหน้า แล้วใช้ `catch` เป็นตัวตัดสินใจ "จบเกม" ทันทีครับ!

```csharp
using System;

namespace ExceptionChessBot
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== บอทเดินหมากรุกตาบอด (The Blind Chess Bot) ===\n");

            string[] board = { "จุดเริ่มต้น", "ทุ่งหญ้า", "ป่า", "ปราสาท" };
            int position = 0; // ตำแหน่งเริ่ม

            Console.WriteLine(">> กด Enter เพื่อเดินหน้า 1 ช่อง (รัวๆ เลยก็ได้) <<\n");

            // ลูปนรก วิ่งไม่หยุด
            while (true)
            {
                Console.ReadLine(); // รอรับคำสั่งกด Enter จากผู้ใช้

                // 🌟 กางตาข่ายนิรภัยไว้ "ข้างในลูป"
                try
                {
                    // ลองพยายามปริ้นกระดาน (จุดเสี่ยงตาย!)
                    Console.WriteLine($"[ระบบ]: ตัวละครเดินมาถึง... {board[position]}");
                    
                    // เดินเสร็จ ขยับตำแหน่งไปรอรอบหน้า
                    position++; 
                }
                // ทันทีที่ position ขยับไปเป็น '4' (ซึ่ง Array ตันแค่ 3)... 
                // มันจะพังที่บรรทัด WriteLine ปีกกาล็อกหลุดกระจาย แล้วพุ่งทะยานลงมาตรงนี้ทันที!
                catch (IndexOutOfRangeException) // ใส่ชื่อคลาสปีศาจลงไปได้เลย ไม่ต้องประกาศตัวแปร 'ex' ก็ได้
                {
                    Console.ForegroundColor = ConsoleColor.Red;
                    Console.WriteLine("\n❌ ตู้มมม! คุณเดินตกขอบแผนที่! ตกเหวตาย!");
                    Console.ResetColor();

                    // ทำลายลูปนรกทิ้ง จบเกมอย่างสมเกียรติ!
                    break;
                }
            }

            Console.WriteLine("\n[ผู้บรรยาย]: จบเกมแล้ว บ๊ายบายยย~");

            /*
               ผลลัพธ์เวลาเล่น:
               [ระบบ]: ตัวละครเดินมาถึง... จุดเริ่มต้น
               [ระบบ]: ตัวละครเดินมาถึง... ทุ่งหญ้า
               [ระบบ]: ตัวละครเดินมาถึง... ป่า
               [ระบบ]: ตัวละครเดินมาถึง... ปราสาท
               ❌ ตู้มมม! คุณเดินตกขอบแผนที่! ตกเหวตาย!
               [ผู้บรรยาย]: จบเกมแล้ว บ๊ายบายยย~
            */
        }
    }
}
```

**สิ่งที่ได้เรียนรู้ (Architecture Lesson):**
เห็นไหมครับว่าเราไม่จำเป็นต้องตั้งชื่อตัวแปร `ex` ไว้ในวงเล็บ `catch` เสมอไป ถ้าเราไม่ได้อยากอ่านค่าความผิดพลาดของมัน (Discard Variable) เราแค่ใส่คลาส `catch (IndexOutOfRangeException)` ดื้อๆ แบบนี้ โค้ดก็จะสะอาดขึ้นอีกระดับนึงครับ! นี่คือท่าโปรดของสถาปนิกเลยครับ!
