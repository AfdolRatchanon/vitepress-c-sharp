# Exception Types (สายพันธุ์ปีศาจ Error)

> 💡 **เป้าหมาย:** สิ้นสุดการใช้ `catch (Exception ex)` แบบเหมารวม! ในโลกสถาปัตยกรรมระดับ Enterprise ข้อผิดพลาดมีหลากหลายสาเหตุ (เช่น กรอกผิด, หารศูนย์, ไฟล์หาย) บทนี้คุณจะได้เรียนรู้วิธี "ตั้งด่านสกัดเฉพาะเจาะจง (Specific Catch)" เพื่อให้โปรแกรมสามารถแก้ไขปัญหาได้ตรงจุดราวกับหมอผ่าตัด!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ลำดับขั้นสายเลือดของปีศาจ (The Exception Hierarchy)

ใน C#... `Exception` เป็นเพียงคลาสระดับปู่ทวด (Base Class) เท่านั้น! 
ไมโครซอฟต์ได้สืบทอดสายเลือดคลาสนี้ออกไปเป็นลูกหลานหลายร้อยสายพันธุ์ (และทั้งหมดลงท้ายด้วยคำว่า Exception) ตัวท็อปที่โปรแกรมเมอร์เจอบ่อยๆ ได้แก่:

- **`FormatException`**: เกิดเมื่อพยายามแปลงร่าง (Parse) ข้อมูลข้ามสายพันธุ์แล้วล้มเหลว เช่น แปลง `"abc"` เป็น `int`
- **`DivideByZeroException`**: เกิดเมื่อเอาเลขคณิตศาสตร์ไปหารด้วย `0`
- **`NullReferenceException`**: (บั๊กอันดับ 1 ของโลก) เกิดเมื่อคุณพยายามงัด Property ของออบเจกต์ที่ยังเป็น `null` (เช่น `student.Name` ทั้งๆ ที่ตัวแปร student ยังไม่ได้ `new`)
- **`IndexOutOfRangeException`**: (ยอดฮิตอันดับ 2) เกิดเมื่อพยายามล้วงมือลงไปใน Array ที่ไม่มีช่องนั้นอยู่ (เช่น มี 5 ช่อง แต่ไปล้วงช่องที่ 10)

---

### 2. ศิลปะการตั้งด่านซ้อนกัน (Multiple Catch Blocks)

คุณสามารถสร้างบล็อก `catch` หลายๆ อันเรียงต่อกันได้เหมือนการเขียน `if / else if`!
เพื่อให้เราสามารถ "ตอบสนอง (Handle)" กับ Error แต่ละสายพันธุ์ได้แตกต่างกัน

> ⚠️ **กฎเหล็กสถาปนิก (Catch Order Rule):** 
> "จงวางกับดักตัวเล็ก (Specific) ไว้ข้างบน... และวางกับดักตัวใหญ่ (Generic `Exception`) ไว้ล่างสุดเสมอ!" 
> เพราะระบบจะร่วงหล่นเช็คจากบนลงล่าง ถ้าคุณเอาปู่ทวด `Exception` ไว้บนสุด มันจะฮุบ Error ทุกชนิดบนโลกทิ้งหมด ทำให้บล็อกด้านล่างกลายเป็นหมัน (Compile พังทันที)!

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สถาปัตยกรรมระบบเครื่อง ATM (ATM Exception Handling)

เราจะสร้างระบบให้ผู้ใช้กรอกจำนวนเงินที่ต้องการถอน แล้วดูว่าระบบเราสามารถรับมือกับการ "ป่วน" ของผู้ใช้แต่ละรูปแบบได้อย่างชาญฉลาดแค่ไหน!

::: code-group
```csharp [Program.cs]
using System;

namespace SpecificExceptions
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบตู้ ATM ธนาคารซีชาร์ป ===\n");

            int accountBalance = 5000;

            try 
            {
                Console.Write(">> ระบุยอดเงินที่ต้องการถอน (บาท): ");
                string input = Console.ReadLine();

                // 💣 จุดเสี่ยงที่ 1: ถ้าพิมพ์ "หนึ่งพัน" แทนตัวเลข -> FormatException
                int withdrawAmount = int.Parse(input);

                // 💣 จุดเสี่ยงที่ 2: ลอจิกบั๊กแอบแฝง (สมมติระบบโง่ เอาเงินไปหารยอดเพื่อนับแบงก์ 1000)
                // ถ้ากรอกเลข 0 มา -> DivideByZeroException
                int numberOf1000Banknotes = withdrawAmount / withdrawAmount; 

                // 💣 จุดเสี่ยงที่ 3: ระบบเช็คยอด
                if (withdrawAmount > accountBalance) {
                    // การทำแบบนี้เดี๋ยวเราจะได้เรียนเต็มๆ ในหัวข้อถัดไปครับ (โยนระเบิดเอง)
                    throw new InvalidOperationException("ยอดเงินของคุณไม่เพียงพอ!");
                }

                Console.ForegroundColor = ConsoleColor.Green;
                Console.WriteLine($"\n✅ ตู้กำลังจ่ายแบงก์... รับเงิน {withdrawAmount} บาท");
                Console.ResetColor();
            }
            // ==========================================================
            // ด่านที่ 1: ดักจับคนกรอกตัวหนังสือ (ตอบโต้ด้วยการด่าแบบสุภาพ)
            // ==========================================================
            catch (FormatException ex) 
            {
                Console.ForegroundColor = ConsoleColor.Yellow;
                Console.WriteLine("⚠️ [ระบบ]: คุณกรอกตัวอักษรมาทำไม!? กรุณากรอกเป็นตัวเลขเท่านั้น!");
                Console.ResetColor();
            }
            // ==========================================================
            // ด่านที่ 2: ดักจับคนป่วนกรอกเลข 0
            // ==========================================================
            catch (DivideByZeroException ex) 
            {
                Console.ForegroundColor = ConsoleColor.Yellow;
                Console.WriteLine("⚠️ [ระบบ]: จะมาถอนเงิน 0 บาททำไมเล่า! เสียเวลาตู้!");
                Console.ResetColor();
            }
            // ==========================================================
            // ด่านที่ 3: ดักจับลอจิกถอนเงินเกิน
            // ==========================================================
            catch (InvalidOperationException ex) 
            {
                Console.ForegroundColor = ConsoleColor.Magenta;
                Console.WriteLine($"💳 [ธนาคาร]: ธุรกรรมถูกปฏิเสธ! {ex.Message}");
                Console.ResetColor();
            }
            // ==========================================================
            // ด่านสุดท้าย: ตาข่ายก้นเหว (Catch-All) ดักจับทุกสิ่งที่คาดไม่ถึง
            // ==========================================================
            catch (Exception ex) 
            {
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("❌ [วิกฤติ]: ตู้ ATM ขัดข้องรุนแรง แจ้งช่างซ่อมบำรุงด่วน!");
                Console.WriteLine($"Log: {ex.Message}");
                Console.ResetColor();
            }
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: การเรียงลำดับที่พังพินาศ (The Hierarchy Crash)**
พิจารณาโค้ดการวางด่านสกัดต่อไปนี้:
```csharp
try {
    int[] nums = new int[2];
    nums[5] = 10; // ระเบิดชัวร์
} 
catch (Exception ex) {
    Console.WriteLine("จับได้หมด!");
}
catch (IndexOutOfRangeException ex) {
    Console.WriteLine("คุณล้วงทะลุ Array!");
}
```
**คำถาม:** เกิดอะไรขึ้นกับโค้ดชุดนี้เวลาสั่ง Compile รันโปรแกรม? (รันผ่านไหม? ถ้าไม่ ทำไมถึงไม่ผ่าน?)

### **โจทย์ที่ 2: อสูรกายล่องหน (The NullReferenceException)**
นี่คือบั๊กที่คนเกลียดที่สุดในโลก:
```csharp
string name = null;
try {
    int length = name.Length; // 💥 พังตรงนี้!
} catch (NullReferenceException ex) {
    Console.WriteLine("คุณกำลังเล่นกับของว่างเปล่า!");
}
```
**คำถามทางสถาปัตยกรรม:** แทนที่เราจะใช้ `try/catch` มาดัก `NullReferenceException` แบบในตัวอย่างนี้... สถาปนิกซีเนียร์มักจะแนะนำให้ **"หลีกเลี่ยง"** การดักมันด้วย try/catch แต่ให้ไปใช้ท่าอื่นแทน ท่าที่ว่านั้นคืออะไรครับ? (มันประหยัด CPU กว่ามหาศาล)

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **แดงเถือก Compile ไม่ผ่านครับ!**
  คอมไพเลอร์ฉลาดพอที่จะด่าคุณว่า: "เห้ย! เอ็งเอาตาข่ายตาถี่ (`Exception` ตัวท็อปสุด) ไปขึงไว้ด่านแรก แล้วตาข่ายด่านสอง (`IndexOutOfRange`) เอ็งจะได้จับอะไรวะ!?" กฎคือต้องเอาคลาสลูกไว้บนสุด แล้วเอาคลาสแม่ (`Exception`) ไว้ล่างสุดเสมอครับ!
- **สำหรับโจทย์ 2:** **ให้ใช้การเขียนตรรกะเช็ค `if (name != null)` ดักล่วงหน้าเอาครับ! (Defensive Programming)**
  กฎเหล็กของสถาปนิกคือ: **"Exception ควรมีไว้รองรับสถานการณ์ที่ 'เหนือการควบคุม' เท่านั้น!"** 
  อะไรก็ตามที่เราสามารถใช้ `if` เช็คป้องกันล่วงหน้าได้ (เช่น ตรวจก่อนว่าของเป็น null ไหม) ให้ใช้ `if` ครับ! เพราะการปล่อยให้เกิด Exception ขึ้นมาในแรม 1 ครั้ง มันกินทรัพยากร CPU และ RAM มหาศาลกว่าการประมวลผล `if` หลายพันเท่าครับ! (try/catch มีไว้จับของหนักๆ เช่น เน็ตหลุด หรือ ไฟล์พัง เท่านั้น)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ทีมกู้ภัยรหัสลับ (The Decoder Rescue Team)**

**ภารกิจของคุณ:** คุณได้รับข้อมูลรหัสลับที่อาจจะปนเปื้อนไวรัส จงกางตาข่าย 3 ชั้นเพื่อคัดกรองมันให้รอดปลอดภัย!

**ข้อมูลตั้งต้น:**
```csharp
string secretData = "99999999999999999999999"; 
```

**เป้าหมาย:** 
1. สร้างบล็อก `try` แล้วพยายามแปลง `secretData` ให้กลายเป็น `int` ด้วยคำสั่ง `int.Parse()`
2. สร้าง `catch` ด่านที่ 1 ดักจับ `FormatException` (กรณีเป็นตัวหนังสือ) แล้วปริ้น: "ไวรัสตัวหนังสือ!"
3. สร้าง `catch` ด่านที่ 2 ดักจับ **`OverflowException`** (กรณีตัวเลขมีขนาดใหญ่เกินกว่ากล่อง `int` จะรับไหว) แล้วปริ้น: "โอเวอร์โหลด! ตัวเลขมหาศาลเกินไป!"
4. สร้าง `catch` ด่านสุดท้าย ดักจับ `Exception` ธรรมดา แล้วปริ้น: "ระเบิดปริศนา!"

จงเขียนโค้ดและรันดูว่ารหัสลับชุดนี้ (`999999...` 20 กว่าตัว) จะพุ่งไปชนกับด่านตาข่ายด่านไหน! 
*(ใบ้: กล่อง `int` เก็บเลขได้เต็มที่แค่ 2 พันล้าน (หลักสิบหลัก) เท่านั้นครับ!)*

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในบรรทัดที่ใช้ `catch (DivideByZeroException ex)` ผมเห็นบางคนเขียนย่อๆ เป็น `catch (DivideByZeroException)` โดยไม่ยอมตั้งชื่อตัวแปร `ex` มันทำแบบนี้ได้ด้วยเหรอครับ?
**แนวคำตอบ:** **ทำได้ครับ และเป็นที่นิยมด้วย!**
ถ้าในบล็อก catch ของคุณ "คุณแค่ต้องการปริ้นข้อความด่าผู้ใช้เฉยๆ" โดยไม่ได้สนใจเลยว่าข้างในตัวแปร `ex.Message` มันฟ้องว่าอะไร (เพราะคุณรู้ดีอยู่แล้วว่า error นี้คือหารศูนย์แน่นอน)... คุณก็สามารถละทิ้งการตั้งชื่อตัวแปร (Discard) ทิ้งไปได้เลยครับ เพื่อให้โค้ดสั้นลง:
`catch (FormatException) { Console.WriteLine("กรอกผิดโว้ย!"); }` โคตรคลีน!

**คำถาม 2:** สมมติผมดัก error ได้ที่ด่าน `FormatException` แล้ว... แต่ผมตัดสินใจว่า "เห้ย เรื่องใหญ่ว่ะ ฉันจัดการเองไม่ไหว" ผมอยากจะโยนระเบิดลูกนี้ "เด้งต่อไปหาหัวหน้า (ให้แอปแครชไปเลย)" ผมจะทำได้ไหมครับ?
**แนวคำตอบ:** **ทำได้ครับ เรียกว่าการ Re-throw (โยนระเบิดซ้ำ)**
ในบล็อก catch ของคุณ คุณแค่เพิ่มคำสั่งสั้นๆ คำเดียวคือ **`throw;`** (ไม่มีอะไรต่อท้าย)
ระบบจะทำการจับ Exception ตัวเดิม (พร้อมกับหลักฐานสถานที่เกิดเหตุทั้งหมด) โยนพุ่งทะลุเพดานขึ้นไปหาระบบปฏิบัติการให้ฆ่าแอปทิ้งทันทีครับ! ท่านี้สถาปนิกใช้บ่อยมากเวลาต้องการ "แค่แอบดักปริ้น Log บันทึกประวัติลงไฟล์" แล้วค่อยโยนระเบิดต่อให้แอปพังอย่างสมเกียรติครับ!
:::
