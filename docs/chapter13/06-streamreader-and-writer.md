# StreamReader & Writer (การสับย่อยกระแสน้ำ)

> 💡 **เป้าหมาย:** ก้าวข้ามขีดจำกัดของ RAM! ฟังก์ชันสายด่วน (ReadAllText) ในบทที่แล้วมีจุดอ่อนร้ายแรงคือมันจะดูดไฟล์ทั้งก้อนขึ้นมาในหน่วยความจำ (RAM) ถ้าไฟล์ใหญ่มหาศาลแอปจะแครชตาย! บทนี้คุณจะได้เรียนรู้วิชา **Stream (สายน้ำ)** ทักษะระดับสถาปนิกที่จะทำให้คุณสามารถอ่านไฟล์ขนาด 100GB ได้อย่างชิลๆ โดยใช้ RAM ไปแค่จิ๊บจ๊อย!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ทำไมต้องใช้ Stream? (ปรัชญาของสายน้ำ)

ลองจินตนาการว่าคุณต้องขนน้ำ 100 ลิตรจากบ่อ ไปใส่แทงค์:
- **`File.ReadAllText` (สายโลภ):** มันพยายามเสกถังน้ำขนาดยักษ์ 100 ลิตร (RAM) ขึ้นมา เพื่อตักพรวดเดียว! ถ้าถังแตก (RAM หมด) ก็ตายหมู่
- **`StreamReader` (สายน้ำ):** มันใช้ "ท่อแป๊บ" เล็กๆ จุ่มลงไป... แล้วค่อยๆ เปิดวาล์วให้ข้อมูลไหลมา "ทีละหยด (ทีละบรรทัด)" พออ่านบรรทัดนั้นเสร็จ ก็โยนทิ้งไป แล้วดูดบรรทัดใหม่มาแทน! 
**ผลลัพธ์:** คุณสามารถขนไฟล์ 100GB ข้ามภูเขาได้ โดยใช้ RAM เท่ากับกะละมังเล็กๆ แค่ 1 ใบเท่านั้น! (Memory-efficient สุดๆ)

---

### 2. อาวุธคู่กาย `StreamReader` และ `StreamWriter`

แก๊งนี้เกิดมาเพื่ออ่าน/เขียน "ข้อความบรรทัดยาวๆ" โดยเฉพาะ (Text Streams)

- **`StreamReader`**: อาชีพ **สูบน้ำ** 
  ท่าไม้ตายคือ `.ReadLine()` (ดูดมา 1 บรรทัด พอสูบจนหมดบ่อ มันจะคืนค่า `null` ออกมาเป็นสัญญาณว่าหมดแล้ว)
- **`StreamWriter`**: อาชีพ **พ่นน้ำ** 
  ท่าไม้ตายคือ `.WriteLine()` (พ่นไป 1 บรรทัด แถมเคาะ Enter ให้สวยๆ ด้วย)

> ⚠️ **กฎเหล็กทวีคูณ:** แก๊ง Stream พวกนี้ มันเอาสายยางไปเสียบคาฮาร์ดดิสก์คุณไว้... ดังนั้น **"คุณต้องครอบมันด้วยปีกกา `using` เสมอ 100% แบบไม่มีข้อแม้!"** ไม่งั้นไฟล์จะโดนล็อกค้างคาดึงไม่ออก!

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สถาปัตยกรรมตัวกรองข้อมูลมหาศาล (Big Data Filter Pipeline)

เราจะจำลองสถานการณ์ที่มีไฟล์ Log โง่ๆ ขนาดใหญ่ยักษ์... แล้วเราจะเปิดท่อดูดมันมา "ทีละบรรทัด" เพื่อสแกนหาคำว่า "Error" ถ้าเจอ เราจะต่ออีกท่อพ่นมันไปเก็บแยกไว้ในฐานข้อมูลลับอีกไฟล์นึง! (นี่คือสถาปัตยกรรมของพวก Log Analyzer ทั่วโลกครับ!)

::: code-group
```csharp [Program.cs]
using System;
using System.IO;

namespace StreamPipelineConcept
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== เครื่องกรองกระแสน้ำข้อมูล (Big Data Log Filter) ===\n");

            string sourceFile = "massive_logs.txt";
            string targetFile = "error_only_logs.txt";

            // (จำลอง) สร้างไฟล์ต้นฉบับที่มีข้อมูลผสมกันมั่วซั่ว
            File.WriteAllText(sourceFile, 
                "INFO: เริ่มต้นระบบ\n" +
                "ERROR: ฐานข้อมูลพัง! ไฟไหม้เซิร์ฟเวอร์!\n" +
                "WARNING: RAM กำลังจะเต็ม\n" +
                "ERROR: หาปุ่มไม่เจอ ผู้ใช้โง่มาก\n" +
                "INFO: ปิดระบบ"
            );

            Console.WriteLine($">> เริ่มเดินเครื่องสูบน้ำจาก '{sourceFile}'");

            // ==========================================================
            // 🌟 สถาปัตยกรรมท่อแฝด (Double using Streams)
            // เปิดท่อดูด (Reader) และ ท่อพ่น (Writer) พร้อมๆ กัน!
            // ==========================================================
            
            int errorCount = 0;

            // กางม่านพลังป้องกันการล็อกไฟล์!
            using (StreamReader reader = new StreamReader(sourceFile))
            using (StreamWriter writer = new StreamWriter(targetFile, append: false))
            {
                string line;
                
                // กระชากมาทีละบรรทัด (ถ้าบรรทัดนั้นมีอยู่จริง ไม่ใช่ null)
                while ((line = reader.ReadLine()) != null)
                {
                    // ประมวลผลลอจิกทีละบรรทัด (RAM กินแค่นิดเดียว เพราะถืออยู่บรรทัดเดียว!)
                    if (line.Contains("ERROR:"))
                    {
                        // ถ้าเจอขยะติดเชื้อ ก็พ่นถ่ายเทไปสู่อีกไฟล์ทันที!
                        writer.WriteLine(line);
                        errorCount++;
                    }
                }
            } // ทันทีที่หลุดปีกกานี้ ท่อทั้ง 2 เส้นจะถูกกระชากปลั๊กทิ้งอย่างปลอดภัย!

            // ==========================================================
            Console.ForegroundColor = ConsoleColor.Green;
            Console.WriteLine($"\n✅ กรองเสร็จสิ้น! พบขยะเชื้อโรคจำนวน {errorCount} ชิ้น");
            Console.WriteLine($"✅ ข้อมูลถูกแยกไปเก็บไว้ที่ '{targetFile}' อย่างปลอดภัยแล้ว");
            Console.ResetColor();

            // ลองโหลดไฟล์ปลายทางมาดูผลลัพธ์
            Console.WriteLine("\n[เปิดดูเนื้อหาในหลุมหลบภัย error_only_logs.txt]:");
            Console.WriteLine(File.ReadAllText(targetFile));
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: อาการลืมกดชักโครก (The Buffer Flush)**
ในการใช้ `StreamWriter` บางครั้งคุณอาจจะเจอเหตุการณ์หลอนๆ คือ: สั่ง `writer.Write("Hello")` ไปแล้ว... แต่พอไปเปิดไฟล์ดูจริงๆ อ้าว! ไฟล์ดันว่างเปล่า! ข้อความไม่ยอมลงไปในฮาร์ดดิสก์! (เหมือนมีผีหลอก)
**คำถาม:** เหตุการณ์นี้มันไปติดอยู่ในสถานที่ที่เรียกว่า "ท่อพักน้ำ (Buffer)" ครับ คุณรู้ไหมว่าทำไมเราต้องมี Buffer? และคำสั่งใดที่จะ "ถีบ" น้ำในท่อพัก ให้พุ่งทะลักลงฮาร์ดดิสก์ทันทีแบบชัวร์ๆ?

### **โจทย์ที่ 2: วิกฤติกระสุนด้านในลูป While (The Infinite Death)**
มีเด็กฝึกงานลอกโค้ดจากบนเน็ต แต่ดันพิมพ์ผิดไปบรรทัดนึง:
```csharp
using (StreamReader reader = new StreamReader("data.txt")) {
    while (reader.ReadLine() != null) {
        Console.WriteLine("เจอ 1 บรรทัด!");
    }
}
```
**คำถาม:** โค้ดนี้ "ดูเหมือนจะ" ถูกต้อง แต่มันไม่ได้เก็บค่าที่อ่านมาไว้ในตัวแปร `line` เหมือนตัวอย่างด้านบน... การทำแบบนี้ จะเกิดผลกระทบอันตรายอะไรกับโปรแกรมครับ? (พิมพ์ออกจอมาได้ไหม?)

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **มันคือการเพิ่ม Performance ครับ และต้องใช้คำสั่ง `.Flush()`**
  ฮาร์ดดิสก์เป็นของที่หมุนช้ามาก (ช้ากว่า RAM เป็นพันเท่า) ถ้าคุณพ่นน้ำทีละหยดแล้วสั่งให้ฮาร์ดดิสก์เขียนทีละหยด คอมจะค้างครับ! StreamWriter เลยต้องมี "ถังพักน้ำ (Buffer)" รอให้หยดน้ำสะสมจนเต็มถังก่อน แล้วค่อยเทสาดลงฮาร์ดดิสก์ทีเดียวตูม! 
  แต่ถ้าคุณด่วนใจร้อน อยากให้มันลงเดี๋ยวนี้! คุณต้องสั่งเตะถังด้วย **`writer.Flush();`** ครับ! (แต่ไม่ต้องห่วง ทันทีที่วงเล็บ `using` ทำงานปิดท่อ มันจะสั่ง Flush ถังนัดสุดท้ายเททิ้งให้เราอัตโนมัติอยู่ดีครับ!)
- **สำหรับโจทย์ 2:** **ข้อมูลหายสาบสูญครับ! แถมคุมไม่ได้ด้วย!**
  คำสั่ง `.ReadLine()` คือการ **"สูบน้ำแล้วโยนทิ้งเลย"** ครับ ทันทีที่คุณเรียกมันในช่อง `while(...)` มันจะไปสูบบรรทัดที่ 1 ขึ้นมาเช็คว่าไม่ใช่ null ใช่ไหม... แล้วพอมันไหลลงมาในปีกกา คุณดันปริ้นแค่ข้อความโง่ๆ... แปลว่าเนื้อหาของบรรทัดที่ 1 คุณโยนมันทิ้งลงพื้นไปแล้วครับ กู้กลับมาไม่ได้แล้ว! 
  **สถาปนิกจึงบังคับเสมอว่า:** ต้องเอาตัวแปรมารอรับมันในวงเล็บ while เลย `while((line = reader.ReadLine()) != null)` แบบนี้คุณถึงจะเอา `line` ไปปริ้นหรือเอาไปใช้งานต่อในปีกกาได้ครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ทีมกู้ภัยไฟล์ยักษ์ (The Giant Log Parser)**

**ภารกิจของคุณ:** จงเขียนลอจิกตามหาบรรทัดที่ซ่อนอยู่ในภูเขาข้อมูล!

**กติกาและเป้าหมาย:** 
1. สมมติว่าไฟล์ `monster.txt` มีข้อมูลอยู่ **1 ล้านบรรทัด** (ถ้าโหลดทั้งก้อนด้วย File.ReadAllText แรมพังชัวร์)
2. สถาปนิกมอบภารกิจให้คุณหาว่า **"ในไฟล์นี้ มีบรรทัดที่เขียนคำว่า 'HACKER' ซ่อนอยู่บ้างไหม?"**
3. จงเขียนโค้ด `using` คุม `StreamReader` แล้วใช้ลูป `while` ลุยเข้าไปทีละบรรทัด
4. **ความท้าทาย:** ทันทีที่คุณสูบน้ำเจอคำว่า "HACKER" เป็นบรรทัดแรก... คุณต้อง **"ตัดท่อทิ้ง (break)"** ทันที ห้ามสูบน้ำต่อเด็ดขาดเพื่อประหยัด CPU! 
5. ให้ประกาศตัวแปร `bool isHacked = false;` ไว้ข้างนอก ถ้าเจอก็ปรับเป็น true 

จงเขียนแค่ท่อนลอจิกนี้ให้สวยงามที่สุด นี่คือข้อสอบสัมภาษณ์งานของบริษัทข้ามชาติ ที่วัดความเข้าใจเรื่อง Memory Efficiency (การประหยัดแรม) ล้วนๆ เลยครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในวงเล็บ `new StreamWriter(targetFile, append: false)` คำว่า `append` แปลว่าอะไรครับ?
**แนวคำตอบ:** **มันคือสวิตช์ควบคุมปรัชญาการทำลายล้างครับ!**
- ถ้า `append: false` (หรือทิ้งไว้ว่างๆ): มันจะสวมบทตัวร้าย! ทันทีที่ท่อเหล็กแทงเข้าไป มันจะล้างข้อมูลเก่าในไฟล์ทิ้งจนเกลี้ยง (Overwrite) เริ่มนับ 1 ใหม่!
- ถ้า `append: true`: มันจะสวมบทนักต่อเติม! มันจะแทงท่อเหล็กทะลุไปโผล่ที่ "บรรทัดล่างสุด" ของไฟล์เก่าเงียบๆ แล้วพ่นข้อมูลใหม่ต่อท้ายยาวลงไปเรื่อยๆ ครับ! (เหมาะกับระบบบันทึก Log รายวันสุดๆ)

**คำถาม 2:** แล้วถ้าเกิดฮาร์ดดิสก์เต็มกลางคัน ระหว่างที่ `StreamWriter` กำลังพ่นน้ำอยู่ล่ะครับ? โปรแกรมผมจะเป็นอะไรไหม?
**แนวคำตอบ:** **ระเบิดเถิดเทิงครับ! (IOException) แต่ไฟล์คุณจะไม่พังตายฟรีนะ!**
ถ้าฮาร์ดดิสก์เต็ม มันจะโยน Exception ปะทะหน้าคุณกลางอากาศ! และกระชากโค้ดคุณหลุดลอยเด้งออกจากวงเล็บ `{ }` ของ `using` ทันที!
แต่... ความสุดยอดของ `using` คือ ทันทีที่คุณกระเด็นหลุดออกมา มันจะเอื้อมมือไปกดเรียก `.Dispose()` เพื่อปิดท่อให้คุณก่อนที่แอปจะพังครับ! 
ผลลัพธ์คือ ข้อมูลส่วนที่พ่นไปก่อนหน้าจะถูก "เซฟลงไฟล์เท่าที่เซฟได้" อย่างปลอดภัย ไฟล์ไม่ Corrupt ครับ! (เห็นความสำคัญของตาข่าย using หรือยังครับ!)
:::
