# try / catch / finally (ตาข่ายนิรภัยกันโปรแกรมร่วง)

> 💡 **เป้าหมาย:** ทำความเข้าใจกลไกการเกิด "จอแดง (Crash)" ของโปรแกรม และเรียนรู้วิธีกางตาข่ายนิรภัย `try/catch` เพื่อรองรับแรงกระแทก ให้โปรแกรมของคุณสามารถมีชีวิตรอดต่อไปได้ แม้จะเกิดข้อผิดพลาดร้ายแรงก็ตาม!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. Exception คืออะไร?

ในภาษา C# คำว่า **Exception (ข้อยกเว้น/เหตุขัดข้อง)** คือ "ออบเจกต์" ตัวหนึ่งที่ระบบจะเสก (Throw) มันพุ่งออกมาจู่โจมโค้ดของคุณ เมื่อคอมพิวเตอร์เจอกับสถานการณ์ที่ "หาทางออกไม่ได้"

เช่น: 
- คุณสั่งเอาเลข `10 / 0` (คอมพิวเตอร์คำนวณไม่ได้ มันจึงปาระเบิด `DivideByZeroException` ใส่คุณ)
- คุณสั่งแปลงข้อความ `"Hello"` เป็นเลข `int` (คอมพิวเตอร์แปลงไม่ได้ มันจึงปา `FormatException` ใส่คุณ)

> ⚠️ **กฎของระบบ:** ถ้ามี Exception พุ่งออกมา แล้วคุณ "ไม่เอามือไปรับมันไว้"... Exception นั้นจะพุ่งทะลุไปถึงระบบปฏิบัติการ (Windows) แล้ว Windows จะสั่ง **"ฆ่าโปรแกรมของคุณทิ้งทันที!"** (นี่คืออาการแอปเด้งออกหน้าเดสก์ท็อปครับ)

---

### 2. สามทหารเสือ: Try, Catch, Finally

เพื่อป้องกันไม่ให้โปรแกรมตาย เราต้องสร้าง "ห้องนิรภัย" 3 โซน:

1. **`try { ... }` (โซนพื้นที่เสี่ยง):** 
   เอาโค้ดที่คุณคิดว่า "มีโอกาสพัง" ไปวางไว้ในนี้ ถ้าโค้ดพังตรงไหน มันจะหยุดทำงานบรรทัดนั้นทันที แล้วโยนความผิดไปให้บล็อก `catch` 
2. **`catch { ... }` (โซนพยาบาล):** 
   ตาข่ายที่เอาไว้รองรับระเบิด! ถ้าในบล็อก `try` เกิดระเบิดขึ้น โค้ดในบล็อกนี้จะทำงานทันที! (เรามักจะเขียนโค้ดแจ้งเตือนผู้ใช้ในนี้ เช่น "คุณกรอกตัวเลขผิดนะ!")
3. **`finally { ... }` (โซนพนักงานทำความสะอาด):** 
   *(ใส่หรือไม่ใส่ก็ได้)* นี่คือบล็อกพิเศษที่ **"ต้องทำงานเสมอ 100% ไม่ว่าโปรแกรมจะพังหรือไม่พังก็ตาม!"** สถาปนิกมักใช้บล็อกนี้ในการ "ปิดการเชื่อมต่อฐานข้อมูล" เพื่อป้องกัน Memory Leak

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สถาปัตยกรรมระบบเครื่องคิดเลขอัจฉริยะ

เราจะมาจำลองการรับข้อมูลจากผู้ใช้ (ซึ่งเป็นสิ่งมีชีวิตที่คาดเดาไม่ได้ที่สุดในโลก) และกางตาข่ายนิรภัยดักจับความมั่วซั่วของพวกเขาครับ!

::: code-group
```csharp [Program.cs]
using System;

namespace ExceptionBasics
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== โปรแกรมหารเลขสุดอันตราย ===");

            // ==========================================================
            // กางพื้นที่เสี่ยงภัย
            // ==========================================================
            try 
            {
                Console.Write(">> กรุณากรอกตัวตั้ง: ");
                string input1 = Console.ReadLine();
                int num1 = int.Parse(input1); // 💣 จุดเสี่ยงที่ 1: ถ้าพิมพ์อักษรมา พังแน่!

                Console.Write(">> กรุณากรอกตัวหาร: ");
                string input2 = Console.ReadLine();
                int num2 = int.Parse(input2); // 💣 จุดเสี่ยงที่ 2

                // 💣 จุดเสี่ยงที่ 3: ถ้า num2 เป็นเลข 0 โลกแตกแน่นอน!
                int result = num1 / num2; 
                
                Console.ForegroundColor = ConsoleColor.Green;
                Console.WriteLine($"\n✅ ผลลัพธ์คือ: {result}");
                Console.ResetColor();
                
                // สังเกต: ถ้าบรรทัดไหนพัง โค้ดบรรทัดด้านล่างในบล็อก try จะไม่ถูกทำงานเลย!
            }
            // ==========================================================
            // ตาข่ายรองรับแรงกระแทก (จับออบเจกต์ Exception ยัดใส่ตัวแปร 'ex')
            // ==========================================================
            catch (Exception ex) 
            {
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("\n❌ ตู้มมม! เกิดข้อผิดพลาดร้ายแรง!");
                
                // เราสามารถดึง "ข้อความอธิบาย Error" จากตัวแปร ex ออกมาดูได้
                Console.WriteLine($"[รายละเอียดจากระบบ]: {ex.Message}");
                Console.ResetColor();
                Console.WriteLine("-> ไม่ต้องตกใจ โปรแกรมยังทำงานต่อได้ ไม่เด้งดับ!");
            }
            // ==========================================================
            // ทีมทำความสะอาด (รันเสมอ 100%)
            // ==========================================================
            finally 
            {
                Console.ForegroundColor = ConsoleColor.Yellow;
                Console.WriteLine("\n🧹 [Finally]: สิ้นสุดกระบวนการคำนวณ (รันเสมอไม่ว่าพังหรือไม่พัง)");
                Console.ResetColor();
            }

            // พิสูจน์ว่าโปรแกรมไม่แครช:
            Console.WriteLine("\n>> โปรแกรมทำงานต่อบรรทัดล่างสุดได้อย่างปลอดภัย จบการทำงาน.");
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: ชะตากรรมของตัวแปร (Variable Scope Curse)**
มีโปรแกรมเมอร์ฝึกหัด พยายามเขียนโค้ดแบบนี้:
```csharp
try {
    int score = 100;
} catch (Exception ex) {
    Console.WriteLine("พัง!");
}
Console.WriteLine(score); // 💥 คอมไพเลอร์ขีดเส้นแดงปรี๊ดด!
```
**คำถาม:** ทำไมถึงเกิด Error บรรทัดสุดท้าย? และถ้าเขาอยากจะปริ้นค่าตัวแปร `score` ตรงนั้นจริงๆ เขาต้องแก้โครงสร้างโค้ดอย่างไร?

### **โจทย์ที่ 2: วิกฤติกระสุนด้าน (The Swallowed Exception)**
หัวหน้าซีเนียร์เห็นโค้ดของจูเนียร์คนหนึ่ง แล้วโกรธมาก สั่งให้แก้ทันที โค้ดมีหน้าตาแบบนี้:
```csharp
try {
    // โค้ดทำธุรกรรมการเงิน 1,000 บรรทัด
} catch (Exception ex) {
    // ปล่อยว่างไว้เฉยๆ (Empty Catch Block)
}
```
**คำถาม:** โค้ดนี้ Compile ผ่านและรันได้ปกติ แถมแอปไม่แครชด้วยซ้ำ! แต่ทำไมในทางสถาปัตยกรรม (Architecture) มันถึงเป็น "หายนะระดับองค์กร (Anti-Pattern)" ครับ?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **เพราะติด Scope ของปีกกา `{ }` ครับ!**
  ตัวแปรที่ถูกสร้าง (Declare) "ข้างใน" ปีกกาของ `try` มันจะมีชีวิตอยู่แค่ในปีกกานั้น พอหลุดออกมามันจะระเหยหายไปทันที (และบล็อก `catch` ก็มองไม่เห็นมันด้วยนะ!)
  **วิธีแก้:** ต้องประกาศตัวแปรไว้ "ข้างนอกและเหนือ" บล็อก try ครับ! (เช่น `int score = 0; try { score = 100; } ...`)
- **สำหรับโจทย์ 2:** **นี่คือการ "ซุกขยะไว้ใต้พรม (Exception Swallowing)" ครับ!**
  ถึงแอปจะไม่แครช... แต่มันแปลว่า "ถ้าโอนเงินล้มเหลว โปรแกรมจะเงียบกริบ ไม่มีการปริ้น Log ไม่มีการแจ้งเตือนผู้ใช้" ผู้ใช้ก็จะงงว่าเงินหายไปไหน ส่วนทีม Dev ก็จะไม่มีวันรู้เลยว่าระบบมีบั๊ก (เพราะไม่มี Error เด้งขึ้นหน้าจอ) ตามจับบั๊กไม่ได้ตลอดกาลครับ! (กฎเหล็ก: ห้ามปล่อย Catch ว่างเด็ดขาด อย่างน้อยต้องสั่งปริ้น Log ออกมา!)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: บอทเดินหมากรุกตาบอด (The Blind Chess Bot)**

**ภารกิจของคุณ:** จงเขียนลอจิกเดินกระดาน Array โดยใช้ `try/catch` เป็นตัวบังคับทิศทาง!

**ข้อมูลตั้งต้น:**
```csharp
string[] board = { "จุดเริ่มต้น", "ทุ่งหญ้า", "ป่า", "ปราสาท" };
```

**เป้าหมาย:** 
1. สร้างตัวแปร `int position = 0;` (ตำแหน่งปัจจุบัน)
2. วนลูป `while (true)` เพื่อให้ผู้ใช้กดปุ่ม "Enter" เพื่อเดินหน้า (+1 position) ไปเรื่อยๆ 
3. ในลูป ให้พยายามปริ้นข้อความ `Console.WriteLine(board[position]);` ออกมา
4. **ความท้าทาย:** เมื่อผู้ใช้เดินไปถึงตำแหน่งที่ 4 (ซึ่ง Array มีแค่ Index 0-3)... มันจะพังปังพินาศ (`IndexOutOfRangeException`) ทันที!
5. **จงใช้ `try/catch`** ไปครอบจุดนั้นไว้ ถ้ามันพัง (เดินตกขอบกระดาน) ให้ `catch` ทำงาน โดยปริ้นคำว่า "ตกเหวตาย! จบเกม!" แล้วใช้คำสั่ง `break;` เพื่อออกจากลูป while ให้สวยงาม!

นี่คือการประยุกต์ใช้ Exception แบบนอกกรอบ (Control Flow via Exception) ลองดูความเท่ของมันครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในเมื่อ `try/catch` มันกันแอปแครชได้แบบ 100% ทำไมเราไม่เอา `try/catch` ไปครอบทับโค้ดใน Main ทั้งกระบิ ตั้งแต่บรรทัดแรกยันบรรทัดสุดท้ายเลยล่ะครับ? ชีวิตจะได้ปลอดภัยตลอดกาล?
**แนวคำตอบ:** **ทำได้ครับ และเป็นสิ่งที่ "ระบบ Framework ใหญ่ๆ" ทำกันอยู่แล้วด้วย!**
ในโปรเจกต์ ASP.NET Core หรือ Unity 3D เบื้องหลังของมันจะมี `try/catch` ขนาดยักษ์ครอบแอปของคุณไว้อีกทีเรียบร้อยแล้วครับ (Global Exception Handler) เพื่อไม่ให้แอปตาย 
**แต่ทำไมเราถึงยังต้องมาเขียน `try/catch` ย่อยๆ เองอีก?** 
เพราะว่าถ้าเราปล่อยให้ Error พุ่งไปหาตัวขนาดยักษ์... โค้ดทั้งหมดที่เหลือจะ "หยุดรัน" ทันทีครับ! (แอปไม่ดับก็จริง แต่หน้าจอค้างไปแล้ว) การเขียนดักเป็นจุดย่อยๆ ทำให้เราสามารถ "ซ่อมแซมแผลเฉพาะจุด" แล้วสั่งให้โปรแกรม "เดินหน้าทำงานส่วนอื่นต่อไปได้" ครับ!

**คำถาม 2:** โค้ดในบล็อก `finally` รันเสมอ 100% จริงเหรอครับ? ถ้าผมแกล้งดึงปลั๊กคอมพิวเตอร์ออก มันจะรันไหม?
**แนวคำตอบ:** **ไม่รันครับ! (ฮา)** 
คำว่า "เสมอ" ในที่นี้หมายถึง ภายใต้สถานการณ์ที่ระบบปฏิบัติการ (OS) ยังปกตินะครับ
- ถ้ารันบล็อก try จบปกติ -> finally ทำงาน
- ถ้าเกิด Error แล้ว catch ดักได้ -> finally ทำงาน
- ถ้าเกิด Error แล้ว catch ดักไม่ได้ (แครช) -> finally ทำงานก่อนแครช!
- ถ้าในบล็อก try มีคำสั่ง `return` ออกจากฟังก์ชัน -> finally ทำงานขัดจังหวะก่อนที่จะ return จริงๆ! (หลอนมากขอบอก)
- **ข้อยกเว้น:** ถ้ามีคนดึงปลั๊กคอม, ปิด Process สดๆ ผ่าน Task Manager (End Task), หรือเรียกคำสั่ง `Environment.Exit(0)`... finally จะโดนเตะก้านคอและไม่ทำงานเด็ดขาดครับ!
:::
