# Closures (อาถรรพ์ตัวแปรแอบแฝง)

> 💡 **เป้าหมาย:** ระวังให้ดี! วิชา Lambda มีคำสาปแอบแฝงที่ชื่อว่า "Closure (โคลส-เชอร์)" บทนี้คุณจะได้เรียนรู้ "เวทมนตร์" ที่ยอมให้ Lambda สามารถดึงตัวแปรจากโลกภายนอกเข้าไปใช้ข้างในได้... และ "คำสาป" ที่จะตามหลอกหลอนคุณเมื่อมันไปทำงานร่วมกับลูป `for`! (นี่คือบั๊กยอดฮิตระดับตำนานที่โปรแกรมเมอร์ทุกคนต้องเคยโดนมาแล้ว 1 ครั้งในชีวิต!)

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. พลังเหนือธรรมชาติ: การรวบหัวรวบหาง (Capturing Variables)

ตามกฎปกติของฟังก์ชัน (Method) มันจะโลกส่วนตัวสูงมาก:
"ตัวแปรที่ประกาศใน Main() จะไม่มีวันถูกส่งเข้าไปในฟังก์ชันอื่นได้ ยกเว้นจะโยนผ่าน (parameter) เท่านั้น"

**แต่กฎนี้ใช้ไม่ได้กับ Lambda ครับ!**
Lambda เป็นปรสิตที่อาศัยอยู่ "ข้างใน" ฟังก์ชันอีกที ดังนั้นมันสามารถ "เอื้อมมือ" ออกมาหยิบตัวแปรของชาวบ้านที่อยู่รอบๆ ตัวมันเอาไปใช้งานหน้าตาเฉยได้เลย! (เราเรียกพฤติกรรมนี้ว่า **Capturing Variables**)

```csharp
void CreateBomb() 
{
    int timer = 5; // ตัวแปรธรรมดาในฟังก์ชัน CreateBomb

    // สร้างกล่องระเบิด (Lambda)
    Action bomb = () => {
        // 🌟 เวทมนตร์ Closure: Lambda แอบหยิบ timer มาใช้ได้ ทั้งๆ ที่ไม่ได้โยนเป็นพารามิเตอร์!
        Console.WriteLine($"ตูมมม! ระเบิดเวลาตั้งไว้ {timer} วินาที");
    };

    bomb(); // สั่งทำงาน
}
```
*ระบบเบื้องหลัง: คอมไพเลอร์ C# จะแอบเอาตัวแปร `timer` ไปแช่แข็ง (freeze) เก็บไว้ในคลาสลับ เพื่อให้ Lambda สามารถดึงมาใช้ได้ทุกเมื่อในอนาคต!*

### 2. คำสาปที่ซ่อนอยู่ (The State Mutation Curse)

ความน่ากลัวของ Closure คือ **"มันไม่ได้ก๊อปปี้ค่านะ! มันผูกวิญญาณติดกับตัวแปรต้นทางเลย!"**
แปลว่า ถ้าตัวแปรต้นทางถูก "เปลี่ยนค่า (Mutate)" ก่อนที่ปุ่ม Lambda จะถูกกด... Lambda จะทำงานด้วย "ค่าใหม่ล่าสุด" เสมอ!

```csharp
int timer = 5;
Action bomb = () => Console.WriteLine($"ตูม! เวลา {timer}");

// เปลี่ยนค่าตัวแปรก่อนที่จะสั่งรัน Lambda!
timer = 99; 

bomb(); // 💥 ผลลัพธ์คือ: ตูม! เวลา 99 (มันตามมาหลอกหลอนถึงที่!)
```

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: กับดักลูปมรณะ (The For-Loop Trap)

นี่คือบั๊กคลาสสิกที่สุดในประวัติศาสตร์ของ C# และ JavaScript! 
สมมติคุณอยากสร้างปุ่ม 3 ปุ่ม และอยากให้ปุ่มที่ 0 ปริ้นเลข 0, ปุ่มที่ 1 ปริ้นเลข 1, ปุ่มที่ 2 ปริ้นเลข 2... คุณเลยใช้ลูป `for` ช่วยสร้าง

::: code-group
```csharp [Program.cs]
using System;
using System.Collections.Generic;

namespace ClosureConcept
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ทดสอบอาถรรพ์ลูปมรณะ (Closure Trap) ===\n");

            // กล่องเก็บปุ่ม 3 ปุ่ม
            List<Action> buttons = new List<Action>();

            // 1. สร้างปุ่มด้วยลูป For (เหมือนจะฉลาด)
            for (int i = 0; i < 3; i++)
            {
                // เรายัดคำสั่งปริ้น i เข้าไปในปุ่ม
                // ⚠️ คำเตือน: Lambda จับวิญญาณตัวแปร 'i' ไว้! มันไม่ได้ก๊อปปี้ค่า!
                buttons.Add(() => Console.WriteLine($"ฉันคือปุ่มหมายเลข: {i}"));
            }

            // --- เมื่อลูปจบการทำงาน ค่าของ i ปัจจุบันคือ 3! ---

            Console.WriteLine(">> ทำการกดปุ่มทั้งหมด:\n");

            // 2. สั่งกดปุ่มทั้ง 3 ปุ่ม
            foreach (Action btn in buttons)
            {
                // 💥 พังพินาศ! ทุกปุ่มปริ้นเลข 3 หมดเลย!!!
                btn(); 
            }

            /*
               เหตุผลความพัง (Architecture Bug):
               Lambda ถูกเก็บไว้เฉยๆ ตอนมันถูกสร้างมันยังไม่ได้รัน! 
               พอมันตื่นขึ้นมามารันตอนจบ มันหันไปหาตัวแปร i... ซึ่งในขณะนั้น i วิ่งจนจบกระดานกลายเป็น 3 ไปเรียบร้อยแล้ว
               มันจึงดึงค่า 3 ไปใช้เหมือนกันหมดทุกปุ่มครับ!
            */

            Console.WriteLine("\n----------------------------------\n");
            Console.WriteLine("=== วิธีแก้บั๊ก (The Solution) ===\n");

            List<Action> goodButtons = new List<Action>();

            for (int i = 0; i < 3; i++)
            {
                // 🌟 ท่าไม้ตาย "สละวิญญาณ (Local Copy)"
                // เราสร้างตัวแปรท้องถิ่นใหม่กิ๊งในแต่ละรอบ เพื่อตัดขาดจากวิญญาณ i ตัวแม่!
                int safe_i = i; 

                // เอา safe_i ยัดเข้า Lambda แทน!
                goodButtons.Add(() => Console.WriteLine($"ปุ่มปลอดภัย: {safe_i}"));
            }

            foreach (Action btn in goodButtons)
            {
                btn(); // ✅ รอด! ปริ้น 0, 1, 2 สวยงามสมบูรณ์แบบ
            }
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: นับยอดคงเหลือ (The Account Balance)**
ลองอ่านการทำงานของโค้ดนี้:
```csharp
int money = 100;
Action buySnack = () => { money -= 20; };

buySnack();
buySnack();
Console.WriteLine(money);
```
**คำถาม:** ผลลัพธ์สุดท้ายที่ปริ้นออกมาคือเลขอะไร? นี่พิสูจน์คุณสมบัติอะไรของ Closure?

### **โจทย์ที่ 2: รอดหรือไม่รอด? (The Foreach Safety)**
เราเห็นลูป `for` พาพังมาแล้ว... แล้วถ้าเราใช้ลูป `foreach` ล่ะ?
```csharp
List<string> names = new List<string> { "A", "B" };
List<Action> actions = new List<Action>();

foreach (string name in names) {
    actions.Add(() => Console.WriteLine(name));
}

actions[0](); actions[1]();
```
**คำถาม:** โค้ดนี้จะปริ้น "B, B" (บั๊ก) หรือปริ้น "A, B" (ปกติ)? มีความแตกต่างทางสถาปัตยกรรมอย่างไรระหว่าง `for` และ `foreach` ในเรื่องนี้?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  ผลลัพธ์คือ **60** ครับ!
  มันพิสูจน์ว่า: "Lambda ไม่ได้แค่ก๊อปปี้ค่าไปอ่านเฉยๆ (Read-only) แต่เราสามารถ **แก้ไขค่า (Mutate)** ตัวแปรต้นทางได้จากภายในตัว Lambda เลย!" (ตัวแปร money กลายเป็นพื้นที่ส่วนกลางที่ถูกรุมใช้)
- **สำหรับโจทย์ 2:** **ตอบว่ารอดครับ! ปริ้น A, B ตามปกติ!**
  นี่คือจุดหักมุมของประวัติศาสตร์ C# ครับ! ในสมัยก่อน C# 5.0 (เมื่อสิบปีที่แล้ว) ลูป foreach ก็บั๊กแบบเดียวกับ for ครับ (ปริ้น B, B)... 
  แต่โปรแกรมเมอร์ทั่วโลกด่ากันระงม ไมโครซอฟต์เลยแอบแก้โครงสร้างสถาปัตยกรรมใหม่ใน C# 5.0 ว่า "ทุกครั้งที่วนลูป foreach ให้เสกตัวแปร `name` ขึ้นมาเป็นตัวใหม่ซิงๆ (Local Scope) เสมอ!" ทำให้ปัจจุบันนี้ ลูป foreach "ปลอดภัยจากวิญญาณร้าย 100%" ครับ! (จงหลีกเลี่ยง `for` คู่กับ Lambda ถ้าไม่จำเป็นจริงๆ)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: มือสังหารรับจ้าง (The Bounty Hunter)**

**ภารกิจของคุณ:** จงสร้างระบบล็อกเป้าหมาย (Target Locking) ด้วยพลังของ Closure

1. สร้างคลาส `BountyHunter`
   - สร้างฟังก์ชัน `public Action LockTarget(string targetName)`
     *(อ่านดีๆ นะครับ: ฟังก์ชันนี้จะ **Return กล่อง Action (Lambda)** กลับมาให้ Main เอาไปรันทีหลัง)*
   - ในฟังก์ชันนี้ ให้ประกาศตัวแปร `int bullets = 3;`
   - เขียนให้มัน `return` ก้อน Lambda ออกไป โดยไส้ใน Lambda ให้ทำสิ่งนี้:
     - ถ้า `bullets > 0` ให้ลดกระสุน 1 นัด และปริ้นว่า `"ปัง! ยิง [เป้าหมาย] โดนเต็มๆ! (เหลือกระสุน [bullets] นัด)"`
     - ถ้า `bullets == 0` ให้ปริ้นว่า `"แกร๊ก! กระสุนหมด ยิง [เป้าหมาย] ไม่ได้แล้ว!"`

2. ใน `Main` (โลกภายนอก):
   - เสกนักล่า `BountyHunter johnWick = new BountyHunter();`
   - สั่งล็อกเป้า: `Action shootAction = johnWick.LockTarget("เอเลี่ยน");`
   - ทดสอบรันการยิง: ลองกดปุ่มรัน `shootAction();` รัวๆ 4 ครั้ง!

**เป้าหมายแฝง:** คุณจะเห็นความหลอนของ Closure อย่างแท้จริง... เพราะตัวแปร `bullets` มันถูกสร้างไว้ "ข้างในฟังก์ชัน LockTarget" ซึ่งตามทฤษฎี พอฟังก์ชันนี้ทำงานจบ (return แล้ว) ตัวแปร bullets ควรจะสลายหายไปจากแรม! แต่เพราะมันถูกวิญญาณ Lambda ดูดไว้... มันจึงมีชีวิตรอดต่อไปได้เรื่อยๆ ครับ! (นี่แหละคือหัวใจของการทำ Factory Pattern ขั้นสูง!)

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** การที่ Lambda แอบดึงตัวแปรมาใช้แบบนี้ (Closure) มันจะทำให้กิน Memory (RAM) ของเครื่องหนักขึ้นไหมครับ?
**แนวคำตอบ:** **กินหนักขึ้นแน่นอนครับ! (Memory Leak Risk)**
เพราะอย่างที่บอกในข้อ Challenge: ปกติพอฟังก์ชันจบ ตัวแปรข้างในจะถูกถังขยะ (Garbage Collector - GC) เก็บกวาดทิ้งทันที 
แต่พอมี Lambda ไปจับมันไว้ (Capture)... GC จะไม่กล้ามาลบตัวแปรนี้ทิ้ง เพราะเห็นว่า "อ้าว ไอ้ Lambda นั่นมันยังถือสายอยู่นี่หว่า เผื่อมันจะใช้อีก" ทำให้ข้อมูลมันค้างเติ่งในแรมไปเรื่อยๆ จนกว่าตัว Lambda เองจะถูกทำลายครับ! สถาปนิกจึงต้องระวังการใช้ Closure ดึงตัวแปรหนักๆ (เช่น ก้อนดาต้าเบส 1GB) เข้าไปใน Lambda อย่างมากครับ!

**คำถาม 2:** โค้ดนี้ปลอดภัยไหมครับ?
`int total = 0;` 
`Task.Run(() => { total += 10; });` // รันใน Thread คู่ขนาน
`Task.Run(() => { total += 20; });`
**แนวคำตอบ:** **โคตรพินาศเลยครับ! (Race Condition)**
คุณกำลังให้ Lambda 2 ตัว (ที่ถูกรันพร้อมกันใน 2 จักรวาล/Thread) ทะลวงเข้ามาดึง/แก้ตัวแปร `total` ตัวเดียวกันแบบพร้อมๆ กันในเสี้ยววินาที! ข้อมูลตีกันพังพินาศแน่นอนครับ! (บั๊กชนิดนี้หาตัวจับยากที่สุดในโลก เพราะเวลารัน 100 รอบ ผลลัพธ์อาจจะเพี้ยนแค่รอบเดียว) 
ถ้าใช้ Closure ในการทำ Multi-Threading... ห้ามแก้ตัวแปรต้นทางเด็ดขาด (Read-only เท่านั้น) หรือถ้าต้องแก้ ต้องมีระบบล็อค (Lock/Mutex) คุ้มกันเสมอครับ!
:::
