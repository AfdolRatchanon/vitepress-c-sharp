# Delegates Basics (ตัวแทนรับมอบอำนาจ)

> 💡 **เป้าหมาย:** ทำลายกรอบความคิดเดิมที่ว่า "ตัวแปรต้องเก็บแต่ข้อมูล (int, string)" วันนี้เราจะมาสร้างตัวแปรชนิดใหม่ ที่สามารถ "เก็บฟังก์ชัน (Method)" เอาไว้ข้างในได้! นี่คือจุดเริ่มต้นของการส่งผ่านพฤติกรรม (Passing Behavior) ในโลกซอฟต์แวร์สมัยใหม่

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. Delegate คืออะไร?

**Delegate (เด-ลิ-เกต)** แปลว่า "ตัวแทน / ผู้รับมอบอำนาจ"
ในภาษา C (รุ่นพ่อ) มันถูกเรียกว่า **Function Pointer** (ตั๋วชี้เป้าไปหาฟังก์ชัน)

ลองคิดภาพตามนี้:
ปกติคุณมีตัวแปร `int x = 5;` แล้วคุณส่ง `x` ไปให้เพื่อน (เพื่อนได้รับตัวเลข 5)
แต่ถ้าคุณมีตัวแปร `MyDelegate d = ฟังก์ชันบวกเลข;` แล้วคุณส่ง `d` ไปให้เพื่อน... เพื่อนจะสามารถเรียกใช้ `d()` เพื่อบวกเลขได้ทันที! ทั้งๆ ที่เพื่อนไม่รู้เลยว่าโค้ดบวกเลขนั้นเขียนว่ายังไง!

---

### 2. ไวยากรณ์แห่งการสร้างใบมอบอำนาจ

การสร้าง Delegate มีกฎเหล็กคือ: **"ต้องระบุรูปร่าง (Signature) ของฟังก์ชันที่มันยอมรับ!"**
เช่น ถ้า Delegate บอกว่า "ฉันรับเฉพาะฟังก์ชันที่ไม่มี Return และต้องรับพารามิเตอร์เป็น string 1 ตัวนะ"... ถ้าคุณพยายามเอาฟังก์ชันที่ Return `int` มาใส่ มันจะดีดทิ้งทันที! (นี่คือ Type-Safety สุดโหดของ C#)

**ขั้นตอนการใช้งาน 3 สเต็ป:**
1. **ประกาศใบมอบอำนาจ:** วางกรอบว่าตัวแทนคนนี้รับรูปร่างฟังก์ชันแบบไหน (มักเขียนไว้นอกคลาส หรือระดับคลาส)
2. **สร้างตัวแปร (จ้างตัวแทน):** เสกตัวแปร แล้วเอาฟังก์ชันจริงยัดใส่มือมัน
3. **กดปุ่มสั่งการ (Invoke):** สั่งให้ตัวแทนลงมือทำงาน!

```csharp
// 1. ประกาศใบมอบอำนาจ (รับฟังก์ชัน void ที่รับ string 1 ตัว)
public delegate void PrintDelegate(string message);

class Program {
    // ฟังก์ชันจริงๆ ที่มีรูปร่างตรงตามใบสัญญาเป๊ะ (void และรับ string)
    static void SayHello(string msg) { Console.WriteLine("Hello " + msg); }

    static void Main() {
        // 2. จ้างตัวแทน แล้วเอาฟังก์ชันยัดใส่มือ (ไม่ต้องมีวงเล็บหลังฟังก์ชันนะ! เราไม่ได้เรียกใช้ เราแค่ชี้เป้า)
        PrintDelegate pointer = SayHello; 
        
        // 3. กดปุ่มสั่งการ!
        pointer("Arthur"); // ผลลัพธ์: Hello Arthur
    }
}
```

### 3. พลังแห่งสายชาร์จ (Multicast Delegate)

ความน่ากลัวของ Delegate ไม่ได้มีแค่นี้ครับ! คุณสามารถเอาฟังก์ชัน **"หลายๆ ตัว"** มาต่อคิวเข้าค่ายลูกเสือ ยัดใส่ Delegate ตัวเดียวกันได้ (ด้วยเครื่องหมาย `+=`)
พอคุณสั่งรัน Delegate 1 ครั้ง... มันจะไล่รันฟังก์ชันทุกตัวในคิวให้แบบเรียงลำดับทันที! (เรียกว่า Multicast)

```csharp
PrintDelegate pointer = SayHello; 
pointer += SayGoodbye; // ยัดฟังก์ชันที่ 2 เข้าไปต่อท้าย!
pointer("Arthur");     // มันจะรันทั้ง Hello Arthur และ Goodbye Arthur ติดกันเลย!
```
*นี่แหละครับคือกลไกเบื้องหลังของ "ปุ่มในเกม" เวลากดปุ่มทีเดียว มันทั้งยิงปืน, ทั้งเล่นเสียงปัง, และทั้งหักกระสุนในแมกกาซีนพร้อมกัน!*

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: สถาปัตยกรรมปลั๊กอินเครื่องคิดเลข (The Pluggable Calculator)

โปรแกรมที่ดี ไม่ควรเขียนคำสั่งบวก ลบ คูณ หาร ฝังไว้ใน Main... เราจะ "รับบริจาค" วิธีการคำนวณผ่านท่อ Delegate แทน! (ใครอยากให้คำนวณแบบไหน ก็โยนสูตรมา!)

::: code-group
```csharp [Program.cs]
using System;

namespace DelegateConcept
{
    // =======================================================
    // 1. ประกาศใบมอบอำนาจ (The Contract)
    // =======================================================
    // กฎ: รับ int 2 ตัว, คืนค่า int กลับมา
    public delegate int MathOperation(int a, int b);

    class Calculator
    {
        // 🌟 ไฮไลต์: เมธอดนี้รับ "ฟังก์ชัน" เข้ามาเป็นพารามิเตอร์! (สุดยอดความยืดหยุ่น)
        public void Execute(int x, int y, MathOperation op)
        {
            Console.WriteLine($"[ระบบ] กำลังคำนวณเลข {x} กับ {y}...");
            
            // เราไม่รู้เลยว่า op ข้างในมันบวก ลบ คูณ หรือหาร... เราแค่มีหน้าที่สั่งให้มันทำงาน (Invoke)
            int result = op(x, y); 
            
            Console.WriteLine($" -> ผลลัพธ์คือ: {result}");
        }
    }

    class Program
    {
        // =======================================================
        // ฟังก์ชันคณิตศาสตร์ต่างๆ (ที่มีรูปร่างตรงตาม MathOperation)
        // =======================================================
        static int Add(int a, int b) { return a + b; }
        static int Subtract(int a, int b) { return a - b; }
        
        static void Main(string[] args)
        {
            Console.WriteLine("=== เครื่องคิดเลขระบบปลั๊กอิน (Pluggable Calculator) ===\n");

            Calculator calc = new Calculator();

            Console.WriteLine(">> ทดสอบโยนสูตรบวกเลขเข้าไป:");
            // 2. โยนชื่อฟังก์ชัน (Add) เข้าไปตรงๆ ได้เลย! ไม่ต้องใส่ () นะ!
            calc.Execute(10, 5, Add); 

            Console.WriteLine("\n>> ทดสอบโยนสูตรลบเลขเข้าไป:");
            // โยนชื่อฟังก์ชัน (Subtract)
            calc.Execute(10, 5, Subtract); 

            // 💥 พังพินาศ!: calc.Execute(10, 5, "บวกเลข");
            // คอมไพเลอร์จะตบหน้า เพราะ "บวกเลข" เป็น string ไม่ใช่ฟังก์ชัน!
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: ตรวจจับสเปคฟังก์ชัน (Signature Matching)**
กำหนดให้มีการประกาศใบมอบอำนาจ:
`public delegate void TextProcessor(string text, int times);`

และมีฟังก์ชัน 4 ตัวดังนี้:
1. `static void RepeatText(string word, int count) { ... }`
2. `static void CountChars(int times, string text) { ... }`
3. `static string GetText(string t, int i) { return t; }`
4. `static void Print(string t) { ... }`

**คำถาม:** ฟังก์ชันข้อไหนบ้างที่สามารถ "ยัด" ใส่ Delegate `TextProcessor` ตัวนี้ได้สำเร็จ? อธิบายเหตุผลของข้อที่ใส่ไม่ได้ด้วย!

### **โจทย์ที่ 2: ปัญหาผลลัพธ์ของ Multicast (The Return Value Trap)**
เราเอาฟังก์ชันต่อคิวกันได้ด้วย `+=` สมมติว่ามี Delegate ที่มีค่า Return แบบนี้:
```csharp
delegate int MathOp();
static int Ret1() { return 1; }
static int Ret2() { return 2; }

MathOp pointer = Ret1;
pointer += Ret2;
int answer = pointer();
```
**คำถาม:** ค่าในตัวแปร `answer` สุดท้ายจะเป็นเลข 1 หรือเลข 2? เพราะอะไร?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **ผ่านแค่ข้อ 1. `RepeatText` ข้อเดียวครับ!**
  - ข้อ 2. ผิด เพราะลำดับพารามิเตอร์สลับกัน (int นำหน้า string ไม่ได้!)
  - ข้อ 3. ผิด เพราะมี Return เป็น `string` (ใบสัญญาบังคับให้เป็น `void`)
  - ข้อ 4. ผิด เพราะรับพารามิเตอร์แค่ 1 ตัว (ใบสัญญาบังคับ 2 ตัว)
  (เห็นความเข้มงวดของ Type-Safety ไหมครับ? นี่แหละที่ทำให้ C# ปลอดภัยจากบั๊ก!)
- **สำหรับโจทย์ 2:** **ตอบเลข 2 ครับ!**
  เวลาเราใช้ `+=` เพื่อรันหลายๆ ฟังก์ชันต่อเนื่องกัน... ถ้าฟังก์ชันนั้นมีการทิ้งของ (Return) กลับมา... **มันจะทิ้งของเก่าทั้งหมด แล้วเก็บเฉพาะชิ้นสุดท้ายของคิว** มาใส่ตัวแปรให้เราครับ! (ดังนั้นในโลกความจริง สถาปนิกมักจะใช้ Multicast Delegate เฉพาะกับฟังก์ชันที่เป็น `void` เท่านั้น เพื่อป้องกันการงุนงงของตัวเลข Return ครับ)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ระบบเซนเซอร์แจ้งเตือนภัย (The Alarm System)**

**ภารกิจของคุณ:** จงสร้างระบบกระจายข่าวสาร (Event Broadcasting) โดยใช้ Multicast Delegate!

1. ประกาศ `delegate void AlarmHandler(string dangerLevel);`
2. สร้างคลาส `Sensor`
   - มีตัวแปร (Field) ชื่อ `public AlarmHandler OnFireAlert;` (นี่คือกล่องเก็บคิวฟังก์ชัน)
   - มีเมธอด `public void TriggerFire()` เมื่อถูกเรียก ให้เช็คว่า `OnFireAlert` เป็น `null` ไหม? ถ้าไม่ ให้กดปุ่มเรียกใช้ตัวแปรนั้น พร้อมส่งคำว่า "ระดับ 5 อพยพด่วน!" ยัดใส่เข้าไป
3. ในคลาส `Program` สร้าง 3 ฟังก์ชัน `static` เตรียมไว้:
   - `CallFireStation(string lvl)` (ปริ้น: 🚒 โทรเรียกดับเพลิง! ภัยระดับ: lvl)
   - `TurnOnSprinklers(string lvl)` (ปริ้น: 💦 เปิดสปริงเกอร์น้ำฉุกเฉิน!)
   - `EvacuateBuilding(string lvl)` (ปริ้น: 🏃‍♂️ เสียงไซเรนดัง! อพยพพนักงาน!)
4. **การทดสอบใน Main:**
   - เสกออบเจกต์ `Sensor s = new Sensor();`
   - ทำการ "ต่อคิว (Subscribe)" ฟังก์ชันทั้ง 3 ตัว ยัดเข้าไปใน `s.OnFireAlert` ด้วยเครื่องหมาย `+=`
   - สั่ง `s.TriggerFire();` (กดปุ่มครั้งเดียว ฟังก์ชันทั้ง 3 ตัวต้องรันเรียงกันพรวดออกมาให้หมด!)

ถ้าคุณทำได้... ขอต้อนรับเข้าสู่โลกของ **Event-Driven Architecture** ครับ! นี่คือกลไกพื้นฐานที่ระบบ Windows ใช้ในการบอกโปรแกรมต่างๆ ว่า "มีคนคลิกเมาส์แล้วนะ!" (Click Event) ครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** คอนเซปต์การโยนฟังก์ชันใส่ตัวแปรแบบนี้... ในภาษาอย่าง JavaScript (JS) เค้าก็ทำกันเป็นเรื่องปกติแถมง่ายกว่านี้เยอะ ไม่เห็นต้องมานั่งประกาศ `public delegate void...` ให้วุ่นวายเลย ทำไม C# ต้องทำให้มันยากด้วยครับ?
**แนวคำตอบ:** **เพราะ "ความปลอดภัย" มันตีเป็นมูลค่าไม่ได้ครับ!**
ใน JS (ที่ไม่มี TypeScript) คุณสามารถโยนฟังก์ชันอะไรก็ได้มั่วๆ ยัดใส่พารามิเตอร์... ผลคือ พอมันรันตอนตี 3 แล้วพารามิเตอร์หน้าตาไม่ตรงกัน เซิร์ฟเวอร์ล่มบึ้มกระจาย! (Runtime Error)
แต่ใน C# ความจุกจิกเรื่องการประกาศ Delegate คือ "เกราะกำบังระดับเพชร" คอมไพเลอร์จะตรวจจับให้คุณตั้งแต่ตอนพิมพ์โค้ดเลยว่า "เห้ย ฟังก์ชันนี้มัน Return ค่าไม่ตรงกับที่ตกลงไว้นะ!" (Compile-Time Error) ทำให้โปรแกรมเมอร์มั่นใจ 100% ว่าถ้ากดรันผ่าน โค้ดจะไม่ระเบิดกลางคันแน่นอนครับ!

**คำถาม 2:** โค้ด Delegate ดูแปลกใหม่มากเลย ผมจำเป็นต้องใช้มันบ่อยแค่ไหนในการเขียน C# ยุคปัจจุบัน?
**แนวคำตอบ:** **ในโลก C# สมัยใหม่ เราแทบจะไม่พิมพ์คำว่า `delegate` กันแล้วครับ!**
อ้าว!? แล้วให้เรียนทำไม? 
คำตอบคือ: แนวคิดการยัดฟังก์ชันยังคงใช้อยู่ 100% (แถมใช้บ่อยมากด้วย) แต่ไมโครซอฟต์รู้ว่าการมานั่งประกาศ `delegate void...` ทุกครั้งมันน่ารำคาญ เขาเลยสร้าง "แม่พิมพ์ Delegate อเนกประสงค์" ไว้ให้คุณเรียกใช้ได้เลยทันที (เช่น Action, Func, Predicate) ซึ่งเรากำลังจะได้เรียนความว้าวของมันในหัวข้อถัดไปครับ! เตรียมตัวโบกมือลาคำว่า delegate แบบดั้งเดิมได้เลย!
:::
