# Projection with Select (การแปรรูปและการฉายภาพ)

> 💡 **เป้าหมาย:** สัมผัสพลังแห่งการ "แปลงกาย" ของ LINQ! `.Select()` ไม่ใช่การกรองคนออก แต่คือการ "หยิบจับ" หรือ "แปรรูป" ออบเจกต์หน้าตาเดิม ให้กลายเป็นออบเจกต์หน้าตาใหม่ทั้งหมด ภายในเสี้ยววินาที! นี่คือฟังก์ชันที่ใช้คู่กับ `.Where()` มากที่สุดราวกับปาท่องโก๋!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. โครงสร้างวิญญาณของ `.Select()`

ในฐานข้อมูล SQL คำสั่ง `SELECT Name FROM Users` หมายถึง: ฉันไม่เอาคอลัมน์อื่น เอามาแค่ชื่อ!
ใน C# คอนเซปต์นี้เรียกว่า **Projection (การฉายภาพเฉพาะส่วน)**

ฟังก์ชัน `.Select()` บังคับรับ Lambda สไตล์ "ผู้แปรรูป (Func<TSource, TResult>)"
- **ฝั่งซ้าย:** คือออบเจกต์ 1 ตัวในสภาพ "ก่อนแปรรูป"
- **ฝั่งขวา:** คือของ "หน้าตาใหม่" ที่คุณอยากจะเสกให้มันกลายเป็น!

```csharp
List<Employee> staff = ...;

// การเด็ดใบไม้ (Plucking): เปลี่ยนจาก "พนักงานทั้งก้อน" ให้กลายเป็นแค่ "string (ชื่อ)"
var justNames = staff.Select(e => e.Name); 
// ผลลัพธ์ที่ได้ จะกลายเป็นกล่องเก็บ string (IEnumerable<string>) ล้วนๆ!
```

---

### 2. ศิลปะแห่งการแปรรูป (Transformation Art)

`.Select()` ไม่ได้มีไว้แค่ "ดึง" ตัวแปรออกมาเท่านั้น แต่มันสามารถใช้เครื่องหมายคณิตศาสตร์หรือลอจิก เพื่อ "แปลงร่าง" ตัวแปรนั้นได้ทันที! 

```csharp
List<int> numbers = new List<int> { 1, 2, 3 };

// เสกให้กลายเป็นเลขกำลังสอง! (1->1, 2->4, 3->9)
var squares = numbers.Select(n => n * n);

// เสกให้กลายเป็นข้อความที่มีวงเล็บครอบ! (1->"[1]", 2->"[2]", 3->"[3]")
var stringTags = numbers.Select(n => $"[{n}]");
```
*เห็นไหมครับ! เราสามารถเสก `List<int>` ให้กลายพันธุ์เป็น `List<string>` ได้ดื้อๆ เลย! นี่คือความอลังการของสถาปัตยกรรมแบบ Functional Programming!*

---

### 3. การสร้างร่างจำแลง (Anonymous Types)

บ่อยครั้งที่เราไม่ได้อยากดึงมาแค่ `Name` ตัวเดียว แต่อยากได้ทั้ง `Name` และ `Salary` (ทิ้งรหัสพนักงาน ทิ้งอายุไป)
ถ้าเราเขียน `Select(e => e.Name, e.Salary)` โค้ดจะพังทันที! เพราะ Select คายของใหม่ได้แค่ "1 ก้อน" ต่อคนเท่านั้น!

**ทางแก้ระดับพระเจ้า:** เราจะเสก "คลาสไร้ชื่อ (Anonymous Type)" ครอบมันไว้ด้วยคีย์เวิร์ด `new { }` !

```csharp
// เสกออบเจกต์ลึกลับ ที่มีแค่ชื่อและเงินเดือน ขึ้นมาในเมมโมรี่!
var miniData = staff.Select(e => new { 
    EmpName = e.Name, 
    Money = e.Salary 
});
```
*นี่คือท่าไม้ตายที่บริษัทต่างๆ ใช้เพื่อ "ปั้น Data" เตรียมโยนส่งข้าม Network (ผ่าน JSON API) เพื่อลดขนาดข้อมูลให้เบาที่สุดครับ!*

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: โรงงานแปรรูปข้อมูลพนักงาน (Data Pipeline Transform)

ลองนำท่อกรอง `.Where()` มาสวมร่างต่อเข้ากับท่อแปรรูป `.Select()` ดูครับ! มันคือโรงงานนรกที่ประมวลผลข้อมูลไหลลื่นแบบไม่ต้องพึ่งตัวแปรพักข้อมูลใดๆ ทั้งสิ้น!

::: code-group
```csharp [Program.cs]
using System;
using System.Collections.Generic;
using System.Linq;

namespace LinqSelectConcept
{
    class Employee
    {
        public string FirstName { get; set; }
        public string LastName { get; set; }
        public string Department { get; set; }
        public decimal Salary { get; set; }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== โรงงานแปรรูปข้อมูล (Map & Reduce) ===\n");

            List<Employee> staff = new List<Employee>
            {
                new Employee { FirstName = "Arthur", LastName = "Pendragon", Department = "Sword", Salary = 50000 },
                new Employee { FirstName = "Merlin", LastName = "Ambrosius", Department = "Magic", Salary = 90000 },
                new Employee { FirstName = "Lancelot", LastName = "Du Lac",  Department = "Sword", Salary = 50000 }
            };

            // ==========================================================
            // เคส 1: การแปรรูป String ธรรมดา (Full Name Generation)
            // ==========================================================
            Console.WriteLine(">> 1. สร้างป้ายชื่อพนักงาน (รวมชื่อ-สกุล ตัวพิมพ์ใหญ่):");

            // ดึงพนักงานทั้งก้อน -> แปลงร่างเป็น string (ชื่อนามสกุลตัวใหญ่)
            var nameTags = staff.Select(e => $"{e.FirstName} {e.LastName}".ToUpper());
            
            foreach (var tag in nameTags) Console.WriteLine($"- ป้ายชื่อ: {tag}");
            Console.WriteLine();

            // ==========================================================
            // เคส 2: Combo! กรอกของเสียออก ก่อนส่งเข้าเครื่องแปรรูป
            // ==========================================================
            Console.WriteLine(">> 2. รายชื่อคนแผนก Sword (เอาเฉพาะชื่อแรก):");

            // 🌟 ท่อประปา 2 จังหวะ:
            // 1. Where: ร่อนเอาเฉพาะคนแผนกดาบ (เหลือ 2 คนเต็มร่าง)
            // 2. Select: ดึงข้อมูล 2 คนที่รอดชีวิต มากระชากเอาเฉพาะ FirstName 
            var swordGuys = staff.Where(e => e.Department == "Sword")
                                 .Select(e => e.FirstName);

            foreach (var name in swordGuys) Console.WriteLine($"- อัศวิน: {name}");
            Console.WriteLine();

            // ==========================================================
            // เคส 3: ท่าไม้ตาย! เสกร่างจำแลง (Anonymous Type Projection)
            // ==========================================================
            Console.WriteLine(">> 3. ปั้น Data สำหรับส่งรายงานลับ (ดึงแค่ ชื่อเต็ม + เงินเดือนเพิ่ม 10%):");

            var secretReport = staff.Select(e => new 
            { 
                FullName = $"{e.FirstName} {e.LastName}", 
                BonusSalary = e.Salary * 1.10m // แอบบวกโบนัส 10% ให้เนียนๆ ในร่างจำแลงเลย!
            });

            foreach (var item in secretReport) 
            {
                // สามารถเข้าถึง Property จำแลงที่เราตั้งชื่อไว้ได้เลย!
                Console.WriteLine($"- ชื่อ: {item.FullName} | จะได้รับเงิน: {item.BonusSalary:N0}");
            }
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: หน้าที่ของ Where ปะทะ Select**
เพื่อป้องกันความสับสนเวลาเขียนโค้ด LINQ...
สมมติคุณมีกล่องเก็บข้อมูลดิบ `List<int> raw = { 10, 20, 30 }`
ถ้าคุณสั่ง `.Where(x => ...)` ผลลัพธ์ที่ได้ ขนาดกล่อง (จำนวนตัวเลขข้างใน) จะมีสิทธิ์เพิ่มขึ้น ลดลง หรือเท่าเดิมครับ?
แล้วถ้าคุณสั่ง `.Select(x => ...)` ล่ะ ผลลัพธ์ที่ได้ ขนาดกล่อง (จำนวนตัวเลขข้างใน) จะเป็นอย่างไร? 

### **โจทย์ที่ 2: สลับท่อประปา**
พิจารณาโค้ดท่อประปา 2 แบบนี้:
- แบบ A: `staff.Where(e => e.Age > 30).Select(e => e.Name)`
- แบบ B: `staff.Select(e => e.Name).Where(e => e.Age > 30)`
**คำถาม:** แบบ A กับ แบบ B ให้ผลลัพธ์เหมือนกันหรือไม่? ถ้าต่างกัน โค้ดไหนที่ Compile พังเละเทะ!

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  - **Where (กรองคนออก):** ขนาดกล่องจะ **"ลดลง หรือ เท่าเดิม"** เท่านั้นครับ! (ไม่มีทางเพิ่มขึ้น)
  - **Select (แปรรูปคน):** ขนาดกล่องจะ **"เท่าเดิมเสมอ 100%"** ครับ! (เข้า 3 ออก 3 แค่เปลี่ยนหน้าตาสิ่งของข้างใน) 
- **สำหรับโจทย์ 2:** 
  **แบบ B คอมไพล์พังเละเทะครับ แดงเถือก!**
  เหตุผลคือ ท่อประปามันไหลจากซ้ายไปขวา... 
  ในแบบ B ท่อแรกคือ `Select` มันได้ทำการกระชากเอามาแค่ "กล่องเก็บรายชื่อ (String)" แล้ว พอไหลไปเข้าท่อ 2 (`Where`) มันดันพยายามเรียก `e.Age` ซึ่งข้อความ string มันไม่มีอายุเว้ย! มันก็เลยพังครับ!
  (กฎเหล็ก: **ต้อง Where กรองตัวตนให้เสร็จก่อน แล้วค่อยจับ Select ลอกคราบเสมอครับ!**)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: เครื่องประมวลผลตะกร้าสินค้า (The Cart Processor)**

**ภารกิจของคุณ:** จงแปลงข้อมูลตะกร้าสินค้า ให้กลายเป็นบิลใบเสร็จที่พร้อมพิมพ์!

**ข้อมูลตั้งต้น:** คุณมีคลาส `CartItem` ที่มี 3 ตัวแปร:
- `public string ProductName { get; set; }`
- `public decimal Price { get; set; }` (ราคาสินค้าต่อชิ้น)
- `public int Quantity { get; set; }` (จำนวนที่ซื้อ)

สมมติใน `Main` มีข้อมูล:
```csharp
List<CartItem> cart = new List<CartItem> {
    new CartItem { ProductName = "เมาส์", Price = 500, Quantity = 2 },
    new CartItem { ProductName = "คีย์บอร์ด", Price = 1500, Quantity = 1 },
    new CartItem { ProductName = "แผ่นรองเมาส์", Price = 100, Quantity = 0 } // ของแถมแต่ไม่ได้เอา
};
```

**ภารกิจ (เขียนเป็น LINQ ห้ามใช้ foreach เดี็ดขาด):**
1. จงใช้ `.Where()` กรองเอาเฉพาะสินค้าที่ จำนวน (Quantity) มากกว่า 0 เท่านั้น! (ทิ้งแผ่นรองเมาส์ไป)
2. ต่อท่อด้วย `.Select()` สร้างคลาสไร้ชื่อ `new { }` เพื่อแปรรูปบิลใบเสร็จ โดยให้มี Property หน้าตาใหม่ 2 ตัวคือ:
   - `ItemName` -> เกิดจากการเอาชื่อสินค้า มาบวกกับคำว่า `(x จำนวนชิ้น)` (เช่น "เมาส์ (x 2)")
   - `TotalPrice` -> เกิดจากการเอา `Price * Quantity`
3. วนลูป foreach นำผลลัพธ์จากท่อไปปริ้นโชว์ให้สวยงาม!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** คลาสไร้ชื่อ (Anonymous Type `new { }`) ที่เราสร้างขึ้นใน Select มันมีข้อเสียอะไรบ้างครับ? ทำไมเราไม่ใช้มันรับส่งข้อมูลระหว่างหน้าจอไปเลย?
**แนวคำตอบ:** **ข้อเสียร้ายแรงของมันคือ "มันส่งข้ามฟังก์ชันไม่ได้ครับ!" (Local Scope Only)**
คลาสไร้ชื่อมันเป็นความลับที่มีชีวิตอยู่แค่ภายในบล็อกปีกกาที่มันเกิดมาเท่านั้น (Method Level) ถ้าคุณพยายามจะเขียนฟังก์ชัน `public ??? GetReport()` คุณจะหาชนิดข้อมูลที่จะมารับมันเพื่อ Return กลับไปไม่ได้เลยครับ! (เพราะมันไม่มีชื่อให้เรียก)
ดังนั้น สถาปนิกจึงใช้มันเพื่อ "ปั้น Data ใช้งานเฉพาะหน้า" หรือ "เตรียมโยนทิ้งเป็น JSON สดๆ" เท่านั้นครับ! ถ้าจะต้องส่งข้อมูลก้อนนี้ไปให้คลาสอื่นทำงานต่อ ยังไงคุณก็ต้องสร้าง `class ReportItem { }` แบบปกติมารองรับมันอยู่ดีครับ (เรียกว่า DTO - Data Transfer Object)

**คำถาม 2:** ถ้าผมมี `List<string> sentences` เก็บประโยคยาวๆ เช่น `{"hello world", "good morning"}` แล้วผมอยากจะแตกทุกคำให้กลายเป็นลิสต์คำศัพท์ `List<string> words = {"hello", "world", "good", "morning"}` ผมจะใช้ `Select` ได้ไหมครับ?
**แนวคำตอบ:** **ใช้ `Select` ธรรมดาไม่ได้ครับ! (จะได้เป็น Array ซ้อน Array แทน!)**
ความท้าทายนี้ถือเป็นระดับสูงครับ! เพราะประโยค 1 ประโยค เวลาโดน `.Split(' ')` มันจะระเบิดตัวเองออกเป็น Array คำศัพท์หลายๆ ตัว (1:Many)... ถ้าใช้ Select คุณจะได้กล่องที่มีกล่องเล็กซ้อนข้างในอีกที (`IEnumerable<string[]>`)
ทางแก้ของ LINQ คือมีฟังก์ชันพี่ชายชื่อ **`.SelectMany()`** ครับ! (หน้าที่ของมันคือ การทุบกล่องเล็กให้แตกกระจาย แล้วเอาของทุกชิ้นในนั้นมาเทรวมกันในกล่องใหญ่ใบเดียว!) ซึ่งมันคือหัวใจสำคัญของการดึงฐานข้อมูลแบบ 1-to-Many ของ Entity Framework ครับ!
:::
