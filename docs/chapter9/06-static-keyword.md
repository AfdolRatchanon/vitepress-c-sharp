# Static Keyword (สิ่งดำรงอยู่ร่วมกัน)

> 💡 **เป้าหมาย:** ไขปริศนาของคำว่า `static` ที่เราพิมพ์ติดมือมาตลอดตั้งแต่บทที่ 1! เรียนรู้ความแตกต่างระหว่าง "สมองของตัวละคร (Instance Members)" และ "เมฆหมอกส่วนกลาง (Static Members)" เพื่อสร้างตัวแปรหรือฟังก์ชันที่คนทั้งโลกแชร์ใช้ร่วมกันโดยไม่ต้องเสียเวลาเรียกผู้รับเหมามาสร้างบ้าน (`new`)

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ความแตกต่างระหว่าง Instance และ Static

จินตนาการว่า Class `Student` คือพิมพ์เขียวสำหรับนักเรียนในห้อง
- **`public string Name;` (ไม่มี static - Instance):** คือ "ป้ายชื่อ" ที่ติดอยู่บนเสื้อของนักเรียนแต่ละคน ป้ายใครป้ายมัน ถ้าเรามีนักเรียน 50 คน (`new Student()` 50 ครั้ง) เราก็จะมีป้ายชื่อ 50 ป้ายที่ไม่เกี่ยวข้องกัน (เปลี่ยนชื่อคนนี้ ไม่กระทบคนอื่น)
- **`public static string SchoolName;` (มี static):** คือ "ป้ายหน้าประตูโรงเรียน" ป้ายนี้มีอยู่เพียง "ป้ายเดียวในจักรวาล!" ไม่ว่าคุณจะสร้างนักเรียนกี่ร้อยคน ทุกคนจะมองขึ้นไปเห็นป้ายโรงเรียนป้ายเดียวกันเป๊ะ... และถ้ามีใครอุตริไปเปลี่ยนชื่อโรงเรียนบนป้าย นักเรียนทุกคนก็จะได้รับผลกระทบนั้นพร้อมกันทันที!

---

### 2. กฎของตัวแปรและฟังก์ชัน Static

1. **ไม่ต้องการคำสั่ง `new`:** สิ่งที่เป็น `static` จะเกิดมาพร้อมกับโปรแกรม (สิงอยู่ในพื้นที่พิเศษที่เรียกว่า High-Frequency Heap) คุณสามารถเรียกใช้มันได้ทันทีผ่าน "ชื่อคลาส" โดยไม่ต้องสร้างบ้าน!
   - ผิด: `Student s1 = new Student(); s1.SchoolName = "ABC";` (ออบเจกต์ห้ามยุ่งกับป้ายหน้าโรงเรียน)
   - ถูก: `Student.SchoolName = "ABC";` (เรียกตรงๆ ผ่านคลาสได้เลย)
2. **`static` ห้ามแตะต้องของที่ไม่มี `static` เด็ดขาด:** (กฎข้อนี้ปราบเซียนมาก!)
   - สมมติคุณมีเมธอด `public static void Shout()` มันจะ "ไม่สามารถ" สั่งปริ้นตัวแปร `Name` ของนักเรียนได้! 
   - เหตุผล: คอมพิวเตอร์จะงงว่า "คุณกำลังให้ตะโกนชื่อใครล่ะ? นักเรียนคนที่ 1? คนที่ 2? หรือคนที่ 50?" (เพราะตัวเมธอดเป็นส่วนกลาง ลอยอยู่เหนือหัวทุกคน มันไม่มีสมองส่วนตัว `this` ให้ใช้นั่นเอง)
   - *(นี่คือสาเหตุที่สมัยเรียนบทที่ 5... ถ้าคุณสร้างฟังก์ชันข้างนอก Main คุณต้องใส่คำว่า static เสมอ เพราะ Main มันเป็น static! สิ่งศักดิ์สิทธิ์ต้องเรียกสิ่งศักดิ์สิทธิ์ระดับเดียวกันเท่านั้น!)*

---

### 3. เมื่อไหร่ควรใช้ `static` (Architectural Decision)

วิศวกรจะเลือกใช้ `static` ใน 2 สถานการณ์หลัก:
1. **เก็บตัวแปรนับจำนวน (Global Counters):** เช่น นับว่ามีมอนสเตอร์ถูกสร้างขึ้นมาในเกมกี่ตัวแล้ว? 
2. **กล่องเครื่องมืออเนกประสงค์ (Utility/Helper Classes):** เครื่องคิดเลขที่ไม่มี State เช่น `Math.Pow()`, `Math.Round()`, `Console.WriteLine()` เราไม่เคยต้องใช้คำสั่ง `new Math()` เลยใช่ไหมครับ? นั่นแหละพลังของ `static`!

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: ระบบนับประชากรโลก (The Population Counter)

นี่คือเทคนิคระดับสถาปัตยกรรม (Architectural Pattern) ในการซ่อนตัวนับจำนวนไว้ใน Constructor! 

::: code-group
```csharp [Program.cs]
using System;

namespace StaticConcept
{
    class Player
    {
        // -----------------------------------------------------------
        // INSTANCE DATA (ป้ายชื่อติดเสื้อ ของใครของมัน)
        // -----------------------------------------------------------
        public string Name { get; set; }

        // -----------------------------------------------------------
        // STATIC DATA (กระดานส่วนกลาง นับจำนวนคนเกิด)
        // -----------------------------------------------------------
        // คนนอกอ่านได้ ห้ามแก้กวนๆ (private set)
        public static int TotalPlayers { get; private set; } = 0; 

        // พิธีกรรมแรกเกิด
        public Player(string name)
        {
            this.Name = name;
            
            // 🌟 เวทมนตร์: ทุกครั้งที่มีคนเรียก new (มีผู้เล่นเกิดใหม่) 
            // ให้ไปสะกิดตัวแปรส่วนกลาง (TotalPlayers) ให้บวก 1 ทันที!
            TotalPlayers++; 
            
            Console.WriteLine($"[เกิด] ผู้เล่น {Name} ถือกำเนิดขึ้นแล้ว!");
        }

        // -----------------------------------------------------------
        // STATIC METHOD (โทรโข่งประกาศส่วนกลาง)
        // -----------------------------------------------------------
        public static void ShowServerStatus()
        {
            // ⚠️ เมธอดนี้ ไม่มีสิทธิ์เรียกตัวแปร Name มาใช้เด็ดขาด!
            Console.WriteLine($"\n🌍 [ประกาศจากเซิร์ฟเวอร์] จำนวนผู้เล่นออนไลน์ทั้งหมด: {TotalPlayers} คน");
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== จำลองเซิร์ฟเวอร์เกม Multiplayer ===\n");

            // สามารถเรียกใช้ Static Method ได้เลย ทั้งๆ ที่ยังไม่เคยสั่ง new Player!
            Player.ShowServerStatus(); // จะปริ้นว่า 0 คน

            // ปล่อยผู้เล่นเกิด
            Player p1 = new Player("Arthur");
            Player p2 = new Player("Ginevra");
            Player p3 = new Player("Lancelot");

            // อัปเดตสถานะเซิร์ฟเวอร์
            Player.ShowServerStatus(); // จะปริ้นว่า 3 คน (เพราะ Constructor ทำงานไป 3 รอบ!)

            // 💥 พังพินาศ!: Console.WriteLine(p1.TotalPlayers);
            // ^ ออบเจกต์ p1 ไม่มีสิทธิ์ดึงตัวแปร static มาใช้โดยตรง ต้องเรียกผ่านคลาส Player.TotalPlayers เท่านั้น!
        }
    }
}
```
:::

### ตัวอย่างที่ 2: สุดยอดคลาสเครื่องมืออเนกประสงค์ (Utility Class)

ในจักรวาล C# ถ้าคุณตั้งใจจะให้คลาสนี้มีแต่ฟังก์ชันคำนวณ (เหมือนคลาส `Math`) และ "ห้ามไม่ให้ใครหน้าไหนมาสั่ง `new` มันเด็ดขาด"... คุณสามารถใส่คีย์เวิร์ด `static` ลงไปที่ **"ระดับคลาส"** ได้เลยครับ!

::: code-group
```csharp [Program.cs]
using System;

namespace StaticUtility
{
    // 🌟 ใส่ static หน้า class!
    // เป็นการประทับตราว่า: คลาสนี้มีไว้เรียกใช้ตรงๆ เท่านั้น ห้ามใครเอาไป new สร้างเป็นออบเจกต์!
    static class StringHelper 
    {
        // ภายใน static class "ทุกอย่าง" ต้องเป็น static ตามไปด้วย
        
        public static string Shout(string text)
        {
            return text.ToUpper() + "!!!";
        }

        public static string Whisper(string text)
        {
            return text.ToLower() + "...";
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ทดสอบเครื่องมือจัดคําอเนกประสงค์ ===");

            // 💥 พังพินาศ!: StringHelper helper = new StringHelper();
            // คอมไพเลอร์จะตบหน้าและบอกว่า "Cannot create an instance of the static class"

            // การใช้งานที่ถูกต้อง: เรียกตรงๆ ผ่านชื่อคลาสเลย! ประหยัดเวลา ประหยัดแรม (ไม่ต้องจอง Heap)
            string loud = StringHelper.Shout("help me");
            Console.WriteLine(loud); // HELP ME!!!

            string quiet = StringHelper.Whisper("DON'T WAKE THE DRAGON");
            Console.WriteLine(quiet); // don't wake the dragon...
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: เครื่องมือแปลงหน่วย (The Converter)**
จงสร้าง `static class TemperatureConverter` ที่ห้ามคน new
และให้มี `static` Method 1 ตัวชื่อ `CelsiusToFahrenheit(double c)` 
ซึ่งรับค่าเซลเซียส และคืนค่าฟาเรนไฮต์กลับไป (สูตรคือ: `(c * 9/5) + 32`) 
ไม่ต้องเขียน Main ครับ ให้เขียนแค่ตัวพิมพ์เขียวคลาสพอ!

### **โจทย์ที่ 2: งูกินหางแห่งความตาย (The Static Trap)**
พิจารณาคลาสหายนะนี้:
```csharp
class Robot {
    public string Name;
    public static void Speak() {
        Console.WriteLine($"My name is {Name}");
    }
}
```
**คำถาม:** ถ้ากดรัน คอมไพเลอร์จะด่าคุณแบบเจ็บแสบที่สุดในบรรทัด `Console.WriteLine` ว่าอะไร? และอธิบายว่าทำไมถึงด่าแบบนั้น?

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  ```csharp
  static class TemperatureConverter {
      public static double CelsiusToFahrenheit(double c) {
          return (c * 9.0 / 5.0) + 32; // อย่าลืมเติม .0 ไม่งั้นมันหารแบบ int นะ!
      }
  }
  ```
- **สำหรับโจทย์ 2:** คอมไพเลอร์จะด่าว่า **"An object reference is required for the non-static field, method, or property 'Robot.Name'"** 
  ความหมายคือ: ฟังก์ชัน `Speak` ของคุณมันลอยอยู่บนฟ้า (static) แต่มันดันทะลึ่งจะก้มลงมาหยิบตัวแปร `Name` ที่อยู่ในสมองของออบเจกต์ (Instance) มาใช้... คอมพิวเตอร์มันไม่รู้ว่า "ชื่อ" ที่คุณอยากจะให้ปริ้นน่ะ มันชื่อของหุ่นยนต์ตัวไหน? 
  ถ้าอยากแก้ คุณต้องปลดคำว่า `static` ออกจาก `Speak()` ให้มันกลายเป็นความสามารถส่วนตัวของหุ่นยนต์แต่ละตัวครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: ธนาคารโลก (The Global Bank Rate)**

**ภารกิจของคุณ:** จงสร้างระบบคำนวณดอกเบี้ยแบบมีดอกเบี้ยกลาง!
1. **คลาส `SavingsAccount`**
   - มี Field สถานะส่วนตัว `public string Owner` และ `public double Balance`
   - มี Field ส่วนกลางระดับโลก `public static double InterestRate = 0.05` (ดอกเบี้ย 5%)
   - มี Constructor ให้รับชื่อและเงินต้นมาตั้งค่า
2. **Method พฤติกรรม:**
   - เมธอดส่วนตัว `public void AddInterest()`: ให้ดึงตัวเลข `InterestRate` ส่วนกลางมาคูณกับ `Balance` ของตัวเองเพื่อเพิ่มเงินให้ตัวเอง! แล้วปริ้นว่า "อัปเดตยอดเงินของ [Owner] กลายเป็น [Balance]"
3. **การทดสอบใน Main:**
   - เสกบัญชี "John" มีเงิน 1000 และ "Alice" มีเงิน 2000
   - สั่ง `John.AddInterest()` (ยอดควรได้ 1050)
   - สั่ง `Alice.AddInterest()` (ยอดควรได้ 2100)
   - **จุดไคลแมกซ์:** จู่ๆ ธนาคารกลางประกาศลดดอกเบี้ย! ให้คุณแก้ค่าผ่านคลาส `SavingsAccount.InterestRate = 0.01;` (เหลือ 1%) ทันทีใน Main!
   - สั่ง `John.AddInterest()` ซ้ำอีกรอบ! ยอดมันจะต้องถูกคูณด้วย 1% เท่านั้น (คือได้เพิ่ม 10.5 บาท กลายเป็น 1060.5)!

นี่คือความน่าขนลุกของคำว่า `static` ครับ... เพียงแค่คุณเปลี่ยนตัวแปรนี้เพียง 1 ครั้งในจักรวาล ออบเจกต์ทุกตัวที่เหลือบนโลกจะได้รับผลกระทบนั้นอย่างหลีกเลี่ยงไม่ได้ทันที!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในบทที่ผ่านมาตอนเราเขียนโค้ดเริ่มต้นของ C# (เช่น .NET Framework รุ่นเก่า) เรามักจะเห็นหน้าตาแบบนี้ `class Program { static void Main() { ... } }` ทำไมฟังก์ชัน Main ที่เป็นจุดสตาร์ทของระบบ ถึงต้องถูกบังคับให้เป็น `static` อย่างหลีกเลี่ยงไม่ได้ด้วยครับ?
**แนวคำตอบ:** เป็นคำถามระดับปรมาจารย์ครับ!
คิดง่ายๆ นะครับ... ตอนที่คุณดับเบิลคลิกเปิดโปรแกรม (exe) โลกคอมพิวเตอร์ตอนนั้นมันยังไม่มีอะไรเลย ว่างเปล่า! ยังไม่มีหุ่นยนต์ผู้รับเหมาไปสั่ง `new Program()` สร้างบ้านบน Heap เลยสักหลัง!
ดังนั้น ถ้า Main ไม่ใช่ `static`... คอมพิวเตอร์ก็ไม่มีปัญญาเรียกฟังก์ชันนี้ขึ้นมาทำงานได้เลยครับ! 
ไมโครซอฟต์จึงต้องประกาศให้ `Main` เป็น `static` ไว้ เพื่อให้ระบบปฏิบัติการ Windows สามารถ "กระชากมันลงมาจากฟ้า แล้วสั่งทำงานได้ทันที โดยไม่ต้องรอให้ใครมาสร้าง Object ให้มัน" ครับผม! (และนี่คือที่มาของความยิ่งใหญ่ของ Static Method!)
:::
