# Fields & Properties (ด่านตรวจคนเข้าเมือง)

> 💡 **เป้าหมาย:** สัมผัสความงามของการออกแบบภาษา C#! เรียนรู้นวัตกรรมที่ชื่อว่า **Properties (`get` / `set`)** ซึ่งเปรียบเสมือนด่านตรวจคนเข้าเมืองสุดอัจฉริยะ ที่ช่วยรวบยอดการเขียน Getter/Setter Method แบบเดิมๆ ให้กลายเป็นความสง่างามบรรทัดเดียว (Elegance & Conciseness)

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ปัญหาของ Getter / Setter ดั้งเดิม (The Java Way)

จากบทที่แล้ว เราเรียนรู้ว่าต้องตั้ง Field เป็น `private` แล้วสร้าง Method มาคุ้มกันมัน:
```csharp
private int _age;

public int GetAge() { return _age; }           // เมธอดสำหรับขอดูค่า
public void SetAge(int value) { _age = value; } // เมธอดสำหรับอัปเดตค่า
```
ถ้าคุณมีตัวแปร 10 ตัว คุณต้องมานั่งเขียน Method จับคู่แบบนี้ 20 ตัว! โค้ดของคุณจะบวมเป่ง (Boilerplate) และเวลาเรียกใช้งานใน Main ก็ดูลำบาก: `player.SetAge( player.GetAge() + 1 );` (ดูไม่เป็นธรรมชาติเอาซะเลย)

---

### 2. กำเนิด Properties (The C# Way)

ไมโครซอฟต์ปิ๊งไอเดีย: "ทำไมเราไม่ทำให้มันเรียกใช้ง่ายเหมือนตัวแปร `public` (เช่น `player.Age = 10;`) แต่ข้างหลังบ้านมันแอบวิ่งไปเข้าด่านตรวจแบบ Method เล่า?"
และนั่นคือที่มาของ **`Properties`** ครับ!

**ไวยากรณ์ (Syntax) ฉบับเต็ม (Full Property):**
ใช้ควบคู่กับตัวแปรเบื้องหลัง (Backing Field)
```csharp
private int _age; // ตัวแปรเบื้องหลัง (Backing Field) เป็นความลับ

public int Age    // Property (สังเกต: ใช้ตัวพิมพ์ใหญ่ ไม่มีวงเล็บ!)
{
    get { return _age; } // ด่านขาออก: เมื่อมีคนขอดูค่า (เช่น Console.Write(p.Age))
    set 
    { 
        // ด่านขาเข้า: เมื่อมีคนส่งค่ามา (เช่น p.Age = -5)
        // C# จะเสกตัวแปรลับชื่อ "value" มาเก็บเลข -5 ไว้ให้เราตรวจ!
        if (value >= 0) { _age = value; } 
    } 
}
```

### 3. Auto-Implemented Properties (ความขี้เกียจที่งดงาม)

ในหลายๆ ครั้ง เราแค่อยากเปิดเผยข้อมูลเฉยๆ ไม่ได้มีด่านตรวจ `if` อะไรซับซ้อน (เช่น รหัสพนักงาน ยังไงก็เป็นอะไรก็ได้) 
C# อนุญาตให้คุณ "ย่อรูป" Property ได้เหลือแค่บรรทัดเดียว!

```csharp
// คอมไพเลอร์จะแอบไปจองที่ดิน private สร้าง backing field ให้เองหลังบ้านแบบอัตโนมัติ!
public string Name { get; set; } 
```

**เวทมนตร์ขั้นกว่า (Read-Only Property):**
ถ้าเราอยากให้คนอื่น "อ่านได้อย่างเดียว แต่ห้ามแก้" (เหมือนบัตรประชาชน) เราสามารถใส่ `private` ล็อกเฉพาะขาเข้า (`set`) ได้!
```csharp
public int ID { get; private set; } // คนนอกอ่านได้ (get) แต่ถ้าใครพยายามเขียนทับ (set) คอมไพเลอร์จะตบมือหัก!
```

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: ระบบจัดการโปรไฟล์พนักงาน (The Enterprise Profile)

ตัวอย่างนี้จะรวมการใช้งาน Properties ทุกรูปแบบ (ทั้งแบบเต็มที่มี Validate, แบบ Auto และแบบ Read-only) ให้อยู่ในคลาสเดียวกัน

::: code-group
```csharp [Program.cs]
using System;

namespace PropertiesConcept
{
    class Employee
    {
        // -----------------------------------------------------------
        // 1. Auto-Implemented Property (สั้น ง่าย สบาย)
        // -----------------------------------------------------------
        // คนนอกสามารถอ่านและเขียนทับได้อิสระ
        public string Name { get; set; }

        // -----------------------------------------------------------
        // 2. Auto-Property แบบ Read-Only (ล็อกขาเข้า)
        // -----------------------------------------------------------
        // ปกติเราจะให้ค่าตั้งต้นไว้เลย คนนอกดูได้ แต่ห้ามแก้!
        public string Department { get; private set; } = "IT Support";

        // -----------------------------------------------------------
        // 3. Full Property (ด่านตรวจคนเข้าเมืองเข้มงวด)
        // -----------------------------------------------------------
        private decimal _salary; // Backing Field สำหรับเก็บค่าจริง

        public decimal Salary
        {
            get 
            { 
                return _salary; // คายเงินเดือนให้เจ้านายดู
            }
            set 
            { 
                // ด่านตรวจจับ (Validation)
                // คีย์เวิร์ด `value` คือคำสงวนของ C# ที่จะดักจับของที่คนส่งมาใส่กล่อง
                if (value < 15000)
                {
                    Console.ForegroundColor = ConsoleColor.Red;
                    Console.WriteLine("🚨 [HR System] ผิดกฎหมายแรงงาน! เงินเดือนต้องไม่ต่ำกว่า 15,000");
                    Console.ResetColor();
                    // ถ้าผิดกฎ ก็ไม่เอา value ไปยัดใส่ _salary 
                }
                else
                {
                    _salary = value; // ตรวจผ่าน ก็เอาของเข้าโกดังส่วนตัว
                    Console.WriteLine($"✅ อัปเดตเงินเดือนเป็น: {_salary} บาท");
                }
            }
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบฐานข้อมูลพนักงาน (HR System) ===\n");

            Employee emp1 = new Employee();

            // 1. ใช้งาน Property เหมือนตัวแปรปกติเลย! (ใช้ง่ายมาก)
            emp1.Name = "Arthur"; 
            Console.WriteLine($"ชื่อพนักงาน: {emp1.Name}");

            // 2. ลองแก้ Property แบบ Read-Only ดูสิ!
            // emp1.Department = "Marketing"; 
            // ^ ถ้าเอาคอมเมนต์ออก คอมไพเลอร์จะ Error ทันทีว่า "set accessor is inaccessible"

            Console.WriteLine($"แผนก: {emp1.Department} (ห้ามย้ายแผนก)\n");

            // 3. ปฏิบัติการแก้เงินเดือน (จะไปปลุกด่านตรวจ set ให้ทำงาน)
            Console.WriteLine(">> เจ้านายพยายามลดเงินเดือนเหลือ 9,000...");
            emp1.Salary = 9000; // มันจะไปเข้าด่านตรวจ value < 15000 แล้วโดนเด้งออก!

            Console.WriteLine("\n>> เจ้านายขึ้นเงินเดือนให้เป็น 50,000...");
            emp1.Salary = 50000; // ผ่านด่านฉลุย!

            // การขอดูเงินเดือน (จะไปปลุกด่านตรวจ get ให้ทำงาน)
            Console.WriteLine($"เงินเดือนปัจจุบัน: {emp1.Salary}");
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: วิเคราะห์โค้ดพัง (The Property Trap)**
มีโปรแกรมเมอร์ฝึกงานคนนึง หวังดีอยากจะป้องกันไม่ให้เลือดเกมเมอร์ติดลบ เขาเลยเขียนโค้ดนี้:
```csharp
public int HP 
{
    get { return HP; }
    set { if (value >= 0) HP = value; }
}
```
**คำถาม:** โค้ดนี้ดูเหมือนจะถูก... แต่ถ้ากดรัน โปรแกรมจะแครช (Crash) ดับสลายหายไปในอากาศด้วยข้อผิดพลาด "StackOverflowException"! คุณคิดว่าเขาทำพลาดเรื่องอะไร? 

### **โจทย์ที่ 2: รื้อถอนตึกเก่า (The Refactoring)**
พิจารณาโค้ดดั้งเดิมแบบ Procedural นี้:
```csharp
private string _username;
public void SetUsername(string name) { _username = name; }
public string GetUsername() { return _username; }
```
**ภารกิจ:** จงยุบรวม 3 บรรทัดนี้ ให้กลายเป็น "Auto-Implemented Property" ของ C# เพียงบรรทัดเดียว!

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** นี่คือความผิดพลาดระดับจักรวาลที่เด็กจบใหม่เจอกันทุกคนเรียกว่า "Infinite Recursion (งูกินหางตัวเอง)" ครับ!
  ในบล็อก `set` เขาเขียนว่า `HP = value;`... อ้าว! `HP` ก็คือตัว Property ของมันเอง! ดังนั้นการเอาค่าไปใส่ตัวมันเอง มันก็จะวิ่งกลับไปปลุกด่าน `set` ของตัวมันเองให้ทำงานซ้ำอีกรอบ... วนลูปไปเรื่อยๆ เป็นล้านรอบจนแรมแตก (StackOverflow) ครับ!
  **วิธีแก้:** เขาต้องใช้ "ตัวแปรเบื้องหลัง" (Backing Field) มารับค่าสิครับ เช่น `_hp = value;` (ซ่อนตัวเล็กไว้หลังบ้าน)
- **สำหรับโจทย์ 2:** สั้นๆ ง่ายๆ สะเทือนวงการ: `public string Username { get; set; }` ครับ!
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: เครื่องปรับอากาศอัจฉริยะ (The Smart Thermostat)**

นี่คือการทำระบบ IoT (Internet of Things) ควบคุมอุณหภูมิห้อง
**ภารกิจของคุณ:** จงสร้างคลาส `Thermostat` ซึ่งมีกฎเกณฑ์ข้อบังคับดังนี้:

1. มี Property ชื่อ `DeviceName` แบบ Auto-Property ที่ให้ใครแก้ก็ได้
2. **ไฮไลต์:** มี Property ชื่อ `TemperatureC` (อุณหภูมิองศาเซลเซียส)
   - ต้องใช้ Full Property (มีด่านตรวจ `get`/`set` และมีตัวแปรลับ `private float _temperature;`)
   - **กฎด่านตรวจขาเข้า (Set):** แอร์เครื่องนี้สามารถทำความเย็นได้ต่ำสุดที่ `16` องศา และสูงสุดที่ `30` องศาเท่านั้น
   - ถ้าใครพยายามตั้งแอร์ `10` องศา ให้ปริ้นด่าว่า "หนาวเกินไป!" แล้วล็อกแอร์ไว้ที่ `16` (บังคับแก้ค่า)
   - ถ้าใครพยายามตั้งแอร์ `50` องศา ให้ปริ้นด่าว่า "ร้อนเกินไป!" แล้วล็อกแอร์ไว้ที่ `30` (บังคับแก้ค่า)
   - ถ้าตั้งปกติ (16-30) ให้อัปเดตได้เลย
3. ใน Main Method:
   - สั่งตั้งอุณหภูมิที่ 25 (ปกติ) -> ลองปริ้นอุณหภูมิมาดู
   - สั่งตั้งอุณหภูมิที่ -5 (หนาวสุดขั้ว) -> ลองปริ้นอุณหภูมิมาดู (มันต้องเป็น 16)
   - สั่งตั้งอุณหภูมิที่ 100 (ไฟไหม้) -> ลองปริ้นอุณหภูมิมาดู (มันต้องเป็น 30)

ถ้าคุณสามารถแปลงลอจิกการ "กรองขยะขาเข้า (Data Sanitization)" ไปยัดไว้ในท่อ `set` ได้อย่างสมบูรณ์แบบ... แปลว่าคุณได้ครอบครองอาวุธที่อันตรายที่สุดในภาษา C# แล้วครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในเอกสารไมโครซอฟต์ระบุว่า `public string Name { get; set; }` เบื้องหลังตอน Compile คอมพิวเตอร์จะแอบเอาไปแปลงร่างกลับเป็น Getter/Setter Method แบบภาษา Java ให้โดยอัตโนมัติ... ถ้าเป็นเช่นนั้น ทำไมการใช้ Property ถึงทำงานช้ากว่าการเอื้อมมือไปล้วง Fields `public string Name;` ตรงๆ ไหม?
**แนวคำตอบ:** **ช้ากว่าจิ๊ดดดดดดเดียวจนแทบเป็นศูนย์ (Negligible) ครับ!**
จริงอยู่ที่การเปิดประตู (Property) ต้องผ่านการเรียกฟังก์ชันซ่อนรูป (Method call overhead) แต่ไมโครซอฟต์ฉลาดมากครับ! คอมไพเลอร์ของ .NET (JIT Compiler) จะทำการ "Inlining" (ทุบกำแพงฟังก์ชันทิ้งแล้วเอาไส้ในไปแปะโดยตรงในตอนรันรหัส) ทำให้ประสิทธิภาพตอนโปรแกรมรันจริง เร็วเท่ากับการล้วง Field ตรงๆ แบบ 100% เลยครับ! (ดังนั้นใช้ Property ไปเถอะ ไม่ต้องกลัวช้า!)

**คำถาม 2:** ถ้าคลาสเรามีแต่ตัวแปร `public` เปล่าๆ ไม่สนใจเรื่องความปลอดภัยเลย... มีความจำเป็นต้องเปลี่ยนมันให้กลายเป็น `{ get; set; }` ไหม? 
**แนวคำตอบ:** **"จำเป็นอย่างยิ่งยวด!" ตามมาตรฐานสถาปัตยกรรม .NET ครับ!**
ในจักรวาล C# มีหลาย Framework (เช่น Entity Framework ที่ต่อฐานข้อมูล, ระบบ Data Binding ของหน้าจอ Windows Forms/WPF, หรือระบบแปลง JSON) **พวกมันจะ "ตาบอด" ทันทีถ้าคุณใช้ Fields เปล่าๆ!** พวกมันถูกดีไซน์มาให้ทำงานกับ **Properties** เท่านั้นครับ! 
ดังนั้น กฎเหล็กของมือโปร: "จงสร้าง Field เป็น private เสมอ และถ้าจะเปิดให้ภายนอกเห็น จงใช้ Property { get; set; } คลุมไว้ทุกกรณี!"
:::
