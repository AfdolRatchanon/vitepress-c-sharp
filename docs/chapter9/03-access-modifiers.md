# Access Modifiers (ยามรักษาความปลอดภัย)

> 💡 **เป้าหมาย:** ทำความรู้จักกับเสาหลักต้นที่ 1 ของ OOP "การห่อหุ้มซ่อนเร้น (Encapsulation)" เรียนรู้ความน่ากลัวของการใช้คำว่า `public` พร่ำเพรื่อ และวิธีการจ้างยาม `private` มาเฝ้าข้อมูลสำคัญ เพื่อป้องกันไม่ให้แฮกเกอร์หรือเพื่อนร่วมทีมมาทำระบบเราพัง!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. วิกฤติการแก้ข้อมูลข้ามหัว (The Naked Data Problem)

ในบทที่ผ่านมา เราสร้าง Class โดยใส่คำว่า `public` นำหน้าทุกอย่าง (`public int HP;`)
**`public` แปลว่า "สาธารณะ"** หมายความว่า ใครก็ตามที่เรียกใช้ Object นี้ สามารถเอื้อมมือมาล้วงหยิบข้อมูล หรือแม้กระทั่ง "เขียนค่าทับ" ลงไปได้โดยตรงตามใจชอบ!

ลองนึกภาพคุณสร้างคลาส `BankAccount` มีเงินอยู่ 1 ล้านบาท
แล้วมีโปรแกรมเมอร์ฝึกงานคนนึง เขียนโค้ดใน Main ว่า:
`myAccount.Balance = -500;` // ทะลึ่งเปลี่ยนยอดเงินในกระเป๋าเราให้ติดลบเฉยเลย!
ถ้าแอปพลิเคชันนี้ขึ้นระบบจริง ธนาคารคุณเจ๊งแน่นอนครับ! นี่คือหายนะของการ "แก้ข้อมูลข้ามหัว"

---

### 2. กำเนิดยามเฝ้าประตู `private` (The Encapsulation)

ในเมื่อเราไม่อยากให้คนนอกมาวุ่นวายกับข้อมูลดิบของเรา... กฎข้อแรกของการเป็นสถาปนิก OOP คือ **"ตัวแปร (Fields) ทุกตัวในโลก ต้องถูกตั้งเป็น `private` เสมอ!"**

**`private` แปลว่า "ส่วนตัว (ความลับสุดยอด)"** 
- ข้อมูลที่ถูกตั้งเป็น `private` จะสามารถมองเห็นและแก้ไขได้ **"เฉพาะภายในคลาสของมันเองเท่านั้น"**
- คนที่อยู่ข้างนอก (เช่น คลาส Program หรือ Main) จะมองไม่เห็นตัวแปรนี้ ราวกับว่ามันไม่มีอยู่จริง!

```csharp
class BankAccount
{
    // ความลับสุดยอด คนนอกห้ามแตะ!
    private decimal balance = 1000; 
}
// ถ้าใน Main เขียน myAccount.balance = 0; คอมไพเลอร์จะ Error ตัวแดงด่าทันที!
```

### 3. แล้วคนนอกจะใช้งานยังไงล่ะ? (The Public Interface)

ถ้าตู้เซฟล็อกสนิท (`private`) แล้วลูกค้าจะถอนเงินยังไง? 
คำตอบคือ: **เราต้องสร้างช่องทางสื่อสาร (Public Methods) ให้คนนอกเรียกใช้ครับ!**

เปรียบเทียบกับโลกจริง:
- **`private balance`:** คือตู้เซฟหลังร้าน (ลูกค้าเดินเข้าไปเปิดเองไม่ได้)
- **`public void Deposit(amount)`:** คือพนักงานหน้าเคาน์เตอร์ (ลูกค้าเดินมาคุยกับพนักงาน แล้วพนักงานที่มีสิทธิ์ จะเดินไปเปิดตู้เซฟให้เอง)

ความเจ๋งคือ: ในเมื่อผ่านพนักงาน (Method) พนักงานย่อมสามารถ "ตรวจสอบ (Validate)" ได้ก่อนว่า ยอดเงินที่ฝากเข้ามามันติดลบหรือเปล่า! นี่แหละครับคือการป้องกันข้อมูล (Data Hiding & Validation) ระดับสูงสุด!

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่าง: ระบบกระเป๋าเงินนิรภัย (The Encapsulated Wallet)

โค้ดนี้สาธิตความแตกต่างอย่างชัดเจน ระหว่างตัวแปรเปลือยเปล่า (Naked) กับตัวแปรที่ถูกห่อหุ้มไว้ (Encapsulated)

::: code-group
```csharp [Program.cs]
using System;

namespace EncapsulationConcept
{
    class Wallet
    {
        // -----------------------------------------------------------
        // 1. DATA (ห้ามใครแตะต้อง!)
        // ทริค: โปรแกรมเมอร์มักจะตั้งชื่อตัวแปร private ให้ขึ้นต้นด้วยตัวพิมพ์เล็ก 
        // หรือมี _ นำหน้า เพื่อแยกให้ชัดเจน
        // -----------------------------------------------------------
        private decimal _money = 0;

        // -----------------------------------------------------------
        // 2. BEHAVIOR (พนักงานหน้าเคาน์เตอร์ ที่คนนอกเรียกใช้งานได้)
        // -----------------------------------------------------------
        
        // เมธอดสำหรับ "ขอดูเงิน" (อนุญาตให้ดู แต่ไม่อนุญาตให้แก้!)
        public decimal CheckBalance()
        {
            return _money;
        }

        // เมธอดสำหรับ "ฝากเงิน" (พร้อมด่านตรวจจับความถูกต้อง)
        public void AddMoney(decimal amount)
        {
            if (amount <= 0)
            {
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("🚨 [System] ตรวจพบการทุจริต! ห้ามฝากเงินยอดติดลบ!");
                Console.ResetColor();
                return; // ดีดตัวออกจากฟังก์ชันทันที
            }

            // ถ้าตรวจสอบผ่าน ถึงจะยอมให้แตะต้องตัวแปร _money
            _money += amount;
            Console.WriteLine($"✅ ฝากเงินสำเร็จ: {amount} บาท");
        }

        // เมธอดสำหรับ "ใช้เงิน" (พร้อมด่านตรวจจับเงินทอน)
        public void Spend(decimal amount)
        {
            if (amount > _money)
            {
                Console.ForegroundColor = ConsoleColor.Red;
                Console.WriteLine("🚨 [System] ยอดเงินไม่พอจ่าย!");
                Console.ResetColor();
                return;
            }

            _money -= amount;
            Console.WriteLine($"💸 จ่ายเงินสำเร็จ: {amount} บาท");
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบกระเป๋าเงินดิจิทัลนิรภัย ===\n");

            Wallet myWallet = new Wallet();

            // 💥 พยายามแฮกระบบ: myWallet._money = 999999; 
            // ^ ลองเอาคอมเมนต์ออกดูครับ โค้ดพังทันทีเพราะ "Inaccessible due to its protection level"

            // -----------------------------------------------------------
            // การใช้งานผ่านช่องทางที่ถูกต้อง (Public Interface)
            // -----------------------------------------------------------
            myWallet.AddMoney(500);  // ฝากปกติ (ผ่าน)
            
            myWallet.AddMoney(-100); // แฮกเกอร์พยายามเจาะระบบ! (จะโดนด่า และเงินไม่เพิ่ม)
            
            myWallet.Spend(200);     // จ่ายค่าข้าว (ผ่าน)
            
            myWallet.Spend(1000);    // ซื้อของแพงเกินตัว (จะโดนด่า และเงินไม่ลด)

            // ขอดูยอดเงินสุดท้าย
            decimal current = myWallet.CheckBalance();
            Console.WriteLine($"\n[ยอดเงินคงเหลือสุทธิ] : {current} บาท");
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: เติมคำในช่องว่าง (Access Modifiers)**
มี Access Modifiers พื้นฐาน 2 ตัวคือ `public` และ `private`
จงระบุว่า:
1. "ตัวแปร (Fields)" ที่เก็บสถานะของระบบ ควรตั้งเป็นอะไร? 
2. "ฟังก์ชัน (Methods)" ที่เตรียมไว้ให้คลาสอื่นเรียกใช้งาน ควรตั้งเป็นอะไร?
3. ถ้าคุณเขียนสร้างตัวแปรในคลาสว่า `int Score;` (ลืมใส่คำนำหน้าใดๆ ทั้งสิ้น) ตัวแปรนั้นจะกลายเป็นอะไรโดยอัตโนมัติ?

### **โจทย์ที่ 2: ช่องโหว่ของโรงพยาบาล (The Naked Variable)**
พิจารณาคลาสนี้:
```csharp
class Patient {
    public int Age;
}
```
เจ้านายสั่งว่า "จงใช้กฎ Encapsulation เพื่อป้องกันไม่ให้คนพิมพ์อายุติดลบ (เช่น -5)"
**ภารกิจ:** จงรื้อคลาสนี้ใหม่ โดยเปลี่ยน `Age` ให้เป็น `private _age` และสร้างเมธอด `public void SetAge(int newAge)` ที่มี `if` คอยดักจับว่า ถ้ายื่นอายุต่ำกว่า 0 มา ให้อย่าอัปเดตค่าและปริ้นด่าผู้ใช้ซะ!

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** 
  1. Fields ควรเป็น `private` เสมอ (ซ่อนไส้ใน)
  2. Methods ควรเป็น `public` (ปุ่มให้คนกด)
  3. ถ้าไม่ใส่คำนำหน้าเลย C# จะตั้งค่า Default ให้เป็น **`private`** อัตโนมัติครับ! (กฎหมายการรักษาความลับเข้มงวดมาก)
- **สำหรับโจทย์ 2:**
  ```csharp
  private int _age;
  public void SetAge(int newAge) {
      if (newAge < 0) { Console.WriteLine("อายุห้ามติดลบ!"); return; }
      _age = newAge;
  }
  ```
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: หมอเถื่อนและยาผีบอก (The HP Validator)**

นี่คือการจำลองระบบป้องกันข้อมูลโกงในโลกของเกม (Anti-Cheat System)!
**ภารกิจของคุณ:** จงสร้างคลาส `Hero` ที่ปกป้องตัวเองจากการโดนฮีล (Heal) จนเลือดทะลุหลอด!

1. **กฎการสร้าง (Encapsulation):**
   - มีฟิลด์ `private int _hp;`
   - มีฟิลด์ `private int _maxHp = 100;` (เลือดเต็มหลอดได้แค่ 100 ห้ามเกินนี้)
2. **ช่องทางการสื่อสาร (Public Interface):**
   - สร้างเมธอด `public void TakeDamage(int damage)` เพื่อหัก `_hp` ลง (ถ้ายิงจนเลือดต่ำกว่า 0 ให้ล็อกค่าไว้ที่ 0 พอ ห้ามเลือดติดลบ)
   - สร้างเมธอด `public void Heal(int amount)` เพื่อเพิ่ม `_hp` ขึ้น **(ด่านตรวจสำคัญ: ถ้าฮีลแล้วเลือดทะลุ _maxHp ให้ปัดเศษทิ้ง แล้วล็อกเลือดไว้ที่ _maxHp เท่านั้น!)**
   - สร้างเมธอด `public void ShowStatus()` เพื่อปริ้นว่า "HP: 80 / 100"
3. **การทดสอบใน Main:**
   - เสก Hero ขึ้นมา (ตอนแรกเกิดสมมติให้เมธอด Heal ช่วยฮีลให้เต็ม 100 ก่อน)
   - โดนตบ 40 (เหลือ 60)
   - ดื่มโพชั่นฮีล 500! (ต้องไม่ทะลุ 100)
   - โดนบอสฟัน 999! (ต้องไม่ติดลบ หยุดที่ 0)

ถ้าคุณเขียนด่านตรวจ `if` เหล่านี้ดักหน้าดักหลังตัวแปร `_hp` ได้สำเร็จ แปลว่าคุณเข้าใจหัวใจของการ "ห่อหุ้มข้อมูล (Encapsulation)" แล้ว 100% ครับ! (บริษัทเกมรักคุณแน่นอน!)

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในเมื่อการปกป้องข้อมูลเป็นสิ่งที่ดี... ทำไมเราไม่สร้าง Method ให้เป็น `private` ไปด้วยเลยล่ะ? ปลอดภัยแบบ 100% ไง!
**แนวคำตอบ:** **ตู้เซฟที่ไม่มีกุญแจเปิดไข มันก็คือเศษเหล็กครับ!**
ถ้าคุณตั้ง Methods เป็น `private` หมด คลาสอื่นก็ไม่สามารถสั่งให้ Object นี้ทำงานอะไรได้เลย (สร้างมาแล้วก็เป็นใบ้ทำอะไรไม่ได้)
แต่อย่างไรก็ตาม... การสร้าง `private Method` ถือเป็นเรื่องปกติมากนะ! เรามักจะใช้มันสำหรับ "ลอจิกช่วยคำนวณยิบย่อยภายใน (Helper/Utility Methods)" ที่ไม่อยากให้คนนอกรู้ เช่น การหักภาษีซับซ้อน เราอาจจะซ่อนสูตรคำนวณไว้ใน `private decimal CalculateTax()` แล้วให้ `public void Withdraw()` ไปเรียกใช้สูตรนี้อีกทีแบบลับๆ ครับ นี่แหละการซ่อนความซับซ้อน (Abstraction) ขนานแท้!

**คำถาม 2:** การต้องมานั่งเขียนเมธอด `SetAge()` หรือ `GetMoney()` บ่อยๆ มันไม่ทำให้โค้ดยาวเฟื้อยและน่าเบื่อหรอกหรือ? ภาษา Java เค้าเขียนกันแบบนี้เต็มไปหมดเลยนะ!
**แนวคำตอบ:** **โคตรน่าเบื่อเลยครับ!**
วิศวกรซอฟต์แวร์ทั่วโลกบ่นเรื่องนี้กับภาษา Java มาเป็นสิบปี เรียกว่าปัญหา "Boilerplate Code" (โค้ดขยะที่ต้องพิมพ์ซ้ำๆ) 
แต่ช้าก่อน... นี่คือ C#! ภาษาที่ถูกสร้างมาเพื่อแก้ความน่ารำคาญของ Java! ไมโครซอฟต์รู้ปัญหานี้ดี จึงได้สร้างสิ่งประดิษฐ์ใหม่ที่ชื่อว่า **"Properties (Get/Set)"** ขึ้นมา เพื่อรวบยอดการทำด่านตรวจข้อมูลแบบบรรทัดเดียวจบ! ซึ่งเรากำลังจะได้เรียนความว้าวของมันในหัวข้อถัดไปครับ! เตรียมตัวทิ้ง `GetMoney()` ไปได้เลย!
