# Queue & Stack (จัดระเบียบสังคม)

> 💡 **เป้าหมาย:** สิ้นสุดยุคแห่งการหยิบของมั่วซั่ว! ค้นพบโครงสร้างข้อมูลระดับองค์กร 2 ตัวที่เกิดมาเพื่อ "วางกฎระเบียบ (Enforce Rules)" ว่าข้อมูลชิ้นไหนจะได้เกิดก่อน ชิ้นไหนจะต้องรอคิว ผ่านสถาปัตยกรรมระดับโลกอย่าง FIFO (มาก่อน-ได้ก่อน) และ LIFO (มาทีหลัง-ได้ก่อน)

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ความอันตรายของ List (The Too-Flexible Problem)

`List` ถือเป็นสุดยอดโกดังยางยืดที่เราโปรดปราน แต่ความอิสระของมัน (คุณสามารถ `Insert` แซงคิวช่องไหนก็ได้ หรือ `RemoveAt` เตะคนตรงกลางทิ้งเมื่อไหร่ก็ได้) อาจนำมาซึ่งบั๊กในระบบที่มีกฎเกณฑ์เข้มงวด

ลองนึกภาพคุณเป็นคนเขียนซอฟต์แวร์ "ระบบกดบัตรคิวโรงพยาบาล"
ถ้าคุณใช้ `List` แล้วมีพยาบาลตัวร้ายแอบพิมพ์โค้ด `patientList.Insert(0, "ญาติผู้อำนวยการ")` (แซงคิวเข้าช่องแรกสุด) ระบบคุณก็จะพังทลายด้านความยุติธรรมทันที

ดังนั้น วิศวกรซอฟต์แวร์จึงสร้าง โครงสร้างหุ้มเกราะ ขึ้นมา 2 ชนิด ที่ตัดสิทธิ์การเสียบกลางทิ้งไปให้หมด เหลือแค่ทางเข้า 1 ทาง และทางออก 1 ทางเท่านั้น!

---

### 2. Queue (คิว) - กฎแห่งความยุติธรรม (FIFO)

**Queue (คิว)** คือท่อตรงยาวๆ มีการทำงานบนคอนเซปต์ **FIFO (First In, First Out = มาก่อน ได้ก่อน)**
คุณสมบัติ: 
- เพิ่มของใหม่ ได้แต่ดันเข้าที่ **"ท้ายแถว (Rear)"** เท่านั้น (ห้ามแซงคิว)
- ดึงของออก ได้แต่หยิบจาก **"หัวแถว (Front)"** เท่านั้น (ห้ามข้ามหน้าข้ามตา)

**ศัพท์เทคนิค (Terminology):**
- **`Enqueue(item)`:** ต่อคิว (เอาของยัดเข้าท้ายแถว)
- **`Dequeue()`:** เรียกคิว (ดึงคนแรกสุดออกจากแถวมาใช้งาน และคนนั้นจะปลิวหายไปจากระบบทันที)
- **`Peek()`:** ชะโงกดู (ขอดูหน้าคนแรกสุดหน่อย แต่ไม่ดึงเค้าออกมานะ ให้อยู่คิวเดิม)

*ตัวอย่างระบบที่ใช้คิว:* คิวรอเครื่องปริ้นเตอร์ (ใครสั่งปริ้นก่อน กระดาษออกก่อน), คิวจับคู่ผู้เล่น (Matchmaking) ในเกม ROV หรือ Valorant

---

### 3. Stack (สแต็ก) - กฎแห่งการซ้อนทับ (LIFO)

**Stack (สแต็ก)** คือปิ่นโต หรือกระบอกใส่ลูกแบดมินตัน มีการทำงานบนคอนเซปต์ **LIFO (Last In, First Out = มาทีหลัง แต่ได้ออกก่อน)**
คุณสมบัติ:
- ทางเข้าและทางออก มีรูเดียวคือ "ปากกระบอก"
- ของชิ้นใหม่ จะถูกวางทับของเก่า (ซ้อนสูงขึ้นเรื่อยๆ)
- เวลาดึงของออก คุณจำเป็นต้องหยิบของที่ "อยู่บนสุด" (ชิ้นล่าสุด) ออกไปก่อนเสมอ!

**ศัพท์เทคนิค (Terminology):**
- **`Push(item)`:** กดทับลงไป (วางของทับไปข้างบนสุด)
- **`Pop()`:** เด้งออกมา (หยิบของบนสุดหลุดออกมา 1 ชิ้น)
- **`Peek()`:** ชะโงกดู (ดูว่าของชิ้นบนสุดคืออะไร แต่ไม่หยิบออก)

*ตัวอย่างระบบที่ใช้สแต็ก:* ปุ่มกดย้อนกลับ (Back button) ในเว็บบราว์เซอร์, ระบบกดปุ่มเลิกทำ `Ctrl+Z` (Undo) ใน Word หรือ Photoshop (เพราะเราต้องการถอนการกระทำ "ล่าสุด" กลับคืนมา) รวมถึงหน่วยความจำ Call Stack (ที่เราเรียนในบท Recursion!)

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: ระบบบริการลูกค้า (Queue Simulator)

โค้ดนี้สาธิตวัฏจักรของการ Enqueue (สร้างคิว) และ Dequeue (ระบายคิว) ซึ่งเป็นหัวใจหลักของ Background Worker ในการออกแบบเซิร์ฟเวอร์

::: code-group
```csharp [Program.cs]
using System;
using System.Collections.Generic; // Queue และ Stack อาศัยอยู่ในนี้!

namespace HospitalQueue
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบเรียกคิวคลินิก (First In, First Out) ===\n");

            // ประกาศคิวที่รองรับข้อมูลแบบข้อความ
            Queue<string> patientQueue = new Queue<string>();

            // 1. กระบวนการจ่ายบัตรคิว (Enqueue)
            patientQueue.Enqueue("คุณสมชาย (ปวดหัว)");
            patientQueue.Enqueue("คุณสมศรี (เจ็บคอ)");
            patientQueue.Enqueue("คุณจอห์น (ผื่นคัน)");

            Console.WriteLine($"[สถานะ] ตอนนี้มีผู้ป่วยรอคิวทั้งหมด {patientQueue.Count} คิว\n");

            // แอบดูว่าใครคือคิวแรกที่กำลังจะได้เรียก (โดยยังไม่ดึงเค้าออกจากแถว)
            string nextInLine = patientQueue.Peek();
            Console.WriteLine($"💡 แจ้งเตือนพยาบาล: เตรียมห้องให้ผู้ป่วยคิวถัดไปคือ {nextInLine}\n");

            // 2. กระบวนการเรียกคิวระบายออก (Dequeue)
            // ใช้ while loop วิ่งกวาดคิว จนกว่าจะไม่มีใครเหลือ (Count = 0)
            while (patientQueue.Count > 0)
            {
                // ดึงคนหัวแถวหลุดออกมา (เค้าจะหายไปจาก patientQueue ทันที)
                string activePatient = patientQueue.Dequeue();
                
                Console.WriteLine($"📢 เชิญ {activePatient} เข้าห้องตรวจครับ");
                
                // หลังจากคนนี้เข้าห้องไป คิวของคนที่สองจะเลื่อนขึ้นมาเป็นอันดับ 1 อัตโนมัติ!
            }

            Console.WriteLine("\n🎉 ตรวจครบทุกคนแล้ว ปิดคลินิกได้!");
            
            // 💥 ระวัง: ถ้าพยายามสั่ง patientQueue.Dequeue() อีกครั้งตอนที่กล่องว่างเปล่า (Count=0) 
            // โปรแกรมจะแครชกระจาย! (InvalidOperationException) นี่คือเหตุผลที่เราต้องเอา while หรือ if ดักไว้เสมอ
        }
    }
}
```
:::

### ตัวอย่างที่ 2: ระบบย้อนเวลา `Ctrl+Z` (Stack Simulator)

การใช้ Stack เป็นศาสตร์ที่สนุกมาก เพราะคุณจะได้เห็นข้อมูลถอยหลังกลับไปหาต้นขั้ว

::: code-group
```csharp [Program.cs]
using System;
using System.Collections.Generic;

namespace UndoSystem
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบจำลองปุ่ม Undo (Last In, First Out) ===\n");

            // ประวัติการพิมพ์ (วางซ้อนกันเป็นกระบอก)
            Stack<string> textHistory = new Stack<string>();

            // 1. ผู้ใช้ทำการพิมพ์ข้อความใหม่ (Push ทับลงไป)
            Console.WriteLine("[ผู้ใช้พิมพ์] 'สวัสดีครับ'");
            textHistory.Push("สวัสดีครับ");

            Console.WriteLine("[ผู้ใช้พิมพ์] 'สบายดีไหม'");
            textHistory.Push("สบายดีไหม");

            Console.WriteLine("[ผู้ใช้พิมพ์] 'ฮ่าๆๆ เผลอพิมพ์ผิด'");
            textHistory.Push("ฮ่าๆๆ เผลอพิมพ์ผิด");

            Console.WriteLine($"\n[สถิติ] มีประวัติถูกบันทึกไว้ในสแต็ก {textHistory.Count} ชิ้น\n");

            // 2. ผู้ใช้กดปุ่ม Ctrl+Z (เพื่อลบคำล่าสุดทิ้ง)
            if (textHistory.Count > 0)
            {
                string discardedWord = textHistory.Pop(); // ดึงชั้นบนสุดหลุดกระเด็นออกไป
                Console.WriteLine($"♻️ ระบบทำการ Undo (ถอนเอาคำว่า '{discardedWord}' ทิ้งไปแล้ว!)");
            }

            // 3. ดูว่าตอนนี้เอกสารเหลืออะไรเป็นหน้าล่าสุด
            if (textHistory.Count > 0)
            {
                // Peek ดูชั้นบนสุดที่เหลืออยู่ (ซึ่งมันถูกเลื่อนระดับขึ้นมาแทนที่ตัวที่โดนเตะทิ้งไปแล้ว)
                Console.WriteLine($"📄 คำที่อยู่บนสุดของเอกสารปัจจุบันคือ: '{textHistory.Peek()}'");
            }
        }
    }
}
```
:::

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: เถียงกับกฎระเบียบ**
ถ้าคุณมีโค้ด `Queue<int> line = new Queue<int>();` แล้วทำการ `Enqueue` เลข 10, 20, 30 ลงไปตามลำดับ 
มีวิธีไหนใน C# ที่จะทำให้เราดึงเลข "20" ออกมาจากคิว โดยที่ "เลข 10 ยังคงต้องคาอยู่ในคิวเหมือนเดิม" หรือไม่? จงให้เหตุผลทางวิศวกรรม

### **โจทย์ที่ 2: รหัสมอสกลับหลัง (The Palindrome Backwards)**
**ภารกิจ:** จงสร้าง `Stack<char>` ขึ้นมา 1 ตัว เพื่อใช้ในการเล่นมายากลกลับคำ
- นำตัวอักษรของคำว่า `"HELLO"` ใส่เข้าไปใน Stack ทีละตัวอักษร ด้วยลูป `foreach`
- จากนั้น ให้สร้างลูป `while` เพื่อดึง (Pop) ตัวอักษรออกมาทีละตัว นำไปพิมพ์โชว์บนหน้าจอจนกว่า Stack จะว่างเปล่า
ผลลัพธ์บนหน้าจอที่คุณควรจะได้คืออักษรคำว่าอะไร? (มันควรจะช่วยตอบคำถามว่าทำไม Stacks ถึงมีประโยชน์มหาศาลในการกลับหลังหันข้อมูล!)

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** **ไม่มีทางทำได้ และห้ามทำเด็ดขาด!** Queue ถูกออกแบบมาเป็น Blackbox ที่ปิดตายด้านข้าง ถ้าคุณอยากดึงเลข 20 คุณ "บังคับ" ต้อง Dequeue เลข 10 ออกมาสังเวยทำลายทิ้งไปก่อนเท่านั้น! (ถ้าคุณเถียงว่าคุณแค่อยากดึง 20... คุณเลือกโครงสร้างข้อมูลผิดแล้วครับ คุณต้องกลับไปใช้ `List`)
- **สำหรับโจทย์ 2:** โค้ดควรหน้าตาแบบ `while (stack.Count > 0) Console.Write(stack.Pop());` และผลลัพธ์ที่คุณจะได้คือ `"OLLEH"`! เพราะตัว 'O' โดนใส่เข้าไปเป็นคนสุดท้าย มันจึงอยู่ชั้นบนสุดและโดนดึงออกมาเป็นคนแรก (First Out)

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: เครื่องตรวจสอบความสมดุลของวงเล็บ (The Parentheses Balancer)**

นี่คือข้อสอบเข้าทำงาน (Whiteboard Interview) ที่โด่งดังที่สุดในโลกเพื่อวัดความเข้าใจเรื่อง Stack!
ตัวคอมไพเลอร์ของ C# รู้ได้อย่างไรว่าคุณ "เปิดปีกกา `{` แล้วลืมปิดปีกกา `}`"? มันใช้ Stack นี่แหละครับ!

**ภารกิจของคุณ:** 
จงเขียนโปรแกรมรับสายข้อความวงเล็บ (เช่น `(())` หรือ `()()` หรือ `(()`) แล้วตรวจสอบว่า วงเล็บเหล่านั้น **"สมดุล (Balanced)"** หรือไม่? (เปิดกี่อัน ต้องปิดให้ถูกจังหวะเท่านั้น)

**วิชามาร (Algorithm Guide):**
1. สร้างกระบอก `Stack<char>` ว่างๆ ขึ้นมา 1 อัน
2. วิ่งลูปกวาดสายตาข้อความทีละตัวอักษร (`foreach (char c in inputText)`)
3. **กฎข้อที่ 1:** ถ้าเจอสัญลักษณ์วงเล็บเปิด `(` ให้ทำการ **กด (Push)** สัญลักษณ์นั้นใส่ลงกระบอก Stack 
4. **กฎข้อที่ 2:** ถ้าเจอสัญลักษณ์วงเล็บปิด `)` ให้เราไปดูที่กระบอก Stack ว่ามีของอยู่ข้างในไหม?
   - ถ้ากระบอก *ว่างเปล่า (Count == 0)*... แปลว่ามีคนทะลึ่งปิดวงเล็บ ทั้งที่ไม่มีการเปิด! -> ฟันธงเลยว่า "พัง (Unbalanced) ❌"
   - แต่ถ้ากระบอก *มีของอยู่*... ให้ทำการ **เด้ง (Pop)** วงเล็บเปิดที่ค้างอยู่ออกมา 1 ตัว ถือว่า "จับคู่ฆ่ากันทิ้งไป (Matched)"
5. พอวิ่งลูปจนจบประโยคปุ๊บ ให้ตรวจสอบสภาพกระบอก Stack เป็นครั้งสุดท้าย...
   - ถ้ากระบอก *ว่างเปล่า* ไม่มีอะไรค้างอยู่เลย -> แปลว่ามันจับคู่เคลียร์กันเสร็จสมบูรณ์ -> "สมดุล (Balanced) ✅"
   - ถ้ากระบอก *ยังมีของค้างอยู่* (Count > 0) -> แปลว่ามีคนเปิดวงเล็บทิ้งไว้ แต่ลืมปิดมัน! -> ฟันธงว่า "พัง (Unbalanced) ❌"

ถ้าคุณประกอบลอจิกตามนี้สำเร็จ คุณสามารถเอาประวัติไปสมัครงานบริษัทยักษ์ใหญ่ได้อย่างเต็มภาคภูมิเลยครับ!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ในเอกสารไมโครซอฟต์ระบุว่า `Queue` และ `Stack` มีเบื้องหลังเป็น "Array ยางยืด (Dynamic Array)" เหมือนกับ `List` เลย ถ้างั้นเราก็ใช้ `List` ตัวเดียวไปเลย ไม่ต้องง้อ Queue กับ Stack สิ มันทำอะไรได้ครอบจักรวาลกว่าตั้งเยอะ?
**แนวคำตอบ:** เป็นความจริงที่ List ทำทุกอย่างที่ Queue และ Stack ทำได้ (เช่น ใช้ `.Add` ดันของเข้าหลัง และใช้ `.RemoveAt(0)` เพื่อดึงของออกหน้าสุด ก็จะกลายเป็นคิวปลอมๆ ได้) 
แต่! **"การปล่อยปละละเลยเรื่องความปลอดภัย ถือเป็นบาปร้ายแรงของวิศวกรซอฟต์แวร์!"** 
การที่คุณบังคับเปลี่ยนตัวแปรให้เป็นชนิด `Queue` มันคือการแสดง "เจตนารมณ์ (Intent)" ที่ชัดเจนให้โปรแกรมเมอร์เพื่อนร่วมทีมรู้ว่า "โกดังใบนี้ ห้ามสอดแทรก หรือข้ามคิวนะ!" มันเป็นเกราะป้องกันทางสถาปัตยกรรม (Architectural Boundary) ที่จะคอมไพล์ไม่ผ่านทันทีที่มีคนพยายามเอาเลข Index ไปจิ้มแทรกกลาง ซึ่งช่วยลดโอกาสเกิดบั๊กจากการทำงานของมนุษย์ลงไปได้ 100% เต็ม! 

**คำถาม 2:** โค้ด `Stack<int>` และบรรทัดต่อมาสั่ง `stack.Pop();` โดยที่กล่องยังว่างเปล่า จะเกิดอะไรขึ้น? มีคำสั่งที่คล้ายๆ `TryGetValue` ของ Dictionary ให้ใช้งานไหม?
**แนวคำตอบ:** ระเบิดตู้ม (InvalidOperationException) แน่นอนครับ! กฎคือห้าม Pop หรือ Dequeue หรือแม้แต่ Peek ของจากกล่องที่ว่างเปล่า (Count = 0) เด็ดขาด 
แต่โชคดีมาก! ใน C# ยุคใหม่ เค้าได้เสริมเขี้ยวเล็บให้พวกมันด้วยเมธอด **`TryDequeue(out int result)`** และ **`TryPop(out int result)`** ซึ่งทำงานเหมือน `TryGetValue` เป๊ะๆ คือถ้ากล่องว่าง มันจะรีเทิร์น `false` แบบชิวๆ ให้คุณรอดพ้นจากการแครชนั่นเอง!
