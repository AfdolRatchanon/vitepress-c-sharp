# Method Overloading (ชื่อซ้ำ ทำงานต่าง)

> 💡 **เป้าหมาย:** ทำความรู้จักกับศิลปะการตั้ง "ชื่อเดียว" แต่ใช้ครอบจักรวาล เพื่อสร้างฟังก์ชันที่ฉลาดพอจะเลือกพฤติกรรมการทำงานของตัวเองได้อัตโนมัติ ตาม "ชนิดของข้อมูล" ที่เราโยนเข้าไป (นี่คือพื้นฐานของแนวคิด Polymorphism ในโลก OOP)

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ปัญหาของการมีชื่อมากเกินไป (The Naming Nightmare)

สมมติว่าคุณต้องการสร้างไลบรารีคณิตศาสตร์ เอาไว้ใช้บวกเลข
- ถ้าจะบวกเลขจำนวนเต็ม 2 ตัว คุณอาจจะตั้งชื่อว่า `AddTwoInts(int a, int b)`
- ถ้าจะบวกเลขทศนิยม คุณอาจจะตั้งชื่อว่า `AddTwoDoubles(double a, double b)`
- ถ้าจะบวกเลข 3 ตัว คุณก็ตั้งชื่อว่า `AddThreeInts(int a, int b, int c)`

ปัญหาคือ ถ้าคุณมีระบบขนาดใหญ่ คนที่เอาไลบรารีคุณไปใช้ ต้องนั่งท่องจำชื่อเมธอดนับร้อยชื่อ! ทั้งๆ ที่พฤติกรรมหลักมันก็คือ **"การบวก (Add)"** เหมือนกันหมด 

---

### 2. ไวยากรณ์แห่งการซ้อนทับ (Method Overloading)

ภาษา C# (และภาษาตระกูล C อื่นๆ) มีความสามารถพิเศษที่อนุญาตให้เรา **"สร้างเมธอดชื่อซ้ำกันได้กี่ตัวก็ได้"** 
โดยมีกฎเหล็กเพียงข้อเดียวคือ: **"ลายเซ็น (Method Signature) ต้องไม่เหมือนกัน"**

ลายเซ็น (Signature) ในภาษา C# ประกอบด้วย 2 อย่าง:
1. ชื่อของเมธอด (Name)
2. รูปร่างของพารามิเตอร์ (Parameter List) เช่น จำนวนตัวแปร, ชนิดของตัวแปร

```csharp
// เมธอดที่ 1: รับ int 2 ตัว
static int Add(int a, int b) { return a + b; }

// เมธอดที่ 2: ชื่อ Add เหมือนกันเป๊ะ! แต่รับ double 2 ตัว (ลายเซ็นต่างกัน -> โหลดทับได้!)
static double Add(double a, double b) { return a + b; }

// เมธอดที่ 3: ชื่อ Add เหมือนกัน! แต่รับ int 3 ตัว (จำนวนไม่เท่ากัน -> โหลดทับได้!)
static int Add(int a, int b, int c) { return a + b + c; }
```

**เวทมนตร์เกิดขึ้นตอนเรียกใช้งาน (Invocation Magic):**
เมื่อคุณพิมพ์คำสั่ง `Add(5, 10);` ใน `Main()` ตัว Compiler ของ C# จะสวมบทบาทเป็นนักสืบ มันจะตรวจดูว่า ข้อมูล 5 และ 10 เป็นชนิด `int` ทั้งคู่ มันจึงทำหน้าที่ "สับสวิตช์" กระโดดไปเรียก **เมธอดที่ 1** ให้โดยอัตโนมัติแบบไร้รอยต่อ! (นี่คือระบบที่เรียกว่า Compile-time Polymorphism)

---

### 3. กรณีต้องห้าม (Invalid Overloading)

สิ่งที่โปรแกรมเมอร์มักจะสับสนมากที่สุดคือ **"การเปลี่ยนแค่ Return Type ไม่ถือว่าเป็นการ Overload ที่ถูกต้อง"**
และจะทำให้โปรแกรม Compile พังทันที!

```csharp
// ❌ โค้ดนี้ Compile ไม่ผ่าน! (Ambiguous Error)
static int GetScore(string name) { return 100; }
static double GetScore(string name) { return 100.5; }
```
**ทำไมถึงพัง?:** เพราะเวลาที่คุณเรียกใช้ `GetScore("John");` ใน Main ทาง Compiler จะไม่รู้เลยว่าตกลงคุณอยากได้กล่อง `int` หรือกล่อง `double` กันแน่ (เพราะข้อมูลที่คุณโยนเข้าไปมันคือ string ตัวเดียวเหมือนกันเป๊ะ) จำไว้เสมอว่า C# ตัดสิน Overload จาก **"สิ่งที่โยนเข้าไป (Inputs)"** ไม่ใช่สิ่งที่พ่นออกมา!

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: ระบบคำนวณพื้นที่รูปทรง (The Geometry Engine)

โค้ดนี้สาธิตถึงเหตุผลทางคณิตศาสตร์ว่าทำไมเราถึงต้องมี Overload (การคำนวณพื้นที่แต่ละรูปทรงใช้สูตรไม่เหมือนกัน แต่เราอยากเรียกชื่อเมธอดเดียวกันเพื่อให้จำง่าย)

::: code-group
```csharp [Program.cs]
using System;

namespace MethodOverloadingDemo
{
    class Program
    {
        // --------------------------------------------------------
        // 1. หาพื้นที่ "สี่เหลี่ยมจัตุรัส" (Square)
        // สูตร: ด้าน x ด้าน (รับค่าเพียง 1 ตัว)
        // --------------------------------------------------------
        static double CalculateArea(double side)
        {
            Console.WriteLine("[ระบบ] ทำงานด้วยสูตร: สี่เหลี่ยมจัตุรัส");
            return side * side;
        }

        // --------------------------------------------------------
        // 2. หาพื้นที่ "สี่เหลี่ยมผืนผ้า" (Rectangle)
        // สูตร: กว้าง x ยาว (รับค่า 2 ตัว)
        // --------------------------------------------------------
        static double CalculateArea(double width, double length)
        {
            Console.WriteLine("[ระบบ] ทำงานด้วยสูตร: สี่เหลี่ยมผืนผ้า");
            return width * length;
        }

        // --------------------------------------------------------
        // 3. หาพื้นที่ "วงกลม" (Circle)
        // เพื่อป้องกันลายเซ็นไปซ้ำซ้อนกับจัตุรัส (รับ 1 ตัวเหมือนกัน) 
        // เราใช้วิธีรับค่าเป็นชนิด string ไปเลย หรือโยนสัญลักษณ์เพิ่ม
        // --------------------------------------------------------
        static double CalculateArea(double radius, string shape)
        {
            Console.WriteLine("[ระบบ] ทำงานด้วยสูตร: วงกลม (พาย x R^2)");
            return Math.PI * (radius * radius); // Math.PI มีค่า 3.14159...
        }

        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบคำนวณพื้นที่อัจฉริยะ ===");

            // ผู้เรียก (Client) ใช้งานสบายมาก ไม่ต้องจำชื่อหลายแบบ
            
            // เคสที่ 1: โยนเลขตัวเดียว คอมไพเลอร์รู้ทันทีว่าเป็นจัตุรัส
            double sqArea = CalculateArea(5.0);
            Console.WriteLine($"พื้นที่จัตุรัส: {sqArea}\n");

            // เคสที่ 2: โยนเลขสองตัว คอมไพเลอร์วิ่งไปหาสูตรผืนผ้าให้เอง!
            double rectArea = CalculateArea(10.0, 4.5);
            Console.WriteLine($"พื้นที่ผืนผ้า: {rectArea}\n");

            // เคสที่ 3: โยนเลขคู่กับข้อความ คอมไพเลอร์วิ่งไปหาสูตรวงกลม
            double circArea = CalculateArea(7.0, "circle");
            Console.WriteLine($"พื้นที่วงกลม: {circArea:N2}\n");
        }
    }
}
```
:::

### ตัวอย่างที่ 2: ความลับของ `Console.WriteLine`

คุณอาจจะไม่รู้ตัว แต่ตลอดเวลาที่คุณเขียน C# มาตั้งแต่บทที่ 1 คุณใช้งานระบบ Method Overloading มาโดยตลอด!
คำสั่ง `Console.WriteLine()` ถือเป็นสุดยอดผลงานศิลปะการ Overload ของไมโครซอฟต์

ลองคลิกขวาที่คำว่า `WriteLine` ใน Visual Studio แล้วเลือก `Go to Definition` ดูสิ คุณจะพบว่าในแกนกลางของ C# มีการเขียนเมธอดนี้เตรียมไว้ถึง **18 รูปแบบ (18 Overloads)** เช่น:
- `public static void WriteLine(int value)`
- `public static void WriteLine(string value)`
- `public static void WriteLine(bool value)`
- `public static void WriteLine(char[] buffer)`
- และอื่นๆ อีกมากมาย

มันถูกออกแบบมาเพื่อให้ไม่ว่าคุณจะโยนตัวเลข โยนข้อความ หรือโยนตรรกะเข้าไป ตัวมันก็สามารถแปลงทุกอย่างออกมาพิมพ์โชว์บนหน้าจอได้อัตโนมัติ โดยที่คุณไม่ต้องไปท่องชื่อว่า `Console.WriteIntegerLine()` ให้ปวดหัวเลย!

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: ลายเซ็นซ่อนเร้น (Signature Ambiguity)**
ถ้าเราประกาศเมธอด 2 ตัวไว้ดังนี้:
1. `static void ProcessData(int a, double b)`
2. `static void ProcessData(double a, int b)`

**คำถาม:** การประกาศแบบนี้ถูกต้องตามกฎ Overload หรือไม่? 
และถ้าใน `Main` เราเผลอเรียก `ProcessData(5, 10);` (โยน int ทั้งสองตัว) จะเกิดอะไรขึ้น? ทำไมคอมพิวเตอร์ถึงเป็นอัมพาต?

### **โจทย์ที่ 2: ผู้เชี่ยวชาญการต่อสติง (The String Weaver)**
**ภารกิจ:** จงสร้างชุดเมธอด Overloading ชื่อ `CombineData` จำนวน 3 รูปแบบ:
1. รับ `string` 2 ตัว -> ให้คืนค่าโดยเอา 2 คำมาต่อกันตรงๆ (เช่น "Hello" + "World" -> "HelloWorld")
2. รับ `string` 3 ตัว -> ให้คืนค่าโดยเอา 3 คำมาต่อกัน และคั่นด้วยขีดล่าง (เช่น "A", "B", "C" -> "A_B_C")
3. รับ `int` 2 ตัว -> ให้คืนค่าเป็นผลลัพธ์การบวกเลข แต่ **ถูกแปลงร่างเป็น String** แล้ว! (เช่น 5, 10 -> "15")

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** กฎของการเรียงพารามิเตอร์ (int, double) กับ (double, int) ถือเป็นลายเซ็นคนละแบบ! ดังนั้นมัน Compile ผ่านตอนสร้าง แต่จะไปพังทลายตอนคุณโยน `(5, 10)` เข้าไป เพราะ 5 และ 10 เป็น int ทั้งคู่ คอมไพเลอร์จะเถียงกันเองว่า "จะแปลง 10 เป็น double แล้วเข้าเมธอดแรกดีไหม? หรือจะแปลง 5 เป็น double แล้วเข้าเมธอดสองดี?" ทำให้เกิด Error `ambiguous call`
- **สำหรับโจทย์ 2:** เมธอดที่ 3 ต้องประกาศว่า `static string CombineData(int x, int y)` และมีโค้ดคือ `int sum = x + y; return sum.ToString();`
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: บิลเงินสดอัจฉริยะ (The Smart Receipt Generator)**

บริษัทค้าปลีกแห่งหนึ่งต้องการระบบพิมพ์ใบเสร็จ แต่สินค้าและลูกค้าแต่ละประเภทมีโครงสร้างการชำระเงินที่แตกต่างกันอย่างสิ้นเชิง 

**ภารกิจของคุณ:** 
ให้สร้างกลุ่มเมธอด Overloading ชื่อ `PrintReceipt` ตามข้อกำหนดต่อไปนี้:

1. **โหมดปกติ (สินค้าราคาเดียว):** 
   - รับค่า: ชื่อสินค้า (`string`), ราคา (`double`)
   - พิมพ์บิล: ชื่อสินค้า พร้อมราคาปกติ

2. **โหมดขายส่ง (ซื้อหลายชิ้น):**
   - รับค่า: ชื่อสินค้า (`string`), ราคา (`double`), จำนวนชิ้น (`int`)
   - พิมพ์บิล: ชื่อสินค้า x จำนวนชิ้น, ราคาสุทธิ (ราคา * จำนวน)

3. **โหมดสมาชิกระดับ VIP (มีส่วนลด):**
   - รับค่า: ชื่อสินค้า (`string`), ราคา (`double`), ส่วนลดเปอร์เซ็นต์ (`double`), รหัสสมาชิก (`string`)
   - พิมพ์บิล: ชื่อสินค้า, ราคาก่อนลด, แจ้งรหัสสมาชิก, ราคาสุทธิหลังหักส่วนลด

เมื่อสร้างทั้ง 3 รูปแบบเสร็จ ให้จำลองสถานการณ์ใน `Main()` โดยทำตัวเป็นแคชเชียร์ ยิงบาร์โค้ดสินค้ารูปแบบต่างๆ แล้วดูความมหัศจรรย์ของ C# ที่วิ่งไปสลับเมธอดพิมพ์บิลได้อย่างแม่นยำด้วยตัวมันเอง!

## 🗣️ ทบทวน (Review)

::: details ❓ คำถามทบทวนความเข้าใจเชิงลึก

**คำถาม 1:** ระหว่างการใช้ฟีเจอร์ "Optional Parameters (ใส่ค่าเริ่มต้นแบบบทที่ 3)" กับฟีเจอร์ "Method Overloading (สร้างหลายอันแบบบทนี้)" มีความแตกต่างและข้อควรระวังในการเลือกใช้งานอย่างไร?
**แนวคำตอบ:** เป็นคำถามสัมภาษณ์งานยอดฮิต!
- ถ้าลอจิกภายในเมธอด **เหมือนกันเป๊ะ** แค่มีบางค่าที่รับมาเพิ่มเติมแล้วเอาไปคำนวณนิดหน่อย -> **ควรใช้ Optional Parameters** เพราะประหยัดบรรทัดโค้ด ไม่ต้องไปก๊อปปี้สร้างเมธอดใหม่ให้ซ้ำซ้อน (เคารพกฎ DRY)
- แต่ถ้าลอจิกภายในเมธอด **แตกต่างกันอย่างสิ้นเชิง** (เช่น สูตรคำนวณพื้นที่วงกลม กับ สี่เหลี่ยม ไม่เหมือนกันเลย) -> **ต้องใช้ Method Overloading** เพื่อแยกความรับผิดชอบ (Separation of logic) ออกจากกันอย่างเด็ดขาด! การเอาลอจิกวงกลมและสี่เหลี่ยมไปยัดรวมในเมธอดเดียวแล้วใส่ optional parameters สลับไปมา ถือเป็นโค้ดที่ขยะและพังง่ายมาก (Code Smell)

**คำถาม 2:** โค้ดด้านล่างนี้ เกิด Overload สำเร็จหรือไม่?
```csharp
static void FireWeapon(int damage) { }
static void FireWeapon(ref int damage) { }
```
**แนวคำตอบ:** **สำเร็จ 100% ครับ!**
นี่คือความลึกซึ้งของภาษา C# การที่ใส่คำว่า `ref` (หรือ `out`) เข้าไปที่หน้าชนิดตัวแปร ถือว่าเป็นการเปลี่ยนแปลง "ลายเซ็น (Signature)" ของเมธอดนั้นไปเรียบร้อยแล้ว ในสายตาของคอมไพเลอร์ `int` แบบธรรมดา กับ `int แบบมีสายยาง (ref)` ถือเป็นวัตถุคนละประเภทกัน ดังนั้นมันจึง Overload ได้อย่างถูกต้องตามกฎหมาย!
:::
