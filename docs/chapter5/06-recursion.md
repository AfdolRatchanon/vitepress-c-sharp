# Recursion (ลูปไร้ลูป)

> 💡 **เป้าหมาย:** เรียนรู้ศาสตร์มืดทางวิศวกรรมคอมพิวเตอร์ที่งดงามที่สุด—การสร้าง "เมธอดที่เรียกใช้งานตัวมันเอง" (Recursion) เพื่อแก้ปัญหาทางคณิตศาสตร์ที่ซับซ้อน หรือการท่องไปในโครงสร้างต้นไม้ (Tree Structures) โดยไม่ต้องพึ่งพา `for` หรือ `while` loop เลยแม่แต่บรรทัดเดียว!

## 📖 ทฤษฎีและแนวคิดระดับลึก (In-Depth Theory & Concepts)

### 1. ปรัชญาของการเรียกตัวเอง (Self-Reference)

ลองจินตนาการถึงภาพของ "กล่องกระจกสะท้อนกันเอง" หรือฉากในหนัง Inception ที่เราหลับฝันซ้อนฝันลงไปเรื่อยๆ
นั่นแหละคือ **Recursion**! 

ในทางการเขียนโปรแกรม Recursion เกิดขึ้นเมื่อ **เมธอดก้าวเข้าไปในปีกกาของตัวเอง แล้วดันมีคำสั่ง "เรียกชื่อตัวเอง" ซ้ำอีกรอบอยู่ข้างในนั้น!**

```csharp
// ตัวอย่างรหัสทำลายล้างตัวเอง (ห้ามรันเด็ดขาด!)
static void BlackHole()
{
    Console.WriteLine("ฉันกำลังถูกดูดเข้าไป!");
    
    // เรียกตัวเอง... และตัวเองก็เรียกตัวเอง... เป็นวงจรทุรกันดาร
    BlackHole(); 
}
```

---

### 2. มหันตภัย StackOverflowException

ถ้าคุณรันโค้ด `BlackHole()` ด้านบน คุณจะไม่ได้เจอกับ Infinite Loop แบบที่เจอใน `while (true)` ที่ค้างไปเรื่อยๆ
แต่โปรแกรมของคุณจะ **"ระเบิดแครช (Crash) ดับกลางอากาศทันที!"** พร้อมสาดข้อความสีแดงฉานว่า `StackOverflowException`

**อธิบายเชิงโครงสร้างหน่วยความจำ (Memory Architecture):**
ทุกครั้งที่มีการเรียกใช้ Method คอมพิวเตอร์จะต้องจองพื้นที่ในหน่วยความจำที่เรียกว่า **"Call Stack"** เพื่อกักเก็บประวัติว่า "ใครเรียกใคร จะได้กลับไปถูกทาง" 
- `Main` เรียก `BlackHole(1)` -> เปลืองที่ไป 1 ชั้น
- `BlackHole(1)` เรียก `BlackHole(2)` -> เปลืองที่ไปอีกชั้น 
- เนื่องจากมันไม่มีจุดหยุด มันจึงสุมชั้นหนังสือสูงขึ้นเรื่อยๆ จนกระทั่ง **หน่วยความจำ RAM ของ Call Stack เต็มขีดจำกัด!** (นี่คือที่มาของชื่อเว็บไซต์โปรแกรมเมอร์ชื่อดัง StackOverflow)

---

### 3. สถาปัตยกรรมของ Recursion ที่สมบูรณ์แบบ (The Anatomy of Safety)

เพื่อไม่ให้แครช การเขียน Recursion ที่ถูกต้อง "บังคับ" ว่าต้องมีองค์ประกอบ 2 ส่วนนี้เสมอ เหมือนหยินกับหยาง:

1. **Base Case (จุดสิ้นสุด / เบรกฉุกเฉิน):**
   - คือเงื่อนไขพื้นฐาน (มักจะเป็น `if`) ที่เป็นเป้าหมายสุดท้าย เมื่อโปรแกรมเดินทางมาถึงจุดนี้ มันจะต้องส่งคำตอบกลับไป (Return) **โดยไม่มีการเรียกตัวเองซ้ำอีกเด็ดขาด** (เปรียบเหมือนการเจอพื้นตู้เซฟในความฝันชั้นในสุด)
2. **Recursive Step (การขุดลึกลงไป / เครื่องยนต์):**
   - คือส่วนที่มันเรียกตัวเอง แต่กฎเหล็กคือ: "ข้อมูลที่ส่งลงไปในรอบถัดไป ต้องเล็กลง หรือขยับเข้าใกล้ Base Case เสมอ!" (เช่น ขุดจาก 5 ลงไป 4, 3, 2, 1)

## 💻 ตัวอย่างโค้ดเชิงลึก (In-Depth Implementation)

### ตัวอย่างที่ 1: แฟกทอเรียล (Factorial) - คลาสสิกของชาวโลก

ในบทที่ 4 เราใช้ `for` loop ในการทำ Factorial (เช่น 5! = 5 * 4 * 3 * 2 * 1)
แต่ในทางคณิตศาสตร์นิยามของ `n!` คือ `n * (n-1)!` ซึ่งนิยามนี้มันเป็น Recursion ในตัวมันเอง!

::: code-group
```csharp [Program.cs]
using System;

namespace RecursionDemo
{
    class Program
    {
        // เมธอดแบบเวทมนตร์ (Recursive Factorial)
        static long CalculateFactorial(int n)
        {
            // ----------------------------------------------------
            // 1. Base Case (จุดเบรก)
            // ----------------------------------------------------
            // ในทางคณิตศาสตร์ 1! คือ 1 (และ 0! ก็คือ 1)
            // ถ้าเป้าหมายลดลงมาเหลือแค่นี้ ให้คืนค่า 1 ทันที ห้ามขุดลึกกว่านี้!
            if (n <= 1)
            {
                return 1;
            }

            // ----------------------------------------------------
            // 2. Recursive Step (การขุดเจาะ)
            // ----------------------------------------------------
            // นำตัวเลขปัจจุบัน ไป "คูณกับผลลัพธ์ของตัวเองที่ตัวเล็กลง 1 ระดับ"
            return n * CalculateFactorial(n - 1);
        }

        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบคำนวณ Factorial (Recursion) ===");
            
            long result = CalculateFactorial(5);
            Console.WriteLine($"คำตอบของ 5! คือ {result}");
        }
    }
}
```
:::

**การไล่สายประวัติศาสตร์ (Call Stack Tracing):**
หลายคนงงว่าโค้ดสั้นๆ นี้ทำงานได้อย่างไร ลองจินตนาการตามภาพนี้:

1. `Main` เรียก `Factorial(5)`
   - มันยังตอบไม่ได้ มันไปสั่ง `return 5 * Factorial(4)` (ค้างไว้ รอคำตอบจากน้อง)
2. `Factorial(4)` ทำงาน
   - ยังตอบไม่ได้ สั่ง `return 4 * Factorial(3)` (ค้างไว้)
3. `Factorial(3)` ทำงาน
   - ยังตอบไม่ได้ สั่ง `return 3 * Factorial(2)` (ค้างไว้)
4. `Factorial(2)` ทำงาน
   - ยังตอบไม่ได้ สั่ง `return 2 * Factorial(1)` (ค้างไว้)
5. `Factorial(1)` ทำงาน **(ฮีโร่มาแล้ว!)**
   - ชนกับ Base Case! เข้าเงื่อนไข `n <= 1` มันจึงสั่ง `return 1` ทันที โดยไม่เรียกใครต่อ!

**จังหวะคลายปม (The Unwinding):**
- `Factorial(1)` คืนเลข `1` กลับไปให้คนที่รออยู่คือบรรทัดที่ 4
- บรรทัดที่ 4 เอาไปเข้าสมการ `2 * 1 = 2` แล้วคืนกลับไปให้คนที่รออยู่
- บรรทัดที่ 3 รับเลข 2 มาทำสมการ `3 * 2 = 6` คืนกลับไป
- บรรทัดที่ 2 รับเลข 6 มาทำสมการ `4 * 6 = 24` คืนกลับไป
- บรรทัดที่ 1 รับเลข 24 มาทำสมการ `5 * 24 = 120` แล้วคืนร่างสุดท้ายส่งตรงเข้าสู่หน้าจอของ `Main` อย่างสง่างาม!!

### ตัวอย่างที่ 2: ระบบค้นหาโฟลเดอร์ซ้อนโฟลเดอร์ (File System Traversal)

แล้ว Recursion มีประโยชน์ในการทำโปรแกรมจริงๆ หรือไม่? 
ตอบเลยว่า **มหาศาล!** ลองนึกภาพ "โฟลเดอร์ใน Windows" ซึ่งใน 1 โฟลเดอร์ อาจจะมีโฟลเดอร์ย่อยอีก 10 อัน และในนั้นก็มีย่อยลงไปอีก (ไม่สิ้นสุด) คุณจะใช้ `for` loop กวาดให้ครบทุกโฟลเดอร์ได้อย่างไร ในเมื่อคุณไม่รู้เลยว่ามันลึกกี่ชั้น? 
มีเพียง Recursion เท่านั้นที่แก้ปัญหานี้ได้!

::: code-group
```csharp [Program.cs]
using System;

namespace FolderCrawler
{
    class Program
    {
        // แบบจำลองการค้นหาไฟล์ด้วย Recursion (Concept Mockup)
        // รับค่า ชื่อโฟลเดอร์ปัจจุบัน และ ระดับความลึก (เพื่อวาดการเยื้อง)
        static void CrawlDirectory(string currentFolder, int depthLevel)
        {
            // สร้างช่องว่างเพื่อเยื้องระดับสายตา ตามความลึก (Visual Indentation)
            string indent = new string('-', depthLevel * 2); 
            Console.WriteLine($"{indent}> กำลังค้นหาในกล่อง: {currentFolder}");

            // Base Case สไตล์ประยุกต์: 
            // ถ้าโฟลเดอร์นี้ไม่มีของอยู่แล้ว (ในชีวิตจริงใช้คำสั่ง Directory.GetDirectories)
            // ในจำลองนี้เราสมมติว่าถ้าความลึกลงไปถึง 3 ชั้น ให้ถือว่าสุดทาง (Base Case)
            if (depthLevel >= 3)
            {
                // หยุดการค้นหาลึกซึ้ง และตีกลับขึ้นข้างบน
                return; 
            }

            // Recursive Step: สมมติว่าทุกกล่อง จะมีกล่องย่อย 2 ใบซ่อนอยู่ข้างใน
            // (เราสั่งเรียกตัวเอง 2 ครั้ง! เพื่อดำดิ่งลงไปทางซ้าย และ ขวา)
            CrawlDirectory($"{currentFolder}_A", depthLevel + 1);
            CrawlDirectory($"{currentFolder}_B", depthLevel + 1);
        }

        static void Main(string[] args)
        {
            Console.WriteLine("=== ระบบหุ่นยนต์ขุดเจาะไฟล์ ===");
            
            // ปล่อยหุ่นยนต์ลงไปในกล่องระดับ 0 (Root)
            CrawlDirectory("Root", 0);
        }
    }
}
```
:::

**ผลลัพธ์บนหน้าจอ (โชว์การกระจายตัวเหมือนรากต้นไม้ Fractal):**
```
> กำลังค้นหาในกล่อง: Root
--> กำลังค้นหาในกล่อง: Root_A
----> กำลังค้นหาในกล่อง: Root_A_A
------> กำลังค้นหาในกล่อง: Root_A_A_A
------> กำลังค้นหาในกล่อง: Root_A_A_B
----> กำลังค้นหาในกล่อง: Root_A_B
------> กำลังค้นหาในกล่อง: Root_A_B_A
------> กำลังค้นหาในกล่อง: Root_A_B_B
--> กำลังค้นหาในกล่อง: Root_B
... (กระจายตัวไปเรื่อยๆ)
```

## 🎯 โจทย์ฝึกปฏิบัติเสริมความเข้าใจ (Mini Exercise)

### **โจทย์ที่ 1: เถียงกับตัวเอง (Recursion vs Iteration)**
พิจารณาฟังก์ชัน 2 ตัวนี้ที่ให้ผลลัพธ์การนับถอยหลังเหมือนกัน:
**แบบที่ 1 (Loop ปกติ):**
```csharp
for (int i = 5; i > 0; i--) Console.WriteLine(i);
```
**แบบที่ 2 (Recursion):**
```csharp
static void CountDown(int i) { 
    if(i == 0) return; 
    Console.WriteLine(i); 
    CountDown(i - 1); 
}
```
**คำถาม:** ในแง่ของ "ประสิทธิภาพทางหน่วยความจำ (Memory Performance)", วิธีการใดถือว่าดีกว่าและปลอดภัยกว่าบนระบบคอมพิวเตอร์ทั่วไป? เพราะเหตุใด?

### **โจทย์ที่ 2: ลำดับเลขคณิตมรณะ (Fibonacci Sequence)**
เลขฟีโบนัชชี (Fibonacci) คืออนุกรมที่ตัวเลขตัวถัดไป เกิดจากการเอาตัวเลข 2 ตัวก่อนหน้ามาบวกกัน 
เช่น: 0, 1, 1, 2, 3, 5, 8, 13, 21 ... 
**ภารกิจ:** จงทำความเข้าใจและเติม Base Case ให้กับสมการ Recursion นี้ให้สมบูรณ์
```csharp
static int Fib(int n)
{
    // จุดเติม Base Case (ตัวเลขตำแหน่งที่ 0 ให้ตอบ 0, ตัวเลขตำแหน่งที่ 1 ให้ตอบ 1)
    if (n == 0) return ???;
    if (n == 1) return ???;

    // Recursive Step: แตกตัวไปบวกเพื่อนบ้านสองคนข้างหลัง
    return Fib(n - 1) + Fib(n - 2); 
}
```

::: details 💡 คำใบ้และแนวทาง (Hints)
- **สำหรับโจทย์ 1:** Loop แบบ `for` กินขาดล้านเปอร์เซ็นต์! เพราะการใช้ `for` ไม่มีการจองหน่วยความจำซ้อนๆ กันใน Call Stack เลย มันแค่วนรอบตัวเองอยู่ใน Memory ช่องเดิม แต่ Recursion ต้องจอง Stack ใหม่ตลอดเวลา ถ้าคุณสั่งนับถอยหลังจาก 1,000,000 การใช้ Recursion จะทำให้โปรแกรมแครช (StackOverflow) แน่นอน! (กฎเหล็ก: ใช้ Recursion เฉพาะเมื่องานมันซับซ้อนจนใช้ลูปธรรมดาเขียนลำบากจริงๆ เท่านั้น)
- **สำหรับโจทย์ 2:** Base Case คือหัวใจ ถ้า n=0 ให้ return 0 และถ้า n=1 ให้ return 1 (โค้ดฟีโบนัชชีนี้สวยงามมาก แต่มันเป็นการแตกแขนงแบบต้นไม้ที่ช้าสุดๆ ในทางปฏิบัติเรียกว่า Exponential Time $O(2^N)$!)
:::

## 🔥 Challenge (โจทย์ท้าทายสุดโหด!)

**โจทย์: มนต์ดำสะกดกลับหลัง (The Recursive Word Reverser)**

เป้าหมายคือการสร้างเครื่องมือที่รับข้อความเข้ามา แล้วตีกลับหลังทั้งหมด 
(เช่น โยน `HELLO` เข้าไป ต้องได้ `OLLEH` กลับมา)
**แต่มีข้อแม้ว่า "ห้ามใช้คำสั่ง `for` หรือ `while` loop อย่างเด็ดขาด!" คุณต้องใช้มนต์ดำ Recursion เท่านั้น**

**กระบวนการคิดของสถาปนิกเวทมนตร์ (Algorithm):**
1. สร้างเมธอด `static string ReverseText(string text)`
2. **Base Case:** ตรวจสอบว่า `if (text.Length <= 1)` ถ้าสายสติงสั้นเหลือแค่ 1 ตัวอักษร (หรือ 0) ก็ไม่มีอะไรให้กลับหลังแล้ว! ให้ส่ง `text` นั้นคืนกลับไปดื้อๆ เลย
3. **Recursive Step:** หากยาวกว่า 1 ตัว
   - แอบตัด **"ตัวอักษรแรกสุด (Index 0)"** เก็บใส่กระเป๋าไว้ (`text.Substring(0, 1)`)
   - เฉือนสายสติงที่เหลือ (ตั้งแต่ตำแหน่งที่ 1 เป็นต้นไป `text.Substring(1)`) แล้ว **โยนก้อนเนื้อที่เหลือนี้กลับเข้าไปในเครื่องจักร ReverseText ซ้ำอีกครั้ง!**
   - คืนค่า `return` โดยนำ "ก้อนเนื้อที่เพิ่งไปกลับหลังเสร็จหมาดๆ" มาตั้งข้างหน้า แล้วนำ "ตัวอักษรแรกที่เก็บไว้ในกระเป๋า" ไปต่อท้ายสุด!

ถ้าคุณสามารถประสานพลัง `Substring` ของ String เข้ากับ Recursion จนเขียนโจทย์ข้อนี้ผ่านได้ คุณบรรลุวิชาการเขียนโปรแกรมเชิงตรรกะในระดับปริญญาตรี (Computer Science 101) ไปเรียบร้อยแล้วครับ!

ไม่ว่าคุณจะมี RAM เหลือล้น 128GB โปรแกรมก็จะแครชทันทีที่ระดับความลึกกินพื้นที่เกินขีด 1MB นี้ (ซึ่งมักจะตกอยู่ที่ความลึกประมาณระดับ 10,000 - 50,000 ชั้น ขึ้นอยู่กับว่าโยนพารามิเตอร์เยอะแค่ไหน)

**คำถาม 2:** โค้ดฟังก์ชัน Recursion ถือว่าเป็น "โค้ดที่อ่านยาก (Unreadable)" และไม่ควรใช้ในบริษัทจริงๆ หรือไม่?
**แนวคำตอบ:** ทั้งใช่และไม่ใช่ 
- **ใช่:** ถ้านำมันมาใช้กับงานง่ายๆ อย่างการรันลูปพริ้นท์ตัวเลข หรือการหาผลรวม ซึ่งสามารถใช้ `for` หรือ `while` เขียนได้ง่ายกว่า 100 เท่า แถมเร็วกว่า 
- **ไม่ใช่ (และขาดไม่ได้):** ในวงการบริษัทซอฟต์แวร์ที่พัฒนาโครงสร้างข้อมูลที่ซับซ้อน เช่น การเขียนระบบประมวลผล "คอมไพเลอร์ภาษา C#", ระบบ "ค้นหาเส้นทางใน GPS Maps", ระบบ "กวาดล้างไฟล์ Directory", การใช้ `for` loop ธรรมดานั้นนับเป็นเรื่องนรกแตก (คุณอาจจะต้องซ้อน `while` และจัดการ Stack เองเป็นร้อยบรรทัด) ในขณะที่ Recursion แค่ 5 บรรทัด สามารถทะลวงโค้ดเหล่านั้นได้อย่างสง่างามและทรงพลัง!
